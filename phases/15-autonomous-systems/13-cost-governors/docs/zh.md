# 操作预算、迭代上限和成本调控器

> 一个中型电商智能体的月度 LLM 成本在其团队启用"订单跟踪"技能后从 $1,200 跃升至 $4,800。这不是定价 bug。这是一个找到了新循环并在其中持续消费的智能体。Microsoft 的 Agent Governance Toolkit（2026 年 4 月 2 日）编纂了对此类问题的防御：每次请求 `max_tokens`、每个任务的 token 和美元预算、每日/每月上限、迭代上限、分层模型路由、提示缓存、上下文窗口化、昂贵操作的 HITL 检查点、预算超支时的终止开关。Anthropic 的 Claude Code Agent SDK 以不同名称提供相同的原语。财务速度限制——例如在 10 分钟内超过 $50 时切断访问——比月度上限更快捕获循环。

**类型：** 学习
**语言：** Python（标准库，分层成本调控器模拟器）
**前置条件：** 第 15 阶段 · 10（权限模式），第 15 阶段 · 12（持久执行）
**时间：** 约 60 分钟

## 问题

自主智能体在每一轮都花费真实金钱。聊天机器人的不良输出是一个不良回复；智能体的不良循环是一张账单。行业记录的失败模式术语是"钱包拒绝服务"——智能体持续推理、持续工具调用、持续计费，没有任何东西阻止它，因为没有设计任何东西来阻止它。

修复不是一个数字。它是一个在不同时间尺度和粒度上的限制栈：每次请求、每个任务、每小时、每天、每月。一个设计良好的栈在几分钟内捕获失控循环，在几小时内捕获缓慢泄漏，在一天内捕获不良发布。相同的栈在智能体是长周期和自主的时候保持预算。

这是一个工程课：数学是平凡的，纪律是团队失败的地方。下面的限制列表全部在 Microsoft Agent Governance Toolkit 或 Anthropic Claude Code Agent SDK 文档中命名。

## 概念

### 成本调控器栈

1. **每次请求 `max_tokens`。** 简单。防止任何单次调用发出无界完成。
2. **每个任务 token 预算。** 在整个运行中，不超过 N 个 token。在上限处硬停止。
3. **每个任务美元预算。** 与 token 相同但是以货币计。Claude Code 中的 `max_budget_usd`。
4. **每个工具调用上限。** 不超过 N 次 `WebFetch` 调用、N 次 `shell_exec` 调用等。
5. **迭代上限（`max_turns`）。** 智能体循环总迭代次数；防止无限推理循环。
6. **每分钟 / 每小时 / 每天 / 每月上限。** 滚动窗口。在不同时间尺度捕获泄漏。
7. **财务速度限制。** 例如"如果在 10 分钟内支出超过 $50，切断访问"。在月度上限触发之前捕获基于循环的消耗。
8. **分层模型路由。** 默认使用较小的模型；仅当分类器判断任务需要时才升级到较大的模型。
9. **提示缓存。** 系统提示和稳定上下文存储在提供商缓存中；重新发送的 token 成本接近零。
10. **上下文窗口化。** 压缩 / 摘要以将活跃上下文保持在阈值以下；直接的 token 成本降低。
11. **昂贵操作的 HITL 检查点。** 在已知昂贵的操作（长工具调用、大下载、昂贵的模型升级）之前，需要人工确认。
12. **预算超支时的终止开关。** 当任何上限触发时，会话中止。上限被记录；需要单独的重新启用路径。

### 为什么是栈，而不是一个上限

单个月度上限仅在钱包已空后才捕获失控智能体。单个每次请求上限在会话级别什么也捕获不了。不同的失败模式需要不同的时间尺度：

- **失控循环**（智能体陷入 5 秒重试）：由速度限制捕获。
- **缓慢泄漏**（智能体每个任务做约 2 倍预期工作）：由每日上限捕获。
- **不良发布**（新版本使用 5 倍 token）：由每周 / 每月上限捕获。
- **合法激增**（真实需求，不是 bug）：由小时 / 日上限捕获，带有清晰日志。

### Claude Code 的预算面

Claude Code Agent SDK 暴露（公开文档）：

- `max_turns` — 迭代上限。
- `max_budget_usd` — 美元上限；超支时会话中止。
- `allowed_tools` / `disallowed_tools` — 工具允许列表和拒绝列表。
- 工具使用前的钩子点，用于自定义成本核算。

与权限模式阶梯（第 10 课）结合。没有 `max_budget_usd` 的 `autoMode` 会话是无治理的自主性。Anthropic 明确框架自动模式为需要预算控制；分类器与成本正交。

### EU AI Act、OWASP Agentic Top 10

Microsoft 的 Agent Governance Toolkit 涵盖 OWASP Agentic Top 10 和 EU AI Act 第 14 条（人类监督）要求。对于在欧盟的生产，日志记录和上限执行不是可选的。

### 观察到的 $1,200 → $4,800 案例

Microsoft 文档中的真实案例：一个电商智能体在添加新工具后月度成本翻了三倍。该工具允许智能体在每个会话期间轮询订单状态。没有循环检测。没有每个工具上限。没有周环比增长警报。修复是每个工具上限加上每日增长警报。这是一个模板：每个新工具面都是一个新的潜在循环；每个新工具都需要自己的上限和自己的警报。

## 使用它

`code/main.py` 模拟一个有和没有分层成本调控器栈的智能体运行。模拟的智能体在几轮后漂移进入轮询循环；分层栈在速度窗口内捕获它，而单个月度上限直到几天后才会触发。

## 交付它

`outputs/skill-agent-budget-audit.md` 审计一个提议的智能体部署的成本调控器栈并标记缺失的层。

## 练习

1. 运行 `code/main.py`。确认速度限制在轮询循环轨迹上先于迭代上限触发。现在禁用速度限制，测量智能体在迭代上限捕获它之前"花费"了多少。

2. 为浏览器智能体（第 11 课）设计每个工具上限集。哪个工具需要最紧的上限？哪个工具可以无风险地无界运行？

3. 阅读 Microsoft Agent Governance Toolkit 文档。列出工具包命名的每种上限类型。将每种映射到一个失败模式（失控循环、缓慢泄漏、不良发布、激增）。

4. 为一个现实任务（例如"分类仓库中的 50 个 issue"）定价隔夜无人值守运行。将 `max_budget_usd` 设置为你点估计的 2 倍。证明 2 倍。

5. Claude Code 的 `max_budget_usd` 在会话总成本上触发。设计一个你将在外部强制执行的补充速度限制。什么触发切断，重新启用是什么样子？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| 钱包拒绝服务 | "失控账单" | 智能体循环产生没有上限阻止的支出 |
| max_tokens | "每次请求上限" | 单次完成大小的上限 |
| max_turns | "迭代上限" | 会话中智能体循环迭代的上限 |
| max_budget_usd | "美元终止开关" | 会话成本上限；超支时中止 |
| 速度限制 | "速率上限" | 短窗口内的支出限制（例如 $50 / 10 分钟） |
| 分层路由 | "先小模型" | 默认使用廉价模型；仅当分类器需要时升级 |
| 提示缓存 | "缓存的系统提示" | 提供商端缓存将重新发送 token 成本降至接近零 |
| HITL 检查点 | "人工批准门" | 昂贵操作前需要人工确认 |
