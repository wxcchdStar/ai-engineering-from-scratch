# 长时间运行的后台智能体：持久执行

> 生产级长周期智能体不在 `while True` 中运行。每个 LLM 调用成为一个带有检查点、重试和重放的活动。Temporal 的 OpenAI Agents SDK 集成于 2026 年 3 月 GA。Claude Code Routines（Anthropic）在没有持久本地进程的情况下运行计划的 Claude Code 调用。会话在人工输入时暂停，在部署中存活，并从按 `thread_id` 键控的最新检查点恢复。在新的人机工程学背后是一个旧的模式——工作流编排——有一个新输入：LLM 调用作为非确定性活动，在恢复时必须被确定性地重放。

**类型：** 学习
**语言：** Python（标准库，最小持久执行状态机）
**前置条件：** 第 15 阶段 · 10（权限模式），第 15 阶段 · 01（长周期智能体）
**时间：** 约 60 分钟

## 问题

考虑一个运行四个小时的智能体。它调用三个工具，提示用户两次，并进行四十次 LLM 调用。中途，它运行的主机重启。会发生什么？

- 在天真的 `while True` 循环中：一切丢失。运行从头开始。三个工具调用（具有真实副作用）再次执行。用户再次被提示他们已经批准的事情。四十次 LLM 调用被重新计费。
- 使用持久执行：运行从最近的检查点恢复。已完成的活动不重新执行；它们的结果从持久日志重放。用户不重新批准他们已经批准的事情。已进行的 LLM 调用不重新计费。

这与工作流引擎已经交付了十年的模式相同（Temporal、Cadence、Uber 的 Cherami）。新的是 LLM 调用现在是一种活动——非确定性、昂贵、具有副作用——并且它们干净地适应这种模式。

本课的运行主题：长周期可靠性衰减（METR 观察到"35 分钟退化"——成功率大约随周期二次方下降）。持久执行使运行能够超过可靠性特征支持的长度，如果设计正确，这是一种新的安全失败方式，如果设计错误，则是不安全的。

## 概念

### 活动、工作流和重放

- **工作流**：确定性编排代码。定义活动的顺序、分支、等待。必须是确定性的，以便可以从事件日志重放而不会出现意外的分歧。
- **活动**：一个非确定性的、可能失败的工作单元。LLM 调用、工具调用、文件写入、HTTP 请求。每个活动都记录其输入和（一旦完成）其输出。
- **事件日志**：持久后备存储。每个活动开始、完成、失败、重试以及每个工作流决策都被记录。
- **重放**：在恢复时，工作流代码从头重新运行；每个已完成的活动返回其记录的结果而不重新执行。只有未完成的活动才实际运行。

这与 React 对虚拟 DOM 重新渲染或 Git 从提交重建工作树的形状相同。编排器中的确定性是使持久性廉价的原因。

### 为什么 LLM 调用适合这种模式

LLM 调用是：
- 非确定性的（temperature > 0；即使 temperature 0 也会跨模型版本漂移）。
- 昂贵的（金钱和延迟）。
- 可能失败的（速率限制、超时）。
- 具有副作用的（如果它们调用工具）。

这正是活动特征。将每个 LLM 调用包装为活动，你就能获得指数退避重试、跨重启的检查点以及用于调试的可重放跟踪。

### 按 `thread_id` 键控的检查点

LangGraph、Microsoft Agent Framework、Cloudflare Durable Objects 和 Claude Code Routines 都收敛到相同的 API 形状：`thread_id`（或等价物）标识会话；每个状态转换持久化到后端（默认 PostgreSQL，SQLite 用于开发，Redis 用于缓存）；恢复读取最新检查点。

后端选择很重要：

- **PostgreSQL**：持久、可查询、在部署中存活。LangGraph 的默认选择。
- **SQLite**：仅本地开发；跨主机丢失数据。
- **Redis**：快速但是临时的，除非配置了 AOF/快照。
- **Cloudflare Durable Objects**：透明分布；由唯一键限定范围；存活数小时到数周。

### 人工输入作为一等状态

提议-然后-提交（第 15 课）需要一个持久的"等待人类"状态。工作流暂停，外部队列持有待处理的请求，批准从该点精确恢复。没有持久性，这是尽力而为的；有了它，隔夜批准到达，工作流在早上继续。

### 35 分钟退化

METR 观察到，每个测量的智能体类别在连续运行约 35 分钟后都表现出可靠性衰减。任务持续时间翻倍大约使失败率翻四倍。持久执行不能修复这一点；它让你运行超过可靠性特征支持的长度。安全的模式是将持久性与要求在重新进入时进行新 HITL 的检查点以及限制总计算量的预算终止开关（第 13 课）结合，无论挂钟时间如何。

### 持久执行是错误答案的情况

- 运行短于几分钟且没有人工输入。开销 > 收益。
- 严格只读的信息检索。
- 正确性需要在一个上下文窗口内端到端的任务（某些推理任务；某些一次性生成）。

## 使用它

`code/main.py` 在标准库 Python 中实现了一个最小的持久执行引擎。它支持：

- `@activity` 装饰器，将输入和输出记录到 JSON 事件日志。
- 一个排序活动的工作流函数。
- 一个 `run_or_replay(workflow, event_log)` 函数，重放已完成的活动而不重新执行它们。

驱动程序模拟一个三活动工作流，中途崩溃，并显示（a）天真重试重新执行一切与（b）重放仅运行缺失的活动。

`outputs/skill-durable-execution-review.md` 审查一个提议的长时间运行智能体部署的正确持久执行形状：活动、确定性、检查点后端、人工输入状态和恢复时 HITL 策略。

## 练习

1. 运行 `code/main.py`。观察天真重试和重放之间活动执行计数的差异。更改崩溃点并显示重放计数相应变化。

2. 将玩具引擎转换为显式使用 `thread_id`。模拟两个并发会话共享引擎，并确认它们的事件日志不冲突。

3. 在玩具引擎中取一个活动。引入非确定性（工作流决策中的挂钟时间戳）。演示重放时的分歧。解释真实引擎如何处理这一点（副作用注册、`Workflow.now()` API）。

4. 阅读 LangChain 的"Runtime behind production deep agents"文章。列出运行时持久化的每个状态，并命名每个覆盖的失败模式。

5. 为一个 6 小时自主编码任务设计检查点策略。你在哪里设置检查点？崩溃后恢复是什么样子？什么需要新的 HITL？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| 工作流 | "智能体的脚本" | 确定性编排代码；可从事件日志重放 |
| 活动 | "一个步骤" | 非确定性单元（LLM 调用、工具调用）；前后记录 |
| 事件日志 | "后备存储" | 每个状态转换的持久记录 |
| 重放 | "恢复" | 重新运行工作流；已完成的活动返回记录的结果而不重新执行 |
| 检查点 | "保存点" | 按 thread_id 键控的持久状态；恢复时最新胜出 |
| thread_id | "会话键" | 限定持久状态的标识符 |
| 35 分钟退化 | "可靠性衰减" | METR：成功率大约随周期二次方下降 |
| 非确定性 | "重放时的漂移" | 挂钟、随机、LLM 输出；必须注册为副作用 |
