# 工具接口——为什么智能体需要结构化I/O

> 语言模型产生token。程序执行动作。两者之间的鸿沟就是工具接口（tool interface）：一个让模型请求动作、宿主执行动作的契约。2026年的每个技术栈——OpenAI、Anthropic和Gemini的函数调用；MCP的`tools/call`；A2A的任务部件——都是同一个四步循环的不同编码。本课命名该循环，并展示运行它所需的最小机制。

**类型：** 学习
**语言：** Python（标准库，无LLM）
**前置课程：** Phase 11（LLM补全API）
**时间：** 约45分钟

## 学习目标

- 解释为什么一个只能生成文本的LLM无法独自对真实世界执行动作。
- 画出四步工具调用循环（描述 → 决策 → 执行 → 观察），并指出每一步由谁负责。
- 将工具描述写成三个部分：名称、JSON Schema输入和一个确定性执行器函数。
- 区分纯函数工具（pure）与有副作用工具（side-effecting），并说明这一区分对安全性的重要意义。

## 问题定义

LLM输出的是下一个token的概率分布。这就是它全部的输出表面。如果你问一个聊天模型"班加罗尔现在天气怎么样"，它可以写出一句看似合理的话，但它无法调用天气API。那句话可能碰巧是对的，也可能已经过时了三天。

弥合这一鸿沟就是工具接口的目的。宿主程序（host program）——你的智能体运行时、Claude Desktop、ChatGPT、Cursor或自定义脚本——向模型宣告一组可调用的工具。当模型判断需要执行动作时，它会输出一个结构化载荷（payload），命名一个工具及其参数。宿主解析该载荷，真正执行工具，并将结果反馈回来。循环持续进行，直到模型决定不再需要调用。

这个契约的第一个版本于2023年6月以OpenAI的"functions"参数形式发布。Anthropic随后在Claude 2.1中加入了`tool_use`块。Gemini几个月后添加了`functionDeclarations`。现在每个提供商都暴露相同的形状：输入是JSON-Schema类型的工具列表，输出是JSON载荷的工具调用。模型上下文协议（Model Context Protocol，2024年11月）将契约泛化，使一个工具注册中心可以服务所有模型。A2A（2026年4月，v1.0）在同一原语之上为智能体间委派添加了一层。

四步循环是所有这些之下的不变量。Phase 13的其余内容都是对它的细化。

## 核心概念

### 第一步：描述（Describe）

宿主使用三个字段声明每个工具。

- **名称。** 稳定的、机器可读的标识符。`get_weather`，而不是"天气工具"。
- **描述。** 一段自然语言简介。"当用户询问特定城市的当前天气状况时使用。不要用于历史数据。"
- **输入Schema。** 一个JSON Schema对象（draft 2020-12），描述工具的参数。

模型接收此列表。现代提供商使用特定于提供商的模板将这些声明序列化到系统提示中，因此作为调用者，你只需处理结构化形式。

### 第二步：决策（Decide）

给定用户的消息和可用工具，模型选择三种行为之一。

1. **直接用文本回答。** 不调用工具。
2. **调用一个或多个工具。** 输出结构化调用对象。在`parallel_tool_calls: true`（OpenAI和Gemini默认开启，Anthropic需要手动启用）下，模型可以在一轮中输出多个调用。
3. **拒绝。** 严格模式的结构化输出可以产生类型化的`refusal`块代替调用。

工具调用载荷有三个稳定字段：调用`id`、工具`name`和JSON `arguments`对象。id的存在是为了让宿主将后续结果与特定调用关联起来，这在并行调用乱序返回时很重要。

### 第三步：执行（Execute）

宿主接收调用，根据声明的schema验证参数，然后运行执行器。无效参数意味着模型幻觉了一个字段或使用了错误的类型——这是弱模型非常常见的失败模式。生产环境的宿主在参数无效时做三件事之一：快速失败并将错误暴露给模型；使用受约束的解析器修复JSON；或者将验证错误包含在提示中重试模型。

执行器本身是普通代码。Python、TypeScript、shell命令、数据库查询。它产生一个结果，通常是字符串，但可以是任何JSON值或结构化内容块（MCP中的文本、图像或资源引用）。结果必须是可序列化的。

### 第四步：观察（Observe）

宿主将工具结果追加到对话中（作为带有匹配`id`的`tool`角色消息），并重新调用模型。模型现在在上下文中拥有工具输出，可以产生最终答案或请求更多调用。这会持续到模型停止输出调用或宿主达到迭代次数的安全限制。

### 信任分割

工具分为两类，这对安全性很重要。

- **纯函数（Pure）。** 只读、确定性、无副作用。`get_weather`、`search_docs`、`get_current_time`。可以安全地推测性调用。
- **有后果的（Consequential）。** 修改状态、花费金钱、接触用户数据。`send_email`、`delete_file`、`execute_trade`。必须设置门控。

Meta 2026年的智能体安全"二规则"（Rule of Two）指出，单轮交互中最多只能组合以下三者中的两个：不可信输入、敏感数据、有后果的动作。工具接口是你执行该规则的地方——通过拒绝调用、要求用户确认或提升权限范围。详见Phase 13 · 15的完整安全章节和Phase 14 · 09的智能体级权限策略。

### 循环在哪里运行

| 上下文 | 谁描述 | 谁决策 | 谁执行 |
|--------|--------|--------|--------|
| 单轮函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | LLM | 应用开发者 |
| MCP | MCP服务器 | 通过MCP客户端的LLM | MCP服务器 |
| A2A | Agent Card发布者 | 调用智能体 | 被调用智能体 |
| Web浏览器（函数调用智能体） | 浏览器扩展 / WebMCP | LLM | 浏览器运行时 |

各处都是相同的四步。列名变了；结构没变。

### 为什么不直接提示模型输出JSON？

"要求模型用JSON回复"是函数调用之前的模式。在前沿模型上约5%到15%的时间会失败，在较小模型上失败率更高。失败模式包括缺少大括号、尾随逗号、幻觉字段和错误类型。然后你需要一个JSON修复步骤、一次重试或一个受约束的解码器。

原生函数调用更好，有三个原因。第一，提供商对模型在精确的调用形状上进行端到端训练，因此有效JSON率在严格模式下攀升到98%到99%。第二，调用载荷位于自己的协议槽中，不在自由文本内——因此工具调用永远不会泄漏到用户可见的回复中。第三，提供商通过受约束解码（constrained decoding）强制schema合规（OpenAI的strict模式、Anthropic的`tool_use`、Gemini的`responseSchema`）。输出保证通过验证。

### 断路器

循环在模型停止输出调用或宿主达到最大轮次数时终止。生产环境的宿主将此设置在5到20轮之间。超过此数，你几乎肯定陷入了模型无法退出的循环。Claude Code默认20次；OpenAI Assistants默认10次；Cursor的agent模式默认25次。

另一种选择——无界循环——每六个月就会以"智能体一夜间花了400美元API费用"的事后分析出现。不要在没有界限的情况下上线。

Phase 14 · 12深入覆盖错误恢复和自愈；Phase 17覆盖生产环境的速率限制。

## 动手实践

`code/main.py`在没有LLM的情况下运行四步循环。一个假的"决策者"函数通过对用户消息的模式匹配模拟模型；执行器、schema验证器和观察步骤的工具是真实的。运行它以查看完整的请求/响应编排和可打印的中间状态，然后在之后的课程中将假决策者替换为任何真实提供商。

需要关注的：

- 工具注册表每个工具包含三个字段：名称、描述、schema和一个执行器引用。
- 验证器是一个最小JSON Schema子集（types、required、enum、min/max），仅使用标准库编写。Phase 13 · 04提供更完整的版本。
- 循环将迭代次数限制在五次。生产智能体正是需要这种断路器。

## 交付产出

本课程产出`outputs/skill-tool-interface-reviewer.md`。给定一个工具定义草案（名称 + 描述 + schema + 执行器大纲），该skill审计其循环适应性：名称是否机器稳定、描述是否是完整的使用简介、schema是否正确使用JSON Schema 2020-12、以及纯函数vs有后果的分类是否明确。

## 练习

1. 向`code/main.py`添加第四个工具`get_stock_price(ticker)`。将其描述写为"当用户按股票代码询问当前股价时使用。不要用于历史价格或市场概览。"运行工具并确认假决策者将提到股票代码的查询路由到新工具。

2. 破坏schema验证器。传递一个`arguments`对象缺少必需字段的调用，确认宿主在执行前拒绝它。然后传递一个带有额外未知字段的调用。决定：宿主应该拒绝还是忽略？用安全论证证明你的选择。

3. 将工具中的每个工具分类为纯函数或有后果的。在需要的注册表条目中添加`consequential: true`标志，并修改循环，使其在选择有后果的工具时打印"would confirm with user"行。这就是每个生产宿主需要的确认门控的形状。

4. 在纸上画出四步循环，并用上面的提供商列表为你喜欢的客户端（Claude Desktop、Cursor、ChatGPT或自定义栈）填写。与Phase 13 · 06中的MCP特定变体交叉参考。

5. 从头到尾阅读OpenAI的函数调用指南。识别一个存在于请求中但不在本课呈现的四步循环中的字段。解释它增加了什么以及为什么它是便利的而非必要的。

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------|----------|
| 工具（Tool） | "模型可以调用的东西" | 名称 + JSON-Schema类型输入 + 执行器函数的三元组 |
| 函数调用（Function calling） | "原生工具使用" | 提供商级别的API支持，用于输出结构化工具调用而非散文 |
| 工具调用（Tool call） | "模型的动作请求" | 模型输出的带有`id`、`name`、`arguments`的JSON载荷 |
| 工具结果（Tool result） | "工具返回了什么" | 执行器的输出，包装在带有匹配id的`tool`角色消息中 |
| 并行工具调用 | "一轮多个调用" | 单个模型轮次中的多个调用对象，独立且可通过id排序 |
| 严格模式（Strict mode） | "保证的JSON" | 受约束解码，强制模型输出根据声明的schema进行验证 |
| 纯函数工具 | "只读工具" | 无副作用；可安全重新运行 |
| 有后果工具 | "动作工具" | 改变外部状态；需要门控、审计或用户确认 |
| 四步循环 | "工具调用周期" | 描述 → 决策 → 执行 → 观察 |
| 宿主（Host） | "智能体运行时" | 持有工具注册表、调用模型并运行执行器的程序 |
