# 边缘推理 — 设备端模型、蒸馏、量化与 WebNN

> 边缘推理（Edge inference）在用户设备上本地运行 LLM，而不是在云中。2026 年的格局：**设备端模型（On-device models）** 如 Llama 3.2 1B/3B、Gemma 3 1B、Phi-4-mini 在旗舰手机上以 10-30 token/秒的速度运行。**模型蒸馏（Model distillation）** 训练一个更小的"学生"模型来模仿一个更大的"教师"模型 — 将 70B 模型的知识压缩到 1-3B 参数中。**边缘量化（Quantization for edge）** 使用 INT4 或 INT8 将模型压缩到 < 2 GB — GGUF 格式（第 17 阶段 · 09）是边缘部署的标准。**推理引擎**：**TensorFlow Lite** 通过 XNNPACK 委托在 Android 上运行量化模型；**ONNX Runtime** 通过跨平台执行提供程序（CPU、GPU、NPU）提供跨平台推理；**Core ML** 在 Apple Silicon 上使用 ANE（Apple Neural Engine，Apple 神经引擎）实现高效推理；**WebNN** 是一个浏览器 API，直接在浏览器中运行神经网络，利用设备 GPU/NPU。边缘推理的用例：隐私（数据不离开设备）、离线（无网络连接）、低延迟（无网络往返）、成本（无服务器费用）。限制：模型质量（较小的模型能力较弱）、电池消耗、设备碎片化。

**类型：** 学习

## 学习目标

- 列举边缘推理的四个用例（隐私、离线、低延迟、成本）和三个限制（模型质量、电池消耗、设备碎片化）。
- 解释模型蒸馏如何将 70B 模型的知识压缩到 1-3B 参数中。
- 区分四个边缘推理引擎：TensorFlow Lite（Android）、ONNX Runtime（跨平台）、Core ML（Apple）、WebNN（浏览器）。
- 说出 GGUF 作为边缘部署标准量化格式的地位。

## 问题

你的聊天应用每天有 1000 万用户。每个查询都发送到你的云端 LLM。月度推理账单：$80,000。用户抱怨"打字时卡顿" — 网络延迟增加了 200-500ms。隐私敏感用户询问"我的消息是否离开了我的设备？"

边缘推理解决了所有三个问题：在设备上本地运行模型消除了服务器成本、网络延迟和隐私顾虑。但代价是模型质量 — 一个 1B 参数的模型无法匹配 70B 模型的推理能力。

诀窍在于知道哪些工作负载可以迁移到边缘，哪些必须留在云端。

## 概念

### 设备端模型

2026 年，以下模型可以在旗舰手机（iPhone 16、Samsung S25）上以可用速度运行：

| 模型 | 大小 | 量化 | 设备速度 | 质量水平 |
|------|------|------|---------|---------|
| Llama 3.2 1B | 1B | INT4 (0.6 GB) | 25-30 tok/s | 基础 |
| Llama 3.2 3B | 3B | INT4 (1.8 GB) | 10-15 tok/s | 中等 |
| Gemma 3 1B | 1B | INT4 (0.6 GB) | 25-30 tok/s | 基础 |
| Phi-4-mini | 3.8B | INT4 (2.2 GB) | 8-12 tok/s | 中等+ |
| Stable LM 2 1.6B | 1.6B | INT4 (1.0 GB) | 15-20 tok/s | 基础+ |

这些模型可以处理：文本摘要、语法纠正、基本问答、关键词提取。它们无法处理：复杂的多步推理、长文档分析、代码生成。

### 模型蒸馏

蒸馏训练一个更小的"学生"模型来模仿一个更大的"教师"模型。流程：
1. 在目标任务上运行教师模型（例如 70B），收集输出。
2. 在教师输出（软标签）和真实标签（硬标签）的组合上训练学生模型（例如 1B）。
3. 学生学会匹配教师的输出分布，而不仅仅是正确答案。

结果：一个 1B 模型在特定任务上可以达到 70B 模型 80-90% 的质量。蒸馏是任务特定的 — 一个为摘要蒸馏的学生模型在代码生成上表现不佳。

### 边缘量化

GGUF 格式（第 17 阶段 · 09）是边缘部署的标准。Q4_K_M（4 位，中等大小）是默认推荐 — 良好的质量/大小权衡。边缘特定的约束：
- 总模型大小 < 4 GB（应用商店下载限制）。
- 推理内存 < 2 GB（与其他应用共享）。
- 电池消耗：持续推理每小时消耗 5-10% 的电池。

### 推理引擎

**TensorFlow Lite + XNNPACK**：Android 原生。XNNPACK 委托在 CPU 上提供优化的浮点和量化推理。还支持 GPU 委托（OpenGL ES）和 NPU 委托（Qualcomm Hexagon、MediaTek APU）。

**ONNX Runtime**：跨平台（Android、iOS、Windows、Linux、macOS）。执行提供程序：CPU（默认）、GPU（CUDA、OpenCL、Metal）、NPU（Qualcomm QNN、Apple CoreML）。适合：需要在多个平台上运行的团队。

**Core ML + ANE**：Apple 原生。ANE（Apple Neural Engine，Apple 神经引擎）是 iPhone/iPad/Mac 上的专用神经引擎硬件。与 CPU 相比，ANE 上的推理速度提升 2-5 倍。模型必须转换为 Core ML 格式（.mlmodelc）。

**WebNN**：浏览器 API。直接在浏览器中运行神经网络，利用设备 GPU/NPU。适合：基于 Web 的聊天应用、浏览器扩展。限制：模型必须 < 1 GB；浏览器支持仍在成熟中（Chrome、Edge 支持；Safari 和 Firefox 部分支持）。

### 边缘 vs 云端决策

| 工作负载 | 边缘还是云端 | 原因 |
|---------|------------|------|
| 语法纠正 | 边缘 | 简单任务，低延迟重要 |
| 文本摘要 | 边缘 | 中等复杂度，设备端模型可以处理 |
| 代码生成 | 云端 | 需要强大的推理能力 |
| 多步推理 | 云端 | 设备端模型无法处理 |
| 离线模式 | 边缘 | 无网络连接 |
| 隐私敏感数据 | 边缘 | 数据不离开设备 |

### 你应该记住的数字

- 设备端模型大小：1-4B 参数，INT4 量化后 0.6-2.5 GB。
- 设备端速度：10-30 token/秒（旗舰手机）。
- 蒸馏质量：1B 学生模型达到 70B 教师模型 80-90% 的质量。
- GGUF Q4_K_M：边缘部署的默认格式。
- ANE 加速：比 CPU 快 2-5 倍。
