# 影子流量、金丝雀发布与渐进式部署（面向 LLM）

> LLM 发布结合了软件部署中最困难的部分：没有单元测试、分散的失败模式、延迟的信号。顺序是（1）**影子模式（Shadow mode）** — 将生产请求复制到候选模型，记录日志，比较，零用户影响；能捕获明显的分布问题，但不是质量保证；（2）**金丝雀发布（Canary rollout）** — 渐进式流量切换 10% → 25% → 50% → 75% → 100%，每步设有检查点；跟踪延迟百分位数、成本/请求、错误/拒绝率、输出长度分布、用户反馈率；（3）**A/B 测试（A/B testing）** 用于稳定性确认后的不同替代方案。**非确定性（Non-determinism）** 是不可消除的 — 由于 GPU 浮点非结合性加上批次大小差异，相同输入在多次运行中准确率变化高达 15%。**成本是一个变量**，而非常量 — 一个 20% 更好的模型每次调用可能贵 3 倍。**回滚速度** 是决定性的：如果回滚需要重新部署，你就太慢了。策略存在于配置/标志中；模型存在于带有固定摘要的注册表中；回滚 = 翻转策略 + 恢复阈值 + 在几秒钟内固定旧模型。

**类型：** 学习
**语言：** Python（标准库，金丝雀推进模拟器）
**前置要求：** 第 17 阶段 · 13（可观测性）、第 17 阶段 · 21（A/B 测试）
**时间：** 约 60 分钟

## 学习目标

- 区分影子模式（零影响比较）、金丝雀（实时流量渐进式）和 A/B（稳定性确认后的比较）。
- 列举五个 LLM 特定的金丝雀指标（延迟、成本/请求、错误/拒绝、输出长度分布、用户反馈）。
- 解释为什么 LLM 非确定性（高达 15%）改变了发布中"稳定"的含义。
- 设计一个以秒为单位（策略翻转）而非小时（重新部署）的回滚路径。

## 问题

你发布了一个新模型。离线评估显示准确率提升 3%。你在生产环境中将其上线。24 小时内，成本上升 40%，用户点踩上升 8%，三张客户工单报告"奇怪的回答"。你回滚。重新部署需要 3 小时。你的周末被毁了。

所有这些都是可以避免的。影子模式会在任何用户看到之前捕获 40% 的成本飙升。金丝雀会在点踩变化时在 10% 处停止。策略标志回滚只需 30 秒。这门学科填补了"离线评估看起来不错"和"真实用户满意"之间的差距。

## 概念

### 影子模式

候选模型接收与生产环境相同的请求；输出被记录，不返回给用户。零用户影响。记录：

- 输出内容（与生产环境的差异）。
- Token 数量（成本差异）。
- 延迟。
- 拒绝和错误。

捕获：成本爆炸、长度退化、明显的拒绝变化、硬错误。不捕获：用户会感知的质量差异。影子是冒烟测试，不是质量测试。

### 金丝雀发布

渐进式流量切换，设有检查点。典型推进：1% → 10% → 25% → 50% → 75% → 100%。每步在 5 个指标上设检查点：

1. **延迟百分位数** — P50、P95、P99。违规：金丝雀 P99 > 基线的 1.5 倍。
2. **每请求成本** — 混合 $。违规：高于基线 > 20%。
3. **错误/拒绝率** — 5xx 加显式拒绝。违规：基线的 2 倍。
4. **输出长度分布** — 均值 + P99。违规：分布偏移。
5. **用户反馈率** — 点踩/工单提交。违规：基线的 1.5 倍。

### 非确定性是新的方差

相同输入产生不同输出。原因：

- GPU 浮点非结合性（浮点归约顺序因批次而异）。
- 批次大小差异（相同提示词在 128 的批次中 vs 16 的批次中）。
- 采样（temperature > 0）。

测量结果：在相同评估集上，多次运行的准确率变化高达 15%。发布中的"稳定"意味着指标在预期方差范围内，而非与基线完全相同。将检查点设在噪声底线之上。

### 成本是一个变量

一个 20% 更好的模型每次调用可能贵 3 倍。成本/请求是五个检查点之一。发布一个"更好"但破坏单位经济学的模型是回滚的理由。

### 回滚是武器

- 策略标志（功能标志系统）：在配置中翻转百分比；几秒钟内完成。
- 模型固定（注册表摘要）：固定的模型不会自动升级。
- 回滚 = 恢复标志 + 将固定摘要设置为上一个。几秒钟，而非几小时。

如果你的堆栈需要重新部署才能回滚，在发布之前先修复这个问题。

### 工具

**Argo Rollouts** / **Flagger** — Kubernetes 渐进式交付控制器。与 Istio/Linkerd 加权路由集成。

**Istio 加权路由** — 服务网格级别的流量分割。

**KServe / Seldon Core** — 模型服务，内置金丝雀。

**功能标志（Feature flags）** — LaunchDarkly、Flagsmith、Unleash。策略级别翻转，无需重新部署。

### 指标节奏

金丝雀检查点每 5-15 分钟检查一次，取决于流量大小。1% 流量，10 请求/分钟，每个窗口提供 50-150 个数据点 — 对延迟足够，但对用户反馈有噪声。10% 提供约 10 倍更多。推进应在每一步暂停足够长的时间以积累足够的样本。

### A/B 步骤是可选的

如果新模型明显不同（不同的行为、不同的成本曲线、不同的语调），在金丝雀通过后在 50% 处进行 A/B 测试。如果只是一个改进版本，当金丝雀检查点通过时跳到 100%。

### 你应该记住的数字

- 金丝雀推进：1% → 10% → 25% → 50% → 75% → 100%。
- 非确定性上限：相同输入下多次运行方差高达 15%。
- 五个金丝雀指标：延迟、成本、错误/拒绝、输出长度、用户反馈。
- 成本检查点：高于基线 > 20% 是违规。
- 回滚：几秒钟，而非几小时。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 影子模式（Shadow mode） | "复制到新模型" | 零影响发送到候选模型以进行日志记录 |
| 金丝雀（Canary） | "渐进式流量" | 带有检查点的渐进式用户暴露发布 |
| 检查点（Gates） | "发布检查" | 阻止推进的指标阈值 |
| 非确定性（Non-determinism） | "LLM 方差" | 不可消除的多次运行差异 |
| 策略标志（Policy flag） | "标志翻转回滚" | 配置级别回滚，几秒钟而非几小时 |
| 模型固定（Model pin） | "注册表摘要" | 模型版本的不可变引用 |
| Argo Rollouts | "K8s 渐进式" | Kubernetes 原生金丝雀/回滚控制器 |
| KServe | "推理 K8s" | 具有金丝雀原语的模型服务 |
| Istio 加权（Istio weighted） | "网格分割" | 服务网格流量分割器 |
