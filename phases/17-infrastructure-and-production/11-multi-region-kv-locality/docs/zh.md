# 多区域 KV 缓存局部性 — 跨区域复制、缓存亲和性与请求路由

> 当 LLM 服务部署在多个区域时，KV 缓存（Key-Value 缓存）局部性成为一个关键问题。用户的请求被路由到最近的区域，但他们的对话历史（KV 缓存）可能位于另一个区域。**跨区域 KV 缓存复制（Cross-region KV cache replication）** 将 KV 缓存从用户上次交互的区域复制到当前区域 — 这避免了重新计算预填充，但增加了跨区域带宽成本和复制延迟。**KV 缓存亲和性（KV cache affinity）** 将用户粘性路由到持有其 KV 缓存的区域 — 这避免了复制，但可能将用户路由到更远的区域（更高的网络延迟）。**请求路由（Request routing）** 必须平衡缓存局部性（将用户发送到持有其缓存的位置）和网络延迟（将用户发送到最近的位置）。2026 年的模式：对于多轮对话，缓存亲和性胜出 — 将用户路由到持有其 KV 缓存的区域，即使该区域不是最近的（KV 缓存重新计算成本 > 额外的网络延迟）。对于单轮请求，网络延迟胜出 — 路由到最近的区域（没有缓存可复用）。实现：第 17 阶段 · 19 的 AI 网关可以基于会话 ID 进行缓存感知路由；第 17 阶段 · 18 的 LMCache 可以跨区域复制 KV 缓存块。延迟与一致性之间的权衡：同步复制保证缓存一致性但增加延迟；异步复制更快但可能提供过时的缓存。

## 学习目标

- 解释多区域 LLM 服务中 KV 缓存局部性与网络延迟之间的权衡。
- 区分跨区域 KV 缓存复制（移动数据）和 KV 缓存亲和性（移动用户）。
- 设计一个缓存感知请求路由策略：多轮对话 → 亲和性；单轮 → 最近区域。
- 说出同步复制与异步复制在延迟和一致性方面的权衡。

## 问题

你的 LLM 服务在 us-east、eu-west 和 ap-southeast 运行。一个用户在 eu-west 进行了 10 轮对话。他们飞往新加坡。他们的下一个请求到达 ap-southeast。该区域没有他们的 KV 缓存。引擎重新计算全部 10 轮预填充 — 8K token × 10 轮 = 80K token 的预填充。TTFT（首 token 时间，Time To First Token）从 200ms 飙升到 4 秒。用户抱怨"聊天机器人变慢了"。

多区域 KV 缓存局部性不是理论问题 — 它是任何拥有移动用户群的全球 LLM 服务的实际问题。

## 概念

### 问题空间

三个相互竞争的约束：
1. **网络延迟**：用户应该被路由到最近的区域（最低延迟）。
2. **缓存局部性**：用户应该被路由到持有其 KV 缓存的区域（避免重新计算预填充）。
3. **一致性**：如果缓存被复制，它必须是一致的（过时的缓存可能导致不正确的输出）。

### 跨区域 KV 缓存复制

将 KV 缓存从源区域复制到目标区域。流程：
1. 用户在 eu-west 进行对话。KV 缓存存储在 eu-west。
2. 用户移动到新加坡。请求到达 ap-southeast。
3. ap-southeast 从 eu-west 拉取 KV 缓存（通过专用互连或对象存储）。
4. ap-southeast 加载 KV 缓存，继续对话。

成本：跨区域带宽（约 $0.02-0.12/GB）+ 复制延迟（50-200ms，取决于区域间距离）。收益：避免重新计算预填充（节省数秒的预填充时间）。

### KV 缓存亲和性

将用户粘性路由到持有其 KV 缓存的区域。流程：
1. 用户在 eu-west 进行对话。会话 ID 映射到 eu-west。
2. 用户移动到新加坡。请求到达 ap-southeast。
3. 网关检查会话 ID → 区域映射。将请求代理到 eu-west。
4. eu-west 处理请求，返回响应。

成本：额外的网络延迟（用户 → ap-southeast → eu-west → ap-southeast → 用户，可能增加 100-300ms）。收益：零复制成本，零一致性顾虑。

### 决策矩阵

| 场景 | 策略 | 原因 |
|------|------|------|
| 多轮对话，长上下文（10+ 轮） | 缓存亲和性 | 重新计算预填充 > 额外网络延迟 |
| 多轮对话，短上下文（2-3 轮） | 缓存复制或最近区域 | 重新计算成本较低 |
| 单轮请求 | 最近区域 | 没有缓存可复用 |
| 用户在同一区域 | 最近区域 | 缓存已经本地化 |

### 实现

**AI 网关（第 17 阶段 · 19）**：基于会话 ID 的缓存感知路由。网关维护会话 ID → 区域的映射（在 Redis 或类似数据库中）。当请求到达时，网关检查映射并将请求路由到正确的区域。

**LMCache（第 17 阶段 · 18）**：跨区域 KV 缓存复制。LMCache 服务器可以配置为将 KV 缓存块复制到其他区域的 LMCache 实例。使用对象存储（S3、GCS）作为异步复制的中间层。

**DNS/Anycast 与网关**：DNS 将用户路由到最近的区域。网关根据会话 ID 覆盖路由。两层路由：网络层（DNS）处理延迟；应用层（网关）处理缓存。

### 同步 vs 异步复制

**同步复制**：在确认请求之前，KV 缓存被复制到目标区域。保证一致性。增加延迟（等待复制完成）。适合：一致性关键型应用。

**异步复制**：KV 缓存在后台复制。更快（不阻塞请求）。可能提供过时的缓存（如果复制尚未完成）。适合：延迟关键型应用，其中过时的缓存优于无缓存。

### 你应该记住的数字

- 跨区域带宽成本：$0.02-0.12/GB。
- 复制延迟：50-200ms（区域间）。
- 缓存亲和性延迟惩罚：100-300ms（额外往返）。
- 预填充重新计算成本：80K token 约 2-4 秒。
- 决策规则：长对话 → 亲和性；短对话 → 最近区域。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| KV 缓存局部性（KV cache locality） | "缓存在哪里" | KV 缓存的物理位置 |
| 跨区域复制（Cross-region replication） | "复制缓存" | 将 KV 缓存复制到另一个区域 |
| 缓存亲和性（Cache affinity） | "粘性路由" | 将用户路由到持有其缓存的区域 |
| 请求路由（Request routing） | "将请求发送到哪里" | 平衡延迟和缓存局部性 |
| 同步复制（Synchronous replication） | "等待复制" | 在确认之前复制；一致性优先 |
| 异步复制（Asynchronous replication） | "后台复制" | 在后台复制；延迟优先 |
| 会话亲和性（Session affinity） | "粘性会话" | 同一会话的请求去往同一区域 |
| Anycast | "网络路由" | 将用户路由到最近区域的网络层 |
