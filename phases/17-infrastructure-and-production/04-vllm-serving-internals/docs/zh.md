# vLLM 服务内部原理 — PagedAttention、连续批处理与 KV 缓存管理

> vLLM 在 2026 年仍然是使用最广泛的开源 LLM 服务引擎。其核心创新是 **PagedAttention** — 受操作系统虚拟内存启发，将 KV 缓存（Key-Value 缓存）划分为固定大小的块（blocks），这些块可以在 GPU 内存中非连续地分配。这消除了传统连续 KV 缓存分配中的碎片化和预留浪费，使内存利用率从约 20-40% 提高到接近 90%。**连续批处理（Continuous batching）** 允许新请求在每次解码迭代时加入正在运行的批次，而不是等待整个批次完成 — 这是静态批处理与连续批处理之间的关键区别。**块管理器（Block manager）** 处理分配、驱逐和写时复制（copy-on-write），用于跨请求共享前缀。**调度器（Scheduler）** 决定每个解码步骤处理哪些请求，平衡延迟（不能饿死短请求）和吞吐量（用长请求填充批次）。预填充（prefill）阶段处理整个输入提示词并生成第一个 token；解码（decode）阶段每次迭代生成一个 token。KV 缓存内存布局：每个块存储一个层中最多 N 个 token 位置的键和值张量；块大小（block_size）是一个关键调优参数 — 较大的块减少管理开销但增加内部碎片。

## 学习目标

- 解释 PagedAttention 如何消除 KV 缓存碎片化，以及为什么连续分配浪费 60-80% 的内存。
- 绘制块管理器架构：分配、驱逐、写时复制、前缀共享。
- 区分预填充（计算密集型）和解码（内存带宽密集型）阶段。
- 描述连续批处理如何允许请求在每次解码迭代时加入，而不是等待批次完成。

## 问题

在 PagedAttention 之前，服务引擎为每个请求的最大可能序列长度分配一个连续的 KV 缓存块。一个最大 8K token 的请求即使只生成 200 个 token 也会预留 8K 的内存。10 个并发请求 = 10 × 8K = 80K 的预留空间，但实际使用量是 10 × 200 = 2K。97.5% 的分配内存被浪费。

这就是为什么 2022 年的 LLM 服务在 80 GB HBM 的 A100 上只能处理 8-12 个并发请求。PagedAttention 将可用并发数提高了 5-10 倍，通过消除预留浪费。

## 概念

### PagedAttention — 虚拟内存之于 KV 缓存

操作系统将物理内存划分为页面（pages），将虚拟地址映射到非连续的物理页面。PagedAttention 对 KV 缓存做了同样的事情：

- **块（Block）** = 一个层中固定数量 token 位置的键和值张量。
- **块表（Block table）** = 将逻辑 token 位置映射到物理块。
- **分配** = 按需分配块；无需预留最大长度。
- **碎片化** = 仅最后一个块有内部碎片（平均半个块）。

结果：内存利用率从约 20-40% 提高到接近 90%。一个 80 GB 的 A100 可以处理 50-100+ 个并发请求，而不是 8-12 个。

### 块管理器

**分配**：当请求到达时，分配一个块用于预填充。随着生成更多 token，按需追加块。

**驱逐**：当 GPU 内存不足时，驱逐最近最少使用的块。被驱逐的块可以交换到 CPU 内存（第 17 阶段 · 18 的 LMCache）。

**写时复制（Copy-on-write）**：当多个请求共享相同的前缀（例如系统提示词）时，它们引用相同的物理块。只有当请求写入（即生成不同的续写）时，块才会被复制。这对 RAG 和对话系统至关重要，其中系统提示词在所有请求中重复出现。

**前缀共享**：块管理器自动检测相同的提示词前缀并共享块。vLLM 0.6.0+ 通过基于哈希的前缀匹配实现了自动前缀缓存。

### 预填充 vs 解码

**预填充（Prefill）**：一次性处理整个输入提示词。计算密集型 — 矩阵乘法占主导。生成第一个 token 和完整的 KV 缓存。延迟与提示词长度成正比。

**解码（Decode）**：每次迭代生成一个 token。内存带宽密集型 — 每个 token 需要从 HBM 读取全部模型权重。延迟与模型大小成正比，与提示词长度无关。

连续批处理在预填充和解码之间交替进行。调度器在每个解码步骤将新请求的预填充混入批次中。

### 连续批处理

静态批处理：等待所有请求完成后再开始新的请求。一个生成了 500 个 token 的请求会阻塞一个生成了 5 个 token 的请求。

连续批处理：在每次解码迭代时，调度器选择一组请求进行处理。新请求可以在任何迭代中加入。已完成的请求退出。批次大小随请求到达和完成而动态变化。

对延迟的影响：短请求不会饿死在长请求后面。对吞吐量的影响：GPU 始终保持高利用率，因为批次永远不会因为等待最慢的请求而空闲。

### 调度策略

vLLM 的调度器使用优先级队列：

1. 首先调度预填充（以避免解码批次饿死）。
2. 在解码请求中，优先处理短队列（较少的待生成 token）。
3. 用长请求填充剩余批次槽位。

这平衡了延迟（短请求快速完成）和吞吐量（长请求保持 GPU 忙碌）。

### 块大小调优

较大的块（例如 32 或 64 个 token）：更少的块需要管理，更少的块表开销。更多的内部碎片（平均浪费半个块）。

较小的块（例如 8 或 16 个 token）：更少的内部碎片。更多的块需要管理，更大的块表。

vLLM 默认值：16（对大多数工作负载是合理的折中）。对于非常长的上下文（128K+），较大的块减少管理开销。

### 你应该记住的数字

- PagedAttention 内存利用率：约 90%（对比连续分配的 20-40%）。
- 并发请求提升：5-10 倍。
- 块大小默认值：16 个 token。
- 预填充：计算密集型。解码：内存带宽密集型。
- 连续批处理：请求在每次解码迭代时加入/退出。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| PagedAttention | "虚拟内存之于 KV" | 将 KV 缓存划分为非连续块 |
| 连续批处理（Continuous batching） | "动态批次" | 请求在每次迭代时加入/退出批次 |
| 块管理器（Block manager） | "内存分配器" | 分配、驱逐、写时复制 KV 块 |
| 预填充（Prefill） | "处理输入" | 一次性处理整个提示词 |
| 解码（Decode） | "生成 token" | 每次迭代生成一个 token |
| 块表（Block table） | "映射表" | 将逻辑位置映射到物理块 |
| 写时复制（Copy-on-write） | "共享直到写入" | 共享块直到请求产生分歧 |
| 前缀共享（Prefix sharing） | "缓存提示词" | 跨请求共享相同的提示词前缀 |
| 调度器（Scheduler） | "批次选择器" | 决定每个步骤处理哪些请求 |
| 块大小（Block size） | "块中的 token 数" | 关键调优参数；默认 16 |
