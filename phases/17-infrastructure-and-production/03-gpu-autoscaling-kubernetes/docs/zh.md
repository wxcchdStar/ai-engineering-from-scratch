# Kubernetes 上的 GPU 自动扩缩容 — KEDA、HPA、VPA、集群自动扩缩器与预热

> 2026 年 Kubernetes 上的 GPU 自动扩缩容使用分层方法。**KEDA（Kubernetes 事件驱动自动扩缩容，Kubernetes Event-Driven Autoscaling）** 是 GPU 工作负载的主要扩缩容器 — 它根据自定义指标（队列深度、请求延迟、GPU 利用率）进行扩缩容，并支持缩容到零。**HPA（水平 Pod 自动扩缩容，Horizontal Pod Autoscaler）** 处理基于 CPU/内存的扩缩容，但缺乏 GPU 指标感知能力。**VPA（垂直 Pod 自动扩缩容，Vertical Pod Autoscaler）** 调整资源请求但不调整 GPU 数量 — 对 GPU 工作负载用处有限。**集群自动扩缩器（Cluster Autoscaler）** 或 **Karpenter** 在 Pod 因 GPU 不足而无法调度时添加 GPU 节点。GPU 特定技术：**时间切片（Time-slicing）** 允许多个 Pod 共享一个 GPU（适合开发/测试，不适合生产）；**MIG（多实例 GPU，Multi-Instance GPU）** 将 A100/H100 硬件分区为隔离的 GPU 切片（适合生产）。冷启动是 GPU 扩缩容的主要问题 — 节点配置（2-4 分钟）+ 容器镜像拉取（1-3 分钟）+ 模型加载（30 秒-5 分钟）。缓解措施：**预热池（Warm pools）**（保持预配置节点处于待机状态）、**模型预加载**（将模型权重放在持久卷上）、**镜像缓存**（在每个节点上预拉取容器镜像）。Karpenter 通过根据 GPU 类型和可用区进行合并配置来减少冷启动时间。

## 问题

你的 vLLM 部署在 4 个 GPU 节点上运行。流量在凌晨 2 点降到零，在上午 10 点飙升到 500 RPS。凌晨 2 点你在为闲置 GPU 付费。上午 10 点你在丢请求，因为扩缩容跟不上。你手动将最小 Pod 数设为 4 以处理峰值 — 现在你全天都在为峰值容量付费。

GPU 自动扩缩容比 CPU 自动扩缩容更难，因为 GPU 是稀缺资源、冷启动时间长，且模型加载不是即时的。做对了，你可以在低谷时段缩容到零，并在不丢请求的情况下处理峰值。做错了，你要么为闲置 GPU 付费，要么在峰值时丢流量。

## 概念

### 分层扩缩容架构

```
请求 → KEDA ScaledObject（队列深度 > 阈值？）
         │
         ▼ 扩容
      HPA（CPU/内存作为备用信号）
         │
         ▼ Pod 无法调度（GPU 不足）
集群自动扩缩器 / Karpenter（添加 GPU 节点）
         │
         ▼ 节点就绪
    Pod 调度 → 模型加载 → 就绪
```

**KEDA**：根据自定义指标进行扩缩容。对于 LLM 服务：队列深度（等待请求数）、请求延迟 P99、GPU 利用率。支持缩容到零 — 当没有请求时，没有 Pod 在运行。这是 GPU 工作负载的主要扩缩容器。

**HPA**：根据 CPU/内存进行扩缩容。对 GPU 工作负载用处有限，因为 GPU 利用率不通过标准指标 API 暴露。作为 KEDA 的备用方案。

**VPA**：调整 CPU/内存请求。不调整 GPU 数量。对 GPU 工作负载基本不相关。

**集群自动扩缩器 / Karpenter**：当 Pod 因资源不足而无法调度时添加节点。Karpenter 更快（秒级 vs 分钟级），因为它直接管理 EC2 Fleet/VMSS，并可以根据 GPU 类型（A100、H100）和可用区进行合并配置。

### GPU 共享策略

**时间切片（Time-slicing）**：多个 Pod 在单个 GPU 上分时运行。NVIDIA GPU Operator 原生支持。开销：上下文切换约 5-10%。适合：开发/测试、低流量服务。不适合：生产，因为缺乏内存隔离 — 一个 Pod 可能 OOM（内存溢出，Out Of Memory）并杀死其他 Pod。

**MIG（多实例 GPU）**：硬件分区。A100 可以分为最多 7 个 GPU 实例；H100 最多 7 个。每个实例有隔离的内存和计算。适合：生产多租户。限制：不如时间切片灵活；实例大小是固定的。

### 冷启动阶段

1. **节点配置**（2-4 分钟）：云提供商配置 VM、附加 GPU、加入集群。Karpenter 通过预配置 AMI（Amazon 机器镜像，Amazon Machine Image）和启动模板将其缩短到约 60 秒。

2. **容器镜像拉取**（1-3 分钟）：vLLM 镜像约 8-15 GB。缓解措施：在每个节点上预缓存镜像（使用 DaemonSet 或节点初始化脚本）；使用镜像流式传输（SOCI、eStargz）。

3. **模型加载**（30 秒-5 分钟）：将模型权重从磁盘/对象存储加载到 GPU 内存。Llama 3.3 70B FP8 约 70 GB — 从 NVMe 加载约 30 秒，从 S3 加载约 3-5 分钟。缓解措施：将模型权重放在持久卷（Persistent Volume）上；使用模型预热（在 Pod 启动时预加载权重）。

### 预热策略

**预热池（Warm pool）**：保持 N 个预配置节点处于待机状态，已安装 GPU 驱动和容器镜像。Karpenter 可以通过最小节点数配置来维护预热池。成本：为闲置节点付费，但消除了 2-4 分钟的节点配置延迟。

**模型预加载**：将模型权重放在节点本地 NVMe 或持久卷上。Pod 启动时，模型已存在于本地磁盘上。将模型加载时间从 3-5 分钟（S3）减少到 30 秒（本地 NVMe）。

**保持连接（Keep-alive）**：将最小 Pod 数设为 1，即使没有流量。消除冷启动，但为闲置 GPU 付费。适合：延迟关键型应用，其中 TTFT SLA < 500ms。

### 你应该记住的数字

- 节点配置：2-4 分钟（Karpenter 约 60 秒）。
- 镜像拉取：1-3 分钟（缓存后 < 10 秒）。
- 模型加载：30 秒（NVMe）到 5 分钟（S3）。
- 时间切片：适合开发，不适合生产。
- MIG：A100/H100 最多 7 个硬件隔离分区。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| KEDA | "事件驱动扩缩容" | Kubernetes 事件驱动自动扩缩容器 |
| HPA | "CPU 扩缩容" | 水平 Pod 自动扩缩容，基于 CPU/内存 |
| VPA | "垂直扩缩容" | 调整资源请求，不调整 GPU 数量 |
| 集群自动扩缩器（Cluster Autoscaler） | "节点扩缩容" | 添加/移除节点 |
| Karpenter | "更快的自动扩缩器" | AWS 原生的快速节点配置器 |
| 时间切片（Time-slicing） | "GPU 共享" | 多个 Pod 分时共享一个 GPU |
| MIG | "GPU 分区" | 多实例 GPU，硬件隔离 |
| 冷启动（Cold start） | "从零扩容" | 节点配置 + 镜像拉取 + 模型加载 |
| 预热池（Warm pool） | "待机节点" | 预配置节点，随时可用 |
| 模型预加载（Model preloading） | "本地权重" | 模型权重存储在本地磁盘上 |
