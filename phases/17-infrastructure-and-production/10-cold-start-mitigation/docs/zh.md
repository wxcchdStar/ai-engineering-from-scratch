# 冷启动缓解 — 模型预加载、预热池、JIT 编译缓存与容器镜像优化

> LLM 服务的冷启动延迟是 GPU 自动扩缩容面临的最大挑战。冷启动有三个阶段：**节点配置**（云提供商配置 VM 并附加 GPU，2-4 分钟）、**容器镜像拉取**（下载 8-15 GB 的 vLLM/TGI 镜像，1-3 分钟）、**模型加载**（将权重从磁盘/对象存储加载到 GPU 内存，30 秒-5 分钟）。缓解策略形成一个分层防御：**模型预加载** 将权重存储在本地 NVMe 或持久卷上（将模型加载从 3-5 分钟减少到 30 秒）。**预热池（Warm pools）** 保持预配置节点处于待机状态，已安装 GPU 驱动和容器镜像（消除节点配置 + 镜像拉取延迟）。**JIT 编译缓存（JIT compilation caching）** 持久化 TRT-LLM 或 PyTorch 的编译产物，使后续启动跳过编译（将 TRT-LLM 启动从数分钟减少到数秒）。**容器镜像优化** 使用精简基础镜像、分层缓存和镜像流式传输（SOCI、eStargz）来减少拉取时间。**保持连接（Keep-alive）** 将最小 Pod 数设为 1，即使没有流量 — 消除冷启动，但为闲置 GPU 付费。策略选择取决于延迟 SLO（服务等级目标）和成本约束 — 预热池增加闲置成本但保证延迟；缩容到零节省成本但增加峰值延迟。

## 问题

凌晨 3 点，流量降到零。你的 KEDA 扩缩容将 vLLM Pod 缩容到零。凌晨 6 点，流量恢复。第一个请求触发扩缩容：节点配置（3 分钟）+ 镜像拉取（2 分钟）+ 模型加载（4 分钟）= 9 分钟的冷启动。前 50 个请求超时。你的 P99 TTFT（首 token 时间，Time To First Token）警报触发。值班工程师被吵醒。

冷启动是 GPU 自动扩缩容中排名第一的运维痛点。解决方案不是"不要缩容到零" — 那是为闲置 GPU 付费。解决方案是分层缓解，使冷启动足够快，以至于用户注意不到。

## 概念

### 冷启动的三个阶段

**阶段 1：节点配置**（2-4 分钟）。云提供商配置 VM、附加 GPU、将节点加入 Kubernetes 集群。Karpenter 通过预配置 AMI（Amazon 机器镜像，Amazon Machine Image）和启动模板将其缩短到约 60 秒。无法完全消除 — 物理硬件需要时间来配置。

**阶段 2：容器镜像拉取**（1-3 分钟）。vLLM 镜像约 8-15 GB（包括 CUDA 运行时、Python 依赖、vLLM 代码）。缓解措施：在每个节点上预缓存镜像（DaemonSet 或节点初始化脚本）；使用镜像流式传输（SOCI、eStargz — 按需拉取文件，而不是整个镜像）。

**阶段 3：模型加载**（30 秒-5 分钟）。将模型权重从磁盘加载到 GPU 内存。Llama 3.3 70B FP8 约 70 GB。从本地 NVMe 加载：约 30 秒（7 GB/s 读取速度）。从 S3/对象存储加载：3-5 分钟（受网络带宽限制）。缓解措施：将权重存储在本地 NVMe 或持久卷上。

### 分层缓解策略

**第 1 层：模型预加载**。将模型权重存储在节点本地 NVMe 或持久卷上。Pod 启动时，模型已存在于本地磁盘上。将阶段 3 从 3-5 分钟减少到 30 秒。成本：NVMe 存储费用（约 $0.08/GB/月）。

**第 2 层：预热池**。保持 N 个预配置节点处于待机状态，已安装 GPU 驱动和容器镜像。消除阶段 1 和阶段 2。成本：为闲置节点付费（按需 GPU 实例价格）。权衡：预热池大小 — 较大的池子 = 更高的成本但更好的峰值处理能力。

**第 3 层：JIT 编译缓存**。TRT-LLM 和 PyTorch 2.0+ 在首次运行时编译 CUDA 内核。将编译产物缓存到持久卷上。后续启动跳过编译。将 TRT-LLM 启动从数分钟减少到数秒。

**第 4 层：容器镜像优化**。使用精简基础镜像（distroless、alpine）。分层缓存 — 将频繁更改的层放在最后。镜像流式传输（SOCI、eStargz）— 按需拉取文件，而不是整个镜像。将阶段 2 从 1-3 分钟减少到 < 10 秒（缓存命中时）。

**第 5 层：保持连接**。将最小 Pod 数设为 1。消除所有冷启动。成本：为 1 个始终运行的 GPU 付费（约 $1-3/小时，取决于 GPU 类型）。

### 策略选择

| 策略 | 冷启动延迟 | 闲置成本 | 最适合 |
|------|-----------|---------|--------|
| 缩容到零 + 无缓解 | 5-12 分钟 | $0 | 非关键批处理 |
| 缩容到零 + 预加载 + 预热池 | 30-60 秒 | 低（预热池） | 成本敏感的在线服务 |
| 最小 Pod 数 = 1 | 0 秒 | 高（1 个 GPU 始终运行） | 延迟关键的在线服务 |
| 预热池 + 保持连接 | 0 秒 | 最高 | 企业级 SLA |

### 你应该记住的数字

- 节点配置：2-4 分钟（Karpenter 约 60 秒）。
- 镜像拉取：1-3 分钟（缓存后 < 10 秒）。
- 模型加载：30 秒（NVMe）到 5 分钟（S3）。
- 分层缓解：预加载 + 预热池 + JIT 缓存 + 镜像优化 + 保持连接。
- 预热池成本：为闲置 GPU 节点付费。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 冷启动（Cold start） | "从零扩容" | 节点配置 + 镜像拉取 + 模型加载 |
| 模型预加载（Model preloading） | "本地权重" | 模型权重存储在本地磁盘上 |
| 预热池（Warm pool） | "待机节点" | 预配置节点，随时可用 |
| JIT 编译缓存（JIT compilation caching） | "缓存内核" | 持久化编译后的 CUDA 内核 |
| SOCI | "AWS 流式传输" | 按需拉取容器镜像文件 |
| eStargz | "懒拉取" | 容器镜像的懒加载 |
| 保持连接（Keep-alive） | "最小 Pod 数 = 1" | 始终运行至少一个 Pod |
| 持久卷（Persistent Volume） | "K8s 存储" | 跨 Pod 重启持久化的存储 |
| NVMe | "本地 SSD" | 高速本地存储（7 GB/s 读取） |
