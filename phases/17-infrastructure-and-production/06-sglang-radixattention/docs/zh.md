# SGLang RadixAttention — 基数树 KV 缓存复用与自动前缀缓存

> SGLang 的 **RadixAttention** 使用基数树（radix tree）数据结构来组织和复用 KV 缓存（Key-Value 缓存）前缀。与 vLLM 基于哈希的前缀匹配不同，基数树自动检测共享前缀，即使它们不完全相同 — "法国的首都是什么？" 和 "法国的首都是什么？巴黎有哪些景点？" 共享前缀 "法国的首都是什么？"，RadixAttention 自动复用第一个查询的 KV 缓存用于第二个查询。这使 SGLang 成为多轮对话和 agent 工作负载的首选引擎，其中前缀复用率很高。**自动前缀缓存（Automatic prefix caching）** 意味着开发者无需显式管理缓存 — 引擎自动检测和复用共享前缀。基准测试显示，在多轮对话基准测试中，相对于无前缀缓存的基线，吞吐量提升 2-3 倍。SGLang 在 2026 年拥有 400,000+ GPU 的生产部署（xAI、LinkedIn、Cursor、Oracle、GCP、Azure、AWS）。基数树结构还支持高效的缓存驱逐 — 驱逐整个子树（例如一个会话的所有轮次），而不是逐个驱逐。

## 问题

用户问："法国的首都是什么？" 你的 LLM 处理提示词，生成 KV 缓存，回答"巴黎"。用户追问："巴黎有哪些景点？" 你的 LLM 处理整个对话历史 — 包括"法国的首都是什么？" — 从头开始。你已经计算过的前缀的 KV 缓存被丢弃了。

在多轮对话中，每个后续轮次都包含所有之前的轮次。如果不复用，你在第 10 轮为前 9 轮重新计算 KV 缓存。使用前缀缓存，你只计算新内容。

SGLang 的 RadixAttention 通过将 KV 缓存组织成基数树来自动化这一过程。共享前缀被自动检测和复用。开发者无需显式管理缓存键或失效策略。

## 概念

### 基数树数据结构

基数树（也称为 Patricia trie 或压缩前缀树）是一种空间优化的 trie，其中具有单一子节点的节点与其父节点合并。在 RadixAttention 中：

- 每个节点存储一个 token 序列的 KV 缓存。
- 边表示 token 序列。
- 查找：沿着树向下遍历，匹配输入 token。
- 插入：为新的 token 序列添加节点，在分叉点分裂现有节点。

示例：
```
根
└── "法国的首都是什么？" [KV 缓存]
    └── "巴黎有哪些景点？" [KV 缓存]
        └── "埃菲尔铁塔开放时间？" [KV 缓存]
```

当"法国的首都是什么？巴黎有哪些景点？埃菲尔铁塔开放时间？"到达时，引擎沿着树向下遍历，找到前两个节点的缓存 KV，只计算"埃菲尔铁塔开放时间？"。

### 自动前缀缓存

vLLM 基于哈希的方法：引擎对提示词前缀进行哈希，检查哈希表是否有匹配。精确匹配或没有 — 部分重叠不被检测到。

RadixAttention 基于树的方法：引擎沿着树向下遍历，匹配尽可能多的 token。部分重叠被自动检测到。"法国的首都是什么？巴黎" 匹配 "法国的首都是什么？" 节点加上 "巴黎" 的部分匹配。

自动意味着开发者无需：
- 标记哪些提示词是可缓存的。
- 管理缓存键。
- 编写驱逐策略。

引擎处理所有这些。

### 多轮对话优化

在多轮对话中：
- 第 1 轮：计算完整提示词的 KV 缓存。存储在基数树中。
- 第 2 轮：复用第 1 轮的 KV 缓存，只计算新用户消息和助手响应。
- 第 N 轮：复用前 N-1 轮的 KV 缓存，只计算增量。

对于 10 轮对话，每轮 500 个 token：无缓存 = 10 × 500 = 5,000 次预填充。有缓存 = 500 + 9 × 50（增量）= 950 次预填充。约 5 倍的预填充计算减少。

### 吞吐量提升

已发布的基准测试显示，在多轮对话基准测试中，相对于无前缀缓存的基线，吞吐量提升 2-3 倍。确切的提升取决于：
- 对话长度（更长的对话 = 更多的复用）。
- 轮次长度分布（一致的轮次长度 = 更好的缓存命中率）。
- 并发用户数（更多的用户 = 更多的缓存压力）。

### 缓存驱逐

基数树支持高效的子树驱逐。当一个会话结束时，其整个子树可以被驱逐。驱逐策略：
- LRU（最近最少使用，Least Recently Used）：驱逐最旧的子树。
- TTL（生存时间，Time To Live）：驱逐超过 N 分钟未访问的子树。
- 内存压力：当 GPU 内存不足时，驱逐最大的子树。
