# LLM 的 FinOps — 单位经济学与多租户归因

> 传统 FinOps（云财务运营，Cloud Financial Operations）在 LLM 支出上失效。成本是 token 交易，而非资源运行时间。标签不映射 — API 调用是交易，而非资产。工程决策（提示词设计、上下文窗口、输出长度）就是财务决策。2026 年的策略手册有三个归因维度，需要在第一天就进行仪器化：按用户（`user_id`）用于席位定价和扩展，按任务（`task_id` + `route`）用于产品表面成本和优先级排序，按租户（`tenant_id`）用于单位经济学和续约。四个 token 层 — 提示词、工具、记忆、响应 — 一个桶隐藏了成本。多租户产品的执行阶梯：按租户限流（预期峰值的 2-3 倍，明确的 429 + retry-after）；每日支出上限（合同上限的 1.5-3 倍；触发限流收紧 + 告警）；支出 z-score > 4 时的终止开关（自动暂停 + 呼叫值班）。归因模式：标签聚合、遥测连接器（trace-ID → 计费；最高准确率）、采样外推、基于模型的分配、事件溯源、实时流式。单位指标：每解决查询的成本、每生成工件的成本 — 而非 $/M token。追溯性标记总是遗漏；在请求创建时进行仪器化。

**类型：** 学习
**语言：** Python（标准库，成本归因模拟器，带终止开关）
**前置要求：** 第 17 阶段 · 13（可观测性）、第 17 阶段 · 14（缓存）
**时间：** 约 60 分钟

## 学习目标

- 解释为什么传统 FinOps（标签 + 层级）在 LLM 支出上失效，并说出三个新的归因维度。
- 列举四个 token 层（提示词、工具、记忆、响应）以及为什么单桶计费隐藏了成本。
- 为多租户产品设计执行阶梯（限流 → 支出上限 → 终止开关）。
- 选择单位指标（每解决查询/工件的成本）而非 $/M token。

## 问题

你的账单显示 $40,000。你不知道：
- 哪个租户花了这笔钱。
- 哪个产品功能驱动了它。
- 是否有任何单个用户是滥用者。
- 是提示词膨胀、工具调用还是记忆放大是罪魁祸首。

供应商端的标签聚合适用于云资源（EC2、S3），其中标签传播到行项目。LLM API 调用不会自动标记 — 你必须在调用点标记 user/task/tenant 并贯穿始终。追溯性归因总是遗漏边缘情况。

## 概念

### 三个归因维度

**按用户（Per-user）**（`user_id`）：谁花费了什么。驱动席位定价、扩展对话、识别重度用户。

**按任务（Per-task）**（`task_id` + `route`）：哪个产品表面花费了什么。驱动功能优先级排序、终止昂贵功能的决策。

**按租户（Per-tenant）**（`tenant_id`）：哪个客户是盈利的。驱动单位经济学、续约定价、层级阈值。

在第一天就在调用点对所有三个进行仪器化。追溯性总是更差。

### 四个 token 层

| 层 | 示例 | 典型占比 |
|----|------|---------|
| 提示词（Prompt） | 系统 + 用户输入 | 40-60% |
| 工具（Tool） | 反馈的工具调用结果 | 20-40%（agent 工作负载） |
| 记忆（Memory） | 先前对话/检索到的文档 | 10-30% |
| 响应（Response） | 模型输出 | 10-30% |

将四者全部放入一个桶会使优化变得盲目。在你的归因模式中将它们分开。

### 执行阶梯

1. **限流（Rate limit）** 按租户。预期峰值的 2-3 倍。返回 429 并带 `Retry-After`。租户看到摩擦；没有意外账单。

2. **每日支出上限（Daily spend cap）** 按租户。合同上限的 1.5-3 倍。触发：收紧限流 + 告警客户成功。

3. **终止开关（Kill switch）** 在支出 z-score > 4 时（相对于租户基线）。自动暂停租户；呼叫值班；升级到运维 + 客户成功。

### 归因模式

- **标签聚合（Tag-and-aggregate）**：标记元数据头；稍后聚合。简单；粗糙。
- **遥测连接器（Telemetry joiner）**：通过 trace ID 将追踪连接到计费。最高准确率。成熟团队的做法。
- **采样 + 外推（Sampling + extrapolation）**：采样 5-10%，相乘。对粗略支出具有成本效益；遗漏尾部。
- **基于模型的分配（Model-based allocation）**：回归推断成本驱动因素。用于没有标签的遗留数据。
- **事件溯源（Event-sourced）**：成本作为流中的事件（Kafka/Kinesis）。实时。
- **实时流式（Real-time streaming）**：仪表盘亚秒级更新。

### 每 X 成本是单位指标

$/M token 是供应商的说法。产品指标：

- 每解决支持工单的成本。
- 每生成文章的成本。
- 每成功 agent 任务的成本。
- 每用户会话分钟的成本。

将成本与产品结果挂钩。否则优化是无锚的。

### 成本归因追踪形态

```
trace_id: abc123
  user_id: u_42
  tenant_id: t_7
  task_id: task_classify_doc
  route: model_haiku
  layers:
    prompt_tokens: 1800
    tool_tokens: 600
    memory_tokens: 400
    response_tokens: 150
  cost_usd: 0.0135
  cached_input: true
  batch: false
```

在每次调用时发出。存储在数据湖中。按维度聚合。第 17 阶段 · 13 可观测性堆栈是它所在的地方。

### 复合节省堆栈

堆栈：缓存 + 批量 + 路由 + 网关。四者全部使用：
- 缓存 L2（第 17 阶段 · 14）：输入便宜约 10 倍。
- 批量（第 17 阶段 · 15）：50% 折扣。
- 路由到廉价模型（第 17 阶段 · 16）：成本降低 60%。
- 网关效率（第 17 阶段 · 19）：冗余 + 重试。

最佳情况叠加：约为原始基线的 5-10%。大多数团队使用了 2-3 个杠杆；很少有团队叠加全部四个。

### 你应该记住的数字

- 归因维度：按用户、按任务、按租户。
- 四个 token 层：提示词、工具、记忆、响应。
- 终止开关：支出 z-score > 4。
- 单位指标：每解决查询的成本，而非 $/M token。
- 叠加优化：可能达到基线的约 5-10%。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 按用户归因（Per-user attribution） | "用户级成本" | 每次调用标记 `user_id` |
| 按任务归因（Per-task attribution） | "功能成本" | `task_id` + `route` 标识产品表面 |
| 按租户归因（Per-tenant attribution） | "客户成本" | `tenant_id`；驱动单位经济学 |
| 四个 token 层（Four token layers） | "成本层" | 提示词 + 工具 + 记忆 + 响应 |
| 限流（Rate limit） | "429 guard" | 在网关强制执行的按租户上限 |
| 每日支出上限（Daily spend cap） | "每日上限" | 租户范围的预算，带告警 |
| 终止开关（Kill switch） | "自动暂停" | 支出 z-score > 4 触发自动暂停 |
| 每解决成本（Cost per resolved） | "产品单位指标" | 成本与产品结果挂钩，而非 token |
| 遥测连接器（Telemetry joiner） | "追踪到计费" | 最高准确率的归因模式 |
| 叠加优化（Stacked optimization） | "缓存+批量+路由+网关" | 复合节省至基线约 5-10% |
