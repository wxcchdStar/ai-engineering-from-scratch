# 提示词与语义缓存 — 精确匹配、嵌入相似度与失效策略

> LLM 缓存（Caching）在 2026 年分为两个层级。**精确匹配缓存（Exact match caching）** 对相同的提示词返回相同的响应 — 简单、快速，但仅对逐字相同的提示词有效。**语义缓存（Semantic caching）** 使用嵌入相似度（embedding similarity）来匹配语义上相似的提示词 — "法国的首都是什么？" 和 "告诉我法国的首都" 的余弦相似度 > 0.95，并命中相同的缓存条目。语义缓存将缓存命中率从 5-15%（仅精确匹配）提高到 40-60%（语义匹配）。**缓存命中率（Cache hit rate）** 是主要指标 — 更高的命中率 = 更低的延迟和成本。**TTL（生存时间，Time To Live）** 控制条目在缓存中保留的时间 — 事实性问答的 TTL 较长（事实变化缓慢）；实时数据的 TTL 较短（股票价格、天气）。**失效策略（Invalidation strategies）** 处理模型更新（新模型版本 → 刷新缓存）和事实变化（已知事实已更改 → 驱逐特定条目）。实现：Redis 用于精确匹配缓存（微秒级延迟）；向量数据库（Milvus、Pinecone、Qdrant）用于语义缓存（毫秒级延迟）。成本影响：缓存命中将输入 token 成本降低 10-20 倍（缓存输入定价 vs 未缓存输入定价），并将 TTFT（首 token 时间，Time To First Token）降低 50-90%（跳过预填充）。

## 学习目标

- 区分精确匹配缓存（逐字相同）和语义缓存（嵌入相似度），并量化命中率差异。
- 解释 TTL 如何因数据类型而异（事实 vs 实时），以及失效策略如何处理模型更新。
- 计算缓存命中的成本节省：输入 token 成本降低 10-20 倍。
- 设计一个包含 Redis（精确匹配）+ 向量数据库（语义匹配）的两层缓存架构。

## 问题

你的客户支持聊天机器人每小时收到 10,000 个查询。分析显示 40% 的查询是语义重复的 — "如何重置密码？"、"重置密码的方法"、"我忘了密码怎么办" — 但措辞不同。精确匹配缓存只捕获了 8%。你每小时为相同的答案向 OpenAI 支付了 4,000 次费用。

语义缓存将这 4,000 次调用减少到约 400 次（90% 的语义缓存命中率）。按每 1K token $3 计算，每天节省约 $250。每年约 $90,000。仅通过添加一个缓存层。

## 概念

### 两层缓存架构

**第 1 层：精确匹配缓存（Redis）**。键 = SHA-256（提示词）。值 = 响应 + 元数据。延迟：< 1ms。命中率：5-15%（取决于提示词模板化程度）。适合：模板化提示词、系统消息、常见问答对。

**第 2 层：语义缓存（向量数据库）**。键 = 嵌入向量（提示词）。值 = 响应 + 元数据。延迟：5-20ms。命中率：40-60%（与精确匹配结合）。适合：用户生成的自然语言查询、语义相似的变体。

### 语义缓存如何工作

1. 计算输入提示词的嵌入向量（使用小型嵌入模型，如 `all-MiniLM-L6-v2`）。
2. 在向量数据库中搜索相似向量（余弦相似度 > 阈值，通常为 0.92-0.95）。
3. 如果找到匹配：返回缓存的响应（缓存命中）。
4. 如果未找到匹配：调用 LLM，缓存响应 + 嵌入向量（缓存未命中）。

阈值调优：较高的阈值（0.95）= 更高质量的匹配，较低的命中率。较低的阈值（0.88）= 更高的命中率，但可能返回不相关的缓存响应。

### TTL 策略

**长 TTL（24 小时-7 天）**：事实性知识（"法国的首都是什么？"）、文档摘要（源文档未更改）、系统提示词响应。

**短 TTL（1-60 分钟）**：实时数据（股票价格、天气）、用户特定数据（"我的订单状态？"）、时效性内容（新闻）。

**无缓存**：每次都需要新响应的查询（创意写作、代码生成，其中相同的提示词应产生不同的输出）。

### 失效策略

**模型更新**：新模型版本 → 刷新所有缓存条目。模型输出分布已更改；缓存的响应可能已过时。

**事实变化**：已知事实已更改（例如 CEO 变更）→ 驱逐特定条目。需要事实数据库或手动失效列表。

**基于 TTL 的失效**：条目在 TTL 后自动过期。最简单，最常用。

**写穿（Write-through）**：更新源数据时更新缓存。适合：文档摘要，其中源文档已更改。

### 成本影响

缓存命中跳过 LLM 调用。节省：
- 输入 token 成本：100% 节省（无输入处理）。
- 输出 token 成本：100% 节省（无输出生成）。
- 延迟：TTFT 从 200-500ms 降低到 5-20ms（缓存查找）。

即使使用供应商的缓存输入定价（比未缓存输入便宜 10-20 倍），语义缓存仍然更便宜，因为它完全跳过了 LLM。

### 你应该记住的数字

- 精确匹配命中率：5-15%。
- 语义缓存命中率：40-60%（与精确匹配结合）。
- 缓存查找延迟：< 1ms（Redis），5-20ms（向量搜索）。
- 成本节省：缓存命中跳过 100% 的 LLM 成本。
- 语义相似度阈值：0.92-0.95（余弦相似度）。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 精确匹配缓存（Exact match caching） | "相同提示词" | 逐字相同的提示词命中缓存 |
| 语义缓存（Semantic caching） | "相似含义" | 嵌入相似度匹配语义相似的提示词 |
| 缓存命中率（Cache hit rate） | "缓存有效率" | 从缓存中提供的请求比例 |
| TTL | "缓存过期时间" | 条目在缓存中保留的时间 |
| 失效（Invalidation） | "缓存刷新" | 移除过时的缓存条目 |
| 余弦相似度（Cosine similarity） | "语义距离" | 两个嵌入向量之间相似度的度量 |
| 嵌入向量（Embedding） | "语义指纹" | 文本的向量表示 |
| 写穿（Write-through） | "更新时刷新" | 源数据更改时更新缓存 |
