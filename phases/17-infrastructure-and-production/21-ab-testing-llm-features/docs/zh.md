# LLM 功能 A/B 测试 — GrowthBook、Statsig 与"凭感觉"问题

> 传统 A/B 测试不是为非确定性 LLM 构建的。关键区别：评估（evals）回答"模型能完成这个任务吗？" A/B 测试回答"用户在乎吗？"两者都需要；凭感觉上线已经过时了。2026 年要测试的内容：提示词工程（措辞）、模型选择（GPT-4 vs GPT-3.5 vs 开源；准确率 vs 成本 vs 延迟）、生成参数（temperature、top-p）。真实案例：一个聊天机器人奖励模型变体实现了 +70% 的对话长度和 +30% 的留存率；Nextdoor AI 主题行实验在奖励函数优化后实现了 +1% 的点击率；Khan Academy Khanmigo 在延迟-vs-数学准确率轴上进行了迭代。平台分化：**Statsig**（2025 年 9 月被 OpenAI 以 $1.1B 收购）— 序贯测试（sequential testing）、CUPED（使用实验前数据的受控实验，Controlled-experiments Using Pre-Experiment Data）、一体化。**GrowthBook** — 开源、数据仓库原生、贝叶斯 + 频率学派 + 序贯引擎、CUPED、SRM（样本比例不匹配，Sample Ratio Mismatch）检查、Benjamini-Hochberg + Bonferroni 校正。你根据数据仓库 SQL 偏好以及"被 OpenAI 收购"是否对你的组织重要来选择。

## 学习目标

- 区分评估（"模型能完成这个任务吗？"）和 A/B 测试（"用户在乎吗？"）。
- 列举三个可测试的轴（提示词、模型、参数）并为每个选择指标。

- 根据数据仓库 SQL 姿态和企业收购立场选择 Statsig 或 GrowthBook。

## 问题

你手动调整了一个系统提示词。感觉更好了。你上线了它。转化率变化在噪声范围内。你责怪指标。或者你上线了一个新模型，转化率没有变化 — 是模型退化还是变化太小无法检测？你不知道，因为你在没有 A/B 测试的情况下上线了。

评估回答模型是否能在标注集上完成一个任务。它们不回答用户是否更喜欢输出。只有受控在线实验能回答这个问题，而且只有当实验有足够的统计功效、控制了非确定性并校正了多重比较时。

## 概念

### 评估 vs A/B 测试

**评估（Evals）** — 离线、标注集、评判者（评分标准或 LLM 作为评判者或人工）。回答："在这个固定分布上，输出是否正确/有帮助/安全？"

**A/B 测试（A/B test）** — 在线、真实用户、随机化。回答："新变体是否移动了重要的用户级指标？"

两者都需要。评估在暴露之前捕获退化；A/B 在产品影响之后确认。

### 要测试什么

1. **提示词工程（Prompt engineering）** — 措辞、系统提示词结构、示例。指标：任务成功率、用户留存率、成本/请求。
2. **模型选择（Model selection）** — GPT-4 vs GPT-3.5-Turbo vs Llama-OSS。指标：准确率（任务）+ 成本/请求 + 延迟 P99。多目标。
3. **生成参数（Generation parameters）** — temperature、top-p、max_tokens。指标：任务特定（输出多样性 vs 确定性）。

### CUPED — 方差缩减

使用实验前数据的受控实验（Controlled-experiments Using Pre-Experiment Data）。在比较后期间之前回归掉前期间方差。典型方差缩减：30-70%。有效样本量免费增加。

实现：Statsig 和 GrowthBook 都实现了。

### 序贯测试

经典 A/B 假设固定样本量。序贯测试（"偷看并决定"）在重复查看下控制假阳性率。始终有效的序贯程序（mSPRT、Howard 置信序列）让你在明确的赢家上提前停止。

### 多重比较校正

以 95% 置信度运行 20 个 A/B 测试，偶然产生一个假阳性。Bonferroni 校正收紧每个测试的 α；Benjamini-Hochberg 控制假发现率。GrowthBook 实现了两者。

### SRM — 样本比例不匹配

分配哈希将用户随机分配到变体。如果 50/50 的分割结果是 47/53，说明出了问题 — SRM 检查标记它。两个平台都实现了。

### Statsig vs GrowthBook

**Statsig**：
- 2025 年 9 月被 OpenAI 以 $1.1B 收购。托管 SaaS。
- 序贯测试、CUPED、保留人群。
- 一体化：功能标志 + 实验 + 可观测性。
- 最适合：团队已经想要捆绑产品，不在乎 OpenAI 的所有权。

**GrowthBook**：
- 开源（MIT）；数据仓库原生（直接从 Snowflake/BigQuery/Redshift 读取）。
- 多个引擎：贝叶斯、频率学派、序贯。
- CUPED、SRM、Bonferroni、BH 校正。
- 自托管或托管云。
- 最适合：数据仓库 SQL 团队，数据团队控制指标层，想要开源。

### 非确定性使统计功效复杂化

相同提示词产生不同输出。传统功效计算假设 IID（独立同分布，Independent and Identically Distributed）观测。在 LLM 非确定性下，有效样本量低于名义值。将所需样本量乘以约 1.3-1.5 倍作为安全边际。

### 真实案例结果

- 聊天机器人奖励模型变体：+70% 对话长度，+30% 留存率。
- Nextdoor 主题行：奖励函数优化后 +1% 点击率。
- Khan Academy Khanmigo：迭代延迟-vs-数学准确率权衡。

### 反模式：凭感觉上线

每个资深工程师都能说出一个因为"感觉更好"而没有 A/B 测试就上线的功能。其中大多数在产品指标上退化了，团队几个月都没注意到。A/B 是强制函数。

### 你应该记住的数字

- Statsig 被 OpenAI 收购：$1.1B，2025 年 9 月。
- GrowthBook：开源 MIT；贝叶斯 + 频率学派 + 序贯。
- CUPED 方差缩减：30-70%。
- LLM 非确定性 → +30-50% 样本量缓冲。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 评估（Eval） | "离线测试" | 模型能力的标注集评估 |
| A/B 测试（A/B test） | "实验" | 对用户进行实时随机比较 |
| CUPED | "方差缩减" | 前期间回归以减少方差 |
| 序贯测试（Sequential test） | "可偷看测试" | 允许提前停止的始终有效程序 |
| 多重比较（Multiple comparison） | "家族错误" | 运行许多测试会膨胀假阳性 |
| Bonferroni | "严格校正" | 将 α 除以测试数量 |
| Benjamini-Hochberg | "BH FDR" | 假发现率控制，不那么保守 |
| SRM | "坏分割" | 样本比例不匹配；分配错误 |
| Statsig | "OpenAI 拥有" | 商业一体化，2025 年被收购 |
| GrowthBook | "开源那个" | MIT 数据仓库原生平台 |
| mSPRT | "序贯概率比测试" | 经典序贯程序 |
