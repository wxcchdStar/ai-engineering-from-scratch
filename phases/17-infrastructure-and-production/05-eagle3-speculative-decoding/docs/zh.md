# EAGLE3 推测解码 — 草稿模型、接受采样与 1.5-2.5 倍加速

> 推测解码（Speculative decoding）通过使用一个更小、更快的**草稿模型（draft model）** 来预测目标模型会生成什么，从而加速 LLM 推理。目标模型在一次前向传播中验证草稿 token — 通过接受采样（acceptance sampling）接受匹配目标分布的 token。**EAGLE3**（2025）是第三代方法，通过基于特征的草稿模型实现了 1.5-2.5 倍的加速比，该草稿模型以目标模型的隐藏状态为条件，而不是从头训练一个独立的草稿模型。关键创新：EAGLE 的草稿模型使用目标模型的嵌入和隐藏状态作为输入特征，使其比独立的草稿模型更准确（更高的接受率）。加速比取决于任务 — 代码生成（低熵）看到 2-2.5 倍；创意写作（高熵）看到 1.2-1.5 倍。内存开销：草稿模型为 70B 目标模型增加约 5-10% 的 GPU 内存。最优草稿模型选择：对于 Llama 3 70B，一个 0.5B 的草稿模型在速度和接受率之间提供了最佳权衡。推测解码与连续批处理组合使用 — vLLM 0.6.0+ 原生支持。

## 学习目标

- 解释推测解码循环：草稿模型生成 K 个 token，目标模型在一次前向传播中验证，通过接受采样接受匹配的 token。
- 区分 EAGLE3 基于特征的方法与独立的草稿模型方法。
- 计算给定接受率和草稿长度的预期加速比。
- 说出 EAGLE3 在代码生成（2-2.5 倍）和创意写作（1.2-1.5 倍）上的加速比范围。

## 问题

自回归解码每次迭代生成一个 token。对于 Llama 3.3 70B，这意味着 70B 参数从 HBM 读取 100 次以生成 100 个 token。内存带宽是瓶颈 — 你受限于 HBM 的速度，而不是计算能力。

推测解码通过一次生成多个 token 来打破这个瓶颈。一个更小、更快的草稿模型预测接下来的 K 个 token。目标模型在一次前向传播中验证所有 K 个 token。如果草稿模型足够准确，你每次目标模型前向传播获得 K 个 token，而不是 1 个。

## 概念

### 推测解码循环

```
1. 草稿模型生成 K 个候选 token（自回归，快速）
2. 目标模型在一次前向传播中处理 [提示词 + K 个候选 token]
3. 接受采样：将目标模型的概率与草稿模型的概率进行比较
4. 接受所有匹配的 token，从第一个不匹配处重新采样
5. 返回接受的 token + 重新采样的 token
```

接受采样保证输出分布与目标模型完全相同 — 推测解码不会降低质量。它纯粹是一种性能优化。

### EAGLE3 基于特征的方法

独立的草稿模型（例如使用 Llama 0.5B 作为 Llama 70B 的草稿模型）需要训练和维护一个单独的模型。草稿模型和目标模型可能具有不同的分词器或词汇表，导致对齐问题。

EAGLE（Extrapolation Algorithm for Greater Language-model Efficiency，大语言模型高效外推算法）使用目标模型自身的特征：
- 目标模型的嵌入层输出。
- 目标模型的最后一层隐藏状态。
- 草稿模型是一个轻量级解码器，以这些特征为条件。

因为草稿模型"看到"了目标模型的内部状态，它比相同大小的独立草稿模型更准确。EAGLE3 通过改进的特征融合和训练稳定性进一步提高了接受率。

### 加速比计算

预期加速比 =（每步接受的 token 数）/（1 + 草稿开销比例）

其中：
- 每步接受的 token 数取决于接受率和草稿长度 K。
- 草稿开销比例 =（草稿模型延迟）/（目标模型延迟）。

对于 EAGLE3 + Llama 70B：
- 接受率约 0.8（代码）到 0.5（创意写作）。
- K = 4-6 个草稿 token。
- 草稿开销约 5-10%。
- 加速比：1.5-2.5 倍。

### 任务依赖性

**高熵任务**（创意写作、闲聊）：低接受率（0.4-0.6），加速比 1.2-1.5 倍。草稿模型难以预测创造性输出。

**低熵任务**（代码生成、翻译、摘要）：高接受率（0.7-0.9），加速比 2-2.5 倍。草稿模型在结构化输出上表现良好。

**基于检索的任务**（RAG）：中等接受率。草稿模型可能无法预测检索到的内容。

### 内存开销

草稿模型为 70B 目标模型增加约 5-10% 的 GPU 内存。对于 0.5B 的草稿模型：约 1 GB（FP16 权重）+ KV 缓存。在 80 GB 的 A100 上可以忽略不计。在 24 GB 的消费级 GPU 上可能很显著。

### 与连续批处理的集成

vLLM 0.6.0+ 原生支持推测解码。调度器将推测解码视为一个原子操作 — 草稿 + 验证作为一个单元进行调度。连续批处理仍然有效：新请求可以在验证步骤之间加入批次。

### 你应该记住的数字

- EAGLE3 加速比：代码 2-2.5 倍，创意写作 1.2-1.5 倍。
- 草稿模型内存开销：70B 目标模型的 5-10%。
- 最优草稿长度 K：4-6 个 token。
- 接受率范围：0.4（创意）到 0.9（代码）。
- 质量影响：零 — 接受采样保证相同的输出分布。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 推测解码（Speculative decoding） | "草稿然后验证" | 草稿模型预测，目标模型验证 |
| 草稿模型（Draft model） | "小模型" | 更小、更快的模型，预测目标输出 |
| 接受采样（Acceptance sampling） | "验证步骤" | 将草稿概率与目标概率进行比较 |
| EAGLE3 | "基于特征的草稿" | 使用目标模型隐藏状态的草稿模型 |
| 接受率（Acceptance rate） | "草稿准确率" | 被接受的草稿 token 的比例 |
| 草稿长度 K（Draft length K） | "一次预测多少 token" | 每次迭代的草稿 token 数 |
| 加速比（Speedup） | "快了多少倍" | 相对于自回归基线的加速比 |
| 熵（Entropy） | "可预测性" | 高熵 = 难以预测；低熵 = 容易预测 |
