# 批量 API — 50% 折扣作为行业标准

> 每个主要供应商都提供异步批量 API（Batch API），享有 50% 折扣和约 24 小时的完成时间。OpenAI、Anthropic、Google 以及大多数推理平台（Fireworks 批量层、Together 批量）都实现了相同的模式。将批量与提示词缓存叠加使用，隔夜流水线的成本可降至同步无缓存成本的约 10%。规则非常简单：如果不是交互式的，就应该走批量。内容生成流水线、文档分类、数据提取、报告生成、批量标注、目录打标签 — 任何能容忍 24 小时延迟的工作负载，只要还没迁移到批量，就是在浪费钱。2026 年的生产模式是将每个新的 LLM 工作负载分流到三个通道：交互式（同步 + 缓存）、半交互式（异步队列 + 回退）、批量（隔夜，叠加缓存输入）。那些假装是交互式但能容忍数分钟延迟的工作负载浪费最多。

**类型：** 学习
**语言：** Python（标准库，批量 vs 同步成本模拟器）
**前置要求：** 第 17 阶段 · 14（提示词与语义缓存）
**时间：** 约 45 分钟

## 学习目标

- 说出三个供应商批量 API（OpenAI、Anthropic、Google）以及通用的 50% 折扣 + 24 小时完成保证。
- 计算在隔夜分类工作负载上叠加批量 + 缓存输入的成本，并与同步无缓存基线进行比较。
- 将工作负载分流到交互式/半交互式/批量，并说明理由。
- 说出两个陷阱：部分交互性（用户期望比 24 小时更快）和输出模式漂移（批量文件格式因供应商而异）。

## 问题

你的团队上线了一个夜间报告生成流水线。50,000 份文档，对每份进行摘要，对摘要进行聚类，起草一份执行简报。同步运行需要 4 小时，每晚 $2,000。你听说了批量 API。

批量可以让你享受 50% 的折扣。你还对系统提示词（在所有 50K 次调用中共享）启用了提示词缓存。叠加后，账单降至每晚 $180 — 约为基线的 9%。同样的流水线，三个配置变更。

批量是 LLM 成本工具包中最便宜的杠杆，但没人去拉。原因主要是组织性的：团队认为"实时"，而 SLA（服务等级协议，Service Level Agreement）实际上是"明早之前"。本课就是关于不要把 90% 的账单留在桌上。

## 概念

### 三个批量 API

**OpenAI Batch API**：上传包含请求列表的 JSONL 文件。承诺 24 小时完成（实际通常约 2-8 小时）。输入和输出 token 享受 50% 折扣。`/v1/batches` 端点。符合条件的缓存输入还可以叠加缓存输入定价。

**Anthropic Message Batches**：上传 JSONL 文件。24 小时完成。50% 折扣。支持 `cache_control` — 缓存写入是显式的，读取在批量内自动进行。

**Google Vertex AI Batch Prediction**：BigQuery 或 GCS 输入。Gemini 享受类似的 50% 折扣。与 Vertex 流水线集成。

### 语义：异步，而非慢速

批量意味着"我承诺在 24 小时内返回" — 而不是"这将需要 24 小时"。典型的 P50 是 2-6 小时。供应商在 GPU 库存未充分利用的非高峰时段调度你的批量作业。

### 与缓存叠加

一个 50K 文档的摘要任务，使用相同的 4K token 系统提示词：

- 同步无缓存：50000 ×（$input × 4000 + $output × 200），按全价计费。
- 同步缓存：系统提示词在首次写入后缓存；其余 49999 次获得 10 倍更便宜的输入。
- 批量缓存：以上所有加上读写均享受 50% 折扣。

叠加方案：批量 + 缓存 = 同步无缓存账单的约 10%。任何隔夜运行且具有共享系统提示词的工作负载都应该使用此方案。

### 工作负载分流

**交互式** — 用户等待响应。TTFT（首 token 时间，Time To First Token）很重要。同步调用 + 提示词缓存。无法批量处理。

**半交互式** — 用户提交任务，几分钟后回来查看。异步队列，如果批量不可用则回退到同步。例如中等规模的 RAG（检索增强生成，Retrieval-Augmented Generation）索引。

**批量** — 用户期望"明早之前"或"下一小时"获得结果。内容流水线、大规模分类、离线分析。始终批量处理，始终叠加缓存。

常见错误：因为流水线是生产环境就将所有内容归类为交互式。生产环境不是延迟规格 — SLA 才是。

### 部分交互性陷阱

某些功能看起来是交互式的，但能容忍 5-10 分钟。例如：带有"刷新"按钮的夜间客户健康报告。用户点击刷新；等待 10 分钟是可以的。团队将其作为同步功能上线。50 次并发刷新比批量处理后通过邮件发送的成本高 10 倍。

要问的问题是："24 小时对这个用户意味着什么？" 如果答案是"他们不会注意到"，就批量处理。

### 输出模式陷阱

批量文件格式因供应商而异：

- OpenAI：JSONL，每行一个请求。
- Anthropic：JSONL，每行一条消息；响应格式嵌入其中。
- Vertex：BigQuery 表或带有 TFRecord 的 GCS 前缀。

跨供应商编写"一个批量客户端"意味着每个供应商都需要适配器代码。宣传多供应商批量的网关（Portkey、LiteLLM 的某些层级）仍然是对原始格式的薄封装。

### 你应该记住的数字

- 跨供应商批量折扣：输入 + 输出统一 50%。
- 完成 SLA：保证 24 小时，典型 P50 为 2-6 小时。
- 叠加批量 + 缓存输入：同步无缓存成本的约 10%。
- 工作负载分流规则：如果 24 小时延迟可接受，始终批量处理。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 批量 API（Batch API） | "异步折扣" | 50% 折扣，24 小时完成 |
| JSONL | "批量格式" | 每行一个 JSON 请求；OpenAI/Anthropic 标准 |
| Message Batches | "Anthropic 批量" | Anthropic 的批量 API 产品名称 |
| 批量预测（Batch prediction） | "Vertex 批量" | Vertex AI 的批量 API 产品 |
| 完成 SLA（Turnaround SLA） | "24 小时承诺" | 保证，而非典型值；典型值为 2-6 小时 |
| 工作负载分流（Workload triage） | "交互性决策" | 交互式/半交互式/批量的路由决策 |
| 输出模式（Output schema） | "响应格式" | 每个供应商的 JSONL 布局；不可移植 |
| 叠加折扣（Stacked discount） | "批量 + 缓存" | 两者同时适用时约为无缓存同步账单的 10% |
