# 百万 Token 上下文的长视频理解

> 一段1小时的4K视频，以24 FPS 进行 patch 化和嵌入，会产生约6000万个 token。一段2小时的播客转录约为30,000个 token。一部完整的蓝光电影，即使经过激进池化压缩，也有数十万个 token。Google 的 Gemini 1.5（2024年3月）以1000万 token 上下文开启了这个时代，在小时级视频上实现了可靠的"大海捞针"召回。LWM（Liu et al., 2024年2月）展示了环形注意力 (ring attention) 的扩展路径。LongVILA 和 Video-XL 进一步扩展了输入规模。VideoAgent 用智能体检索替代了原始上下文。每种方法在计算量、召回率和工程复杂度上有不同的权衡。本课并排解读这些方案。

**类型：** 构建（Build）
**语言：** Python（标准库，大海捞针模拟器 + 智能体检索路由器）
**前置课程：** Phase 12 · 17（视频时间 Token）
**时长：** ~180 分钟

## 学习目标

- 计算不同 FPS 和池化条件下长视频的总视觉 token 数。
- 解释三种扩展路径：暴力上下文（Gemini 1.5）、环形注意力（LWM）、token 压缩（LongVILA / Video-XL）。
- 比较原始上下文视频 VLM 与智能体检索视频 VLM（VideoAgent）在准确率和延迟上的表现。
- 为30分钟视频设计大海捞针测试，并测量特定分钟处的召回率。

## 问题

Qwen2.5-VL 尺寸 patch 在384原生分辨率下，单帧约729个 token。经过 3x3 池化为每帧81个 token。30分钟片段以 1 FPS = 1800帧 = 145,800个 token。2025年的开源 VLM 勉强可以处理。以 2 FPS 计为291,600个 token——只有最大的上下文才能容纳。

2小时电影以 1 FPS 为583k token。超出了2026年大多数开源模型的能力；需要 Gemini 2.5 Pro 或更激进的池化。

三种扩展路径应运而生。

## 核心概念

### 路径一：暴力上下文（Gemini 1.5, Claude Opus）

用硬件堆叠解决问题。将上下文扩展到百万级 token，在一次前向传播中处理所有内容。

Gemini 1.5 Pro 发布时支持100万 token；Gemini 1.5 Ultra 达到1000万；2026年的 Gemini 2.5 Pro 可靠处理数小时视频。论文（arXiv:2403.05530）记录了在约950万 token 时大海捞针召回率达99.7%。

工程实现：自定义注意力实现，带有内存层级（局部 + 全局 + 稀疏）加上 MoE 专家路由以提升长上下文效率。未完整公开细节。非开源。

### 路径二：环形注意力（LWM, LongVILA）

环形注意力 (Ring attention) 将长序列分布到多个设备上，形成一个"环"，每个设备持有一个分块。跨全序列的注意力通过每个设备将其分块发送给环中下一个设备、计算部分注意力、再聚合来实现。

LWM（Liu et al., 2024）以此方式训练了100万 token 上下文模型。训练计算随上下文线性增长，而非二次增长——注意力的二次开销被分摊到环中的各设备上。

LongVILA（arXiv:2408.10188）将此模式适配到 VLM。1400帧视频，每帧192个 token = 268k 上下文，通过8路并行的环形注意力训练。

### 路径三：Token 压缩（Video-XL, LongVA）

比暴力上下文更经济：在 LLM 看到序列前进行激进压缩。

Video-XL（arXiv:2409.14485）使用视觉摘要 token：每 N 帧片段产生一个"摘要" token，该 token 对这 N 帧做注意力。推理时 LLM 每个片段只看到一个摘要 token，大幅缩减上下文。

LongVA 通过"长上下文迁移"技术将 LLM 上下文从200k扩展到200万。先在长上下文文本上训练，再通过共享表示迁移到长上下文视频。

Token 压缩以牺牲特定时间戳的召回率换取可扩展性。模型知道大致发生了什么，但有时会遗漏精确帧。

### 路径四：智能体检索（VideoAgent）

不将完整视频喂给 LLM。而是将视频视为数据库，用 LLM 来查询它。

VideoAgent（arXiv:2403.10517）：

1. LLM 阅读问题。
2. LLM 向检索工具请求相关片段（"给我看有猫的片段"）。
3. 工具返回匹配的片段时间戳。
4. LLM 通过 VLM 阅读这些片段。
5. LLM 组合答案或提出后续查询。

这是将 LLM 作为智能体的模式应用于长视频。推理更经济（仅编码相关片段），但工程更复杂（检索质量成为瓶颈）。

### 大海捞针基准测试

标准长上下文测试：在视频中随机位置插入一个唯一的视觉或文本标记，然后提出需要回忆它的查询。

指标：跨视频长度和标记位置的 Recall@k。

Gemini 2.5 Pro 在长达90分钟视频上召回率 >99%。开源72B模型（Qwen2.5-VL-72B, InternVL3-78B）在30分钟时约85-90%，超过60分钟后下降。

VideoAgent 在2小时以上可以匹敌或超越原始上下文模型，因为如果工具足够好，检索可以命中"针"。

### 如何选择路径

15分钟片段需要前沿精度：开源72B + 原生上下文通常够用。选 Qwen2.5-VL-72B。

30分钟到1小时内容：开源用 LongVILA 或 Video-XL；闭源用 Gemini 2.5 Pro。质量要求决定选择——前沿级别走闭源。

2小时以上内容：VideoAgent 或类似检索模式。或者，将内容摘要为更小的块，喂入层级摘要。

### 2026年生产模式

实践中，生产级长视频管线是混合式的：

1. 对整段视频运行动态 FPS 采样 + 激进池化（得到约100k token 的全局表示）。
2. 传给72B VLM 做全局摘要。
3. 如果用户问细节问题，以摘要为索引运行智能体检索。

这结合了暴力上下文用于全局理解和检索用于局部细节。

1. 一段45分钟讲座，1 FPS，每帧81个 token。总 token 数？能放入哪些模型的上下文？

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------|----------|
| 暴力上下文 (Brute context) | "堆更多 token" | 将 LLM 上下文扩展到百万级 token；一次传播处理所有内容 |
| 环形注意力 (Ring attention) | "LWM 式并行" | 分布式注意力模式，每个设备持有一个分块并进行旋转 |
| Token 压缩 (Token compression) | "摘要 token" | 通过学习到的压缩器在 LLM 之前减少每片段 token 数 |
| 大海捞针 (Needle-in-haystack) | "NIH 测试" | 在随机位置插入唯一标记，测试时要求模型回忆 |
| 智能体检索 (Agentic retrieval) | "LLM 作为查询规划器" | LLM 向检索工具请求相关片段，通过 VLM 阅读后组合答案 |
| VideoAgent | "视频检索模式" | 典型智能体检索设计：问题 -> 工具 -> 片段 -> 答案 |
