# LLaVA-OneVision：单图、多图、视频统一于一个模型

> 在 LLaVA-OneVision（Li et al., 2024年8月）之前，开源 VLM 世界有各自独立的发展脉络：LLaVA-1.5 做单图，Mantis 和 VILA 做多图，Video-LLaVA 和 Video-LLaMA 做视频。每个模型在自己的基准测试上胜出，却在其他场景上表现不佳。LLaVA-OneVision 提出，一套课程式训练流程就能让一个模型在三种场景中全面领先，且涌现的任务迁移效应（单图能力迁移到视频、多图推理能力迁移到单图）超过了专家模型的简单加总。方法看起来很简单：一个跨场景保持恒定的视觉 token 预算 (visual-token budget)，加上一个从单图到 OneVision（多图）再到视频的显式课程。本课将深入解析预算分配、课程设计和涌现行为。

**类型：** 构建
**语言：** Python（标准库，token 预算求解器 + 课程规划器）
**前置课程：** Phase 12 · 05（LLaVA），Phase 12 · 06（任意分辨率）
**时间：** ~180 分钟

## 学习目标

- 设计一个在单图、多图和视频输入之间保持恒定的视觉 token 预算。
- 安排一个将技能从单图迁移到视频同时避免灾难性遗忘的训练课程。
- 解释为什么当课程设计得当时，一个模型在相同参数量下能超越多个专家模型。
- 列举 LLaVA-OneVision 报告的三种涌现能力：多摄像头推理 (multi-camera reasoning)、标记点提示 (set-of-mark prompting)、iPhone 截图智能体 (iPhone-screenshot agent)。

## 问题所在

图像、多图像和视频各自以不同方式对模型施加压力。

单图需要高分辨率 token（AnyRes，~2880 个视觉 token）来捕获 OCR 和细粒度细节。每个样本的预算：一张图，2880 个 token。

多图需要多张中等分辨率的图像（每张 ~576 个 token），这样跨图像的推理才能装入上下文。每个样本的预算：4-8 张图，每张 576 个 token，共 2300-4600 个 token。

视频需要大量低分辨率帧（池化后每帧 ~196 个 token）来捕获时间动态。每个样本的预算：8-32 帧，每帧 196 个 token，共 1600-6200 个 token。

如果分别训练模型，你就只能选一种预算。如果训练一个统一模型，你需要让预算能在各场景间合理地缩放，同时不会撑爆上下文窗口。

在 OneVision 之前，默认的答案是"训练一种场景，忽略其他"。Video-LLaVA 通过额外的训练阶段将视频能力硬改接入图像模型。LLaVA-NeXT 通过分块 (tiling) 添加了多图支持。没有一个方案能同时干净地处理三种场景。

## 核心概念

### OneVision token 预算

LLaVA-OneVision 选择了一个统一的视觉 token 预算，大约 3000-4000 个 token 每样本，按场景不同分配：

- 单图：AnyRes-9（3×3 块 + 缩略图），每块 384 分辨率含 729 个 patch，通过激进的 2×2 双线性池化 → 每块 182 个 token。总计：9 × 182 + 182 = 1820 个 token。或者 AnyRes-4，每块 729 个 token = 2916 + 729。
- 多图：每张图中等分辨率（384，无分块），729 个 token 无池化。6 张图的预算 → 4374 个 token。
- 视频：32 帧，384 分辨率，激进的 3×3 双线性池化 → 每帧 81 个 token。总计：32 × 81 = 2592 个 token。

这种分配使总 token 数大致保持恒定。LLM 永远不会遇到撑爆上下文的 batch。编码器对不同场景产出不同的几何形状，但 LLM 消耗相同的预算。

### 三阶段课程

LLaVA-OneVision 分三个阶段训练：

1. 单图 SFT（阶段 SI）。所有数据都是单图加文本。使用高分辨率 AnyRes 输入训练。这教会模型感知、OCR 和细粒度理解。使用 LLaVA-NeXT 数据加 OneVision 专属单图数据。
2. OneVision SFT（阶段 OV）。混合单图 + 多图 + 视频（均匀采样帧）。使用统一 token 预算训练。这教会模型处理异构 batch 形状。不重置权重——从阶段 SI 继续。
3. 任务迁移（阶段 TT）。继续使用目标任务混合训练，通常更侧重多图或视频（取决于产品需求）。可选的部署微调。

关键点：课程顺序很重要。先训视频或先训多图，即使使用相同的数据，图像性能也比先训单图差。论文对此进行了明确的消融实验。

### 课程为何有效

单图训练建立了感知基础。Patch token 携带细粒度视觉特征；LLM 学会将它们与文本整合。多图和视频引入了结构性挑战（哪张图是哪张，什么先发生），没有强大的感知基础很难学好这些。

如果从头一起训练所有场景，模型会欠拟合感知（每个 batch 中单图数据有限）而过拟合结构（大量多图/视频数据）。结果：一个模型能遵循跨图推理模式，但视觉理解很肤浅。

课程排序让你从阶段 SI 获得感知强度，然后从阶段 OV 获得组合/时序推理能力，两者兼得。

### 涌现的跨场景技能

LLaVA-OneVision 论文报告了三种涌现能力：

1. 多摄像头推理 (Multi-camera reasoning)。分别在多图和视频上训练；在推理时，被要求对多摄像头驾驶场景进行推理。尽管训练中从未见过这种确切格式，模型仍能正确整合各视角。
2. 标记点提示 (Set-of-mark prompting)。用户在图像中用编号标记注释对象；模型推理"标记3相对于标记7在做什么"。训练中既没有标记也没有注释；从空间定位 + 多图引用的组合中学会。
3. iPhone 截图智能体 (iPhone-screenshot agent)。用户提供 iPhone 屏幕截图，要求规划下一步点击。训练数据来自 UI 截图、用户工作流视频和多图前后对比。泛化到了智能体使用场景。

这些都不是训练任务；它们从课程的组合结构中涌现。

### 视觉 token 池化

token 预算需要池化。OneVision 在 2D patch 网格上使用双线性插值 (bilinear interpolation)：24×24 = 576 个 patch 变成 12×12 = 144（2x 缩减因子）或 8×8 = 64（3x 缩减因子）。池化在 patch 网格空间而非 token 空间中进行，以保持局部性。

每种场景的池化因子本身也是一个超参数。更少的池化 = 更多 token = 更丰富的表示。更多的池化 = 更少 token = 可容纳更多帧/图像。

### LLaVA-OneVision-1.5

2025 年的后续工作（LLaVA-OneVision-1.5，arXiv 2509.23661）在训练数据、模型权重和代码上"完全开放"。在一些基准测试上追平了闭源模型的差距，使该方法大众化。相同的课程、更多数据、更好的基座 LLM。没有架构变化。

### 与 Qwen2.5-VL 的对比

Qwen2.5-VL（Lesson 12.09）做了不同的选择。它使用 M-RoPE 和动态 FPS 而非固定池化。其预算随输入缩放——1 分钟的视频比 5 秒视频使用更多 token。LLaVA-OneVision 固定预算并缩放池化。两者都有效；它们在可配置性与可预测性之间做了不同的权衡。

## 上手实践

`code/main.py` 是一个 OneVision 风格 VLM 的课程和预算规划器。给定每个样本的 token 预算和目标场景混合比（如 40% 单图、30% 多图、30% 视频），它能：

- 为每种场景分配分辨率、池化因子和帧数。
- 检查每种场景是否都在共享预算内。
- 报告预期 token 数量、LLM FLOPs 以及哪些场景 token 不足。
- 打印逐阶段的训练计划。

用它来规划 OneVision 微调或验证 VLM 部署的每请求成本。

## 交付产出

本课产出 `outputs/skill-onevision-budget-planner.md`。给定目标任务分布和每样本预算，它输出 AnyRes 因子、每帧池化、视频帧数和课程阶段权重。在训练或微调统一场景 VLM 时使用。

## 练习

1. 你的产品支持 80% 单图、10% 多图（2-4 张图）、10% 视频（8-16 帧）。设计 token 预算。从减少多图预算中省下的额度，你会分配到哪里？

2. 阅读 LLaVA-OneVision 第 4.3 节（涌现能力）。提出一种课程可能会解锁但论文未报告的第四种涌现技能。

3. 交换课程顺序——先训多图，然后单图，再视频。预测哪些基准测试会下降以及原因。

4. 论文报告视频基准测试仅使用每个样本 8 帧训练。这能泛化到推理时的 30 秒视频吗？什么先崩溃——token 预算还是时序推理？

5. 对 24×24 patch 进行双线性池化到 12×12 是每个维度 4 倍缩减。用 Python 标准库实现该池化，并验证每个 2×2 块的均值与双线性输出一致。

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------|----------|
| OneVision 场景 (OneVision scenario) | "单图、多图或视频" | 统一 VLM 处理的三种输入形状之一；预算在各场景间保持恒定 |
| Token 预算 (Token budget) | "每个样本多少 token" | LLM 每个训练/推理样本看到的视觉 token 总数，通常 3000-4000 |
| 课程 (Curriculum) | "训练顺序" | 阶段排序（单图 → 多图 → 视频），为涌现迁移而设计 |
| 双线性池化 (Bilinear pooling) | "Token 缩减" | 对 patch 网格（2D）应用双线性插值以减少 token 数，同时保持局部性 |
| 涌现技能 (Emergent skill) | "没训过但能用" | 推理时出现的能力，训练数据中没有对应样本，源于课程的组合效应 |
| AnyRes-k | "k 块配置" | k 个固定分辨率子块加一个缩略图，典型 k ∈ {4, 9} |
| 任务迁移 (Task transfer) | "跨场景泛化" | 在单图上学到的技能通过共享骨干网络应用到视频（反之亦然） |
