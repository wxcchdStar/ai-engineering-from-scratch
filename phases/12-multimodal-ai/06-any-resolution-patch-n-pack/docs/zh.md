# 任意分辨率视觉：Patch-n'-Pack（打包）与 NaFlex（原生弹性分辨率）

> 真实图像不是 224x224 的正方形。一张收据是 9:16，一张图表是 16:9，一张医学扫描可能是 4096x4096，一张手机截屏是 9:19.5。2024 年之前的 VLM 方案——把所有图像缩放到固定正方形——丢弃了让 OCR、文档理解和文本行解析能够工作的关键信号。NaViT（Google，2023）展示了可以将可变分辨率的 patch 连同块对角掩码一起打包进单个 transformer batch。Qwen2-VL 的 M-RoPE（多模态旋转位置编码）（2024）完全抛弃了绝对位置编码表。LLaVA-NeXT 的 AnyRes（多分辨率切分）将高清图像切分为一张基础图加多张子图。SigLIP 2 的 NaFlex（原生弹性分辨率）变体（2025）现在是开放 VLM 的默认编码器，只需一个模型检查点即可服务所有宽高比。本课端到端地实现了 patch-n'-pack（打包）。

**类型：** 构建
**语言：** Python（标准库，patch packer + 块对角掩码）
**前置要求：** 第 12 阶段 · 01（ViT patch），第 12 阶段 · 05（LLaVA）
**时间：** 约 120 分钟

## 学习目标

- 将一批可变分辨率图像的 patch 打包为一条序列，并构建块对角注意力掩码。
- 在 AnyRes（多分辨率切分）分块（LLaVA-NeXT）、NaFlex（原生弹性分辨率）（SigLIP 2）和 M-RoPE（多模态旋转位置编码）（Qwen2-VL）之间根据任务选择合适方案。
- 在不缩放的情况下，为 OCR、图表和摄影计算 token 预算。
- 说出正方形缩放的三种失败模式：文字被压扁、内容被裁切、padding token 浪费。

## 问题所在

Transformer 期望一个序列。一个 batch 是若干等长序列的堆叠。如果你的图像是 224x224，每次得到 196 个 patch token，无需 padding，大功告成。在 224 上训练，在 224 上推理，再也不必考虑分辨率的问题。

但现实世界并不配合。文档是纵向的（8.5x11 英寸，约 2:3）。图表截屏是横向的（16:9）。收据又高又窄（1:3）。医学影像以 2048x2048 甚至更大的尺寸传输。移动设备截屏是 1170x2532（0.46:1）。

2024 年之前的三种方案及其失败原因：

1. 缩放到固定正方形（224x224 或 336x336）。挤压会扭曲文字和人脸。降采样会破坏图表标签和 OCR 内容。这是 LLaVA-1.5 之前的标准做法。
2. 裁剪到固定宽高比。这会丢掉图像的大部分内容，而选择裁剪位置本身就是一个视觉问题。
3. 填充到最长边。解决了扭曲问题，但对于纵向图像，超过 50% 的 token 被浪费在 padding 上。所有这些填充 token 还要付出平方级别的注意力计算成本。

2024-2025 年的答案：让 transformer 以图像的原生分辨率消费 patch，并设法将异构 batch 打包进一个序列，同时不浪费计算。

## 核心概念

### NaViT 与 patch-n'-pack（打包）

NaViT（Dehghani et al.，2023）是展示这一方案在大规模下可行的论文。其思路是纯机械性的：

1. 对 batch 中的每张图像，按选定的 patch 大小（如 14）计算其原生 patch 网格。
2. 将每张图像的 patch 展平为各自变长的序列。
3. 将所有图像的 patch 拼接为一个 batch 级别的长序列。
4. 构建块对角注意力掩码，使图像 A 的 patch 只在图像 A 内部互相关注。
5. 携带逐 patch 的位置信息（2D RoPE 或分数位置嵌入）。

一个包含三张图像的 batch——336x336（576 token）、224x224（256 token）和 448x336（768 token）——变成一条 1600 token 的序列，配一个 1600x1600 的块对角掩码。无需 padding。不浪费计算。Transformer 能处理任意宽高比。

NaViT 还在训练中引入了分数 patch 丢弃——在 batch 内随机丢弃 50% 的 patch——既起到正则化作用，又加速了训练。SigLIP 2 继承了这一做法。

### AnyRes（多分辨率切分）（LLaVA-NeXT）

LLaVA-NeXT 的 AnyRes（多分辨率切分）是务实的选择。给定一张高清图像和一个固定编码器（CLIP 或 SigLIP，336），对图像进行分块：

1. 从预定义的网格布局集合——(1x1)、(1x2)、(2x1)、(1x3)、(3x1)、(2x2) 等——中选择最适合图像宽高比的网格。
2. 将完整图像按网格分块；每块成为一个 336x336 的裁剪图。
3. 同时生成一张缩略图：将整张图像缩放到 336x336 作为全局上下文 token。
4. 通过冻结的 336 编码器编码每一块。拼接分块 token + 缩略图 token。

对于一张 672x672 的图像，采用 2x2 网格加缩略图：4 * 576 + 576 = 2880 个视觉 token。代价高昂但有效——LLM 同时看到局部细节和全局上下文。

当编码器已冻结且只支持单一分辨率时，AnyRes（多分辨率切分）是首选方案。但它会导致大图的 token 数量爆炸（一张 1344x1344 的图像用 4x4 网格是 9216 + 576 ≈ 9800 token，几乎填满 8k LLM 上下文）。

### M-RoPE（多模态旋转位置编码）（Qwen2-VL）

Qwen2-VL 引入了多模态旋转位置编码。与 NaViT 的分数位置或 AnyRes（多分辨率切分）的分块加缩略图不同，每个 patch 携带 3D 位置（时间、高度、宽度）。查询/键的旋转能够处理任意的 H、W 和时间长度。

M-RoPE（多模态旋转位置编码）无需重新训练即可提供原生动态分辨率。推理时，输入任意 HxW 图像，patch 嵌入器生成 H/14 x W/14 个 token，每个 token 获得其 (t=0, r=行, c=列) 位置，RoPE 以正确的频率旋转注意力，就完成了。Qwen2.5-VL 和 Qwen3-VL 延续了这一方案。InternVL3 的 V2PE 是同一思路，按模态使用可变编码。

与 AnyRes（多分辨率切分）不同，M-RoPE（多模态旋转位置编码）在原生分辨率下的 token 数量是 O(H x W / P^2)——没有乘法级的分块开销。与 NaViT 不同，它仍然期望每次前向只处理一张图像。跨分辨率批处理仍然需要在之上叠加 patch-n'-pack（打包）。

### NaFlex（原生弹性分辨率）（SigLIP 2）

NaFlex（原生弹性分辨率）是 SigLIP 2 模型检查点的原生弹性模式。单个模型在推理时服务多种序列长度（256、729、1024 token）。内部在训练时使用 NaViT 风格的 patch-n'-pack（打包）和逐 patch 的绝对分数位置。卖点在于：一个检查点，推理时根据任务选择 token 预算。

对于语义任务（分类、检索），256 token。对于 OCR 或图表理解，1024 token。无需重新训练。

### 打包掩码

块对角掩码是大多数实现容易出错的地方。对于一条覆盖图像 `i=0..B-1`、长度分别为 `n_i`、总长度为 `N_total` 的打包序列，形状为 `(N_total, N_total)` 的掩码 `M` 在两个索引落在同一图像块内时为 1，否则为 0。你可以从累计长度列表构建它：

```
offsets = [0, n_0, n_0+n_1, ..., N_total]
M[i, j] = 1 当且仅当存在 b 使得 offsets[b] <= i < offsets[b+1] 且 offsets[b] <= j < offsets[b+1]
```

在 PyTorch 中这只需一行代码，使用 `torch.block_diag` 或显式的 gather。FlashAttention 的变长路径（`cu_seqlens`）完全跳过掩码，直接使用累计长度张量在序列内部进行注意力计算——对于典型 batch，比稠密掩码快约 10 倍。

### Token 预算

按任务选择策略：

- OCR / 文档：1024-4096 token。SigLIP 2 NaFlex（原生弹性分辨率）用 1024，或 AnyRes（多分辨率切分）3x3 + 缩略图。
- 图表和 UI：原生 384-448 分辨率下 729-1024 token。Qwen2.5-VL 动态分辨率，配合 max_pixels 上限。
- 自然照片：256-576 token 就足够了。下游 LLM 能看到足够的信息。在内容密度高的地方才值得投入 token。
- 视频：空间池化后每帧 64-128 token，2-8 FPS。第 12.17 课将涵盖这一点。

2026 年的生产实践：为每种任务选择一个 max_pixels 上限，在该上限内按原生宽高比编码，打包 batch，并跳过 padding。Qwen2.5-VL 正是通过暴露 `min_pixels` 和 `max_pixels` 来提供这一控制旋钮的。

## 使用

`code/main.py` 实现了针对具有整数像素坐标的异构图像批次的 patch-n'-pack。它会：

- 接收一组 (H, W) 图像尺寸。
- 在 patch 大小为 14 时计算每张图像的 patch 序列长度。
- 将它们打包成一个总长度为 `sum(n_i)` 的序列。
- 构建块对角注意力掩码（为清晰起见使用密集形式）。
- 对比打包成本与方形缩放和 AnyRes 分块的成本。
- 打印一个混合批次（收据、图表、截图、照片）的 token 预算表。

运行它。跑出来的数字就是 2026 年所有开源 VLM 都使用 patch-n'-pack 的原因。

## 练习

1. 一张收据尺寸为 600x1500（宽高比 1:2.5）。在 patch 大小为 14 时，原生分辨率下需要多少 token？方形缩放至 336 后又需要多少 token？实际应用中哪种方式损失更多 OCR 精度？

2. 为一个包含四张图像的批次构建块对角掩码，其长度分别为 256、576、729、1024。验证注意力矩阵尺寸为 2585x2585，且恰好有 `256^2 + 576^2 + 729^2 + 1024^2` 个非零条目。

3. 对于一张 1792x896 的图像，patch 大小为 14，比较：(a) 方形缩放至 336 后编码，(b) AnyRes 2x1 + 缩略图，(c) 原生分辨率下的 M-RoPE。哪种方式使用的 token 最少？哪种方式保留的细节最多？

4. 实现分数级 patch 丢弃：给定一个打包序列，均匀随机丢弃 50% 的 token，并相应地更新块对角掩码。测量掩码稀疏度的变化。

5. 阅读 Qwen2-VL 论文（arXiv:2409.12191）第 3.2 节。用两句话描述 `min_pixels` 和 `max_pixels` 控制什么，以及为什么两个边界都很重要。

## 关键术语

| 术语 | 人们常说的 | 实际含义 |
|------|-----------|----------|
| Patch-n'-pack | "NaViT 式打包" | 将不同图像的可变长度 patch 序列拼接成一个批次维度 |
| 块对角掩码 | "打包掩码" | 注意力掩码，限制每张图像的 patch 只能关注自身，而不能关注打包中的相邻图像 |
| AnyRes | "LLaVA-NeXT 分块" | 将高分辨率图像拆分为固定尺寸分块的网格，外加一张全局缩略图；用固定编码器编码每一个分块 |
| NaFlex | "SigLIP 2 原生灵活" | 单个 SigLIP 2 检查点，在推理时无需重新训练即可服务 256/729/1024 token 预算 |
| M-RoPE | "多模态 RoPE" | 3D 旋转位置编码（时间、行、列），无需位置表即可处理任意 H、W、T |
| cu_seqlens | "FlashAttention 打包" | FlashAttention 变长路径使用的累积长度张量，替代密集的块对角掩码 |
| min_pixels / max_pixels | "分辨率边界" | Qwen2.5-VL 的按请求调节旋钮，用于限制极小或极大输入上的 token 数量 |
| 视觉 token 预算 | "每张图像多少 token" | 每张图像输出的 patch token 大致数量；决定了 LLM 的提示预算和注意力成本 |
