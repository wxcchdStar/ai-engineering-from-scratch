# 多模态智能体与计算机使用（Capstone）

> 2026年的前沿产品是一个多模态智能体，能够读取截图、点击按钮、导航网页UI、填写表单，并端到端地完成工作流。SeeClick和CogAgent（2024）证明了GUI定位（grounding）这一基础能力。Ferret-UI扩展到移动端。ChartAgent引入了图表的视觉工具使用。VisualWebArena和AgentVista（2026）是前沿模型追逐的基准——即使是Gemini 3 Pro和Claude Opus 4.7在AgentVista的困难任务上也只得到约30%的分数。本Capstone汇聚了Phase 12的所有线索：感知（高分辨率VLM）、推理（带工具使用的LLM）、定位（坐标输出）、长程记忆和评估。

**类型：** Capstone
**语言：** Python（标准库，动作schema + 智能体循环骨架）
**前置课程：** Phase 12 · 05（LLaVA）、Phase 12 · 09（Qwen-VL JSON）、Phase 14（智能体工程）
**时间：** 约240分钟

## 学习目标

- 设计多模态智能体循环：感知 → 推理 → 行动 → 观察 → 重复。
- 构建VLM可输出为JSON的GUI定位输出schema（点击坐标、输入文本、滚动、拖拽）。
- 比较纯截图智能体 vs 可访问性树智能体 vs 混合智能体。
- 在VisualWebArena的小型切片上搭建多模态智能体基准评估。

## 问题定义

一个订票网站的工作流："帮我找一个4月15日去东京的航班，靠过道座位，800美元以下，帮我预订。"

多模态智能体需要：

1. 对浏览器进行截图。
2. 将截图 + URL + 目标解析为计划。
3. 输出结构化动作：click（在x,y处）、type "Tokyo"（在元素E处）、scroll down、select（单选按钮）。
4. 将动作应用到浏览器。
5. 观察新状态（下一张截图）。
6. 重复直到任务完成。

每一步都是一次多模态VLM调用。VLM输出必须是可解析的JSON。错误会在各步之间累积，因此恢复机制很重要。

## 核心概念

### GUI定位——基础能力

GUI定位是：给定一张截图和一条自然语言指令，输出要点击的(x, y)坐标（或其他动作）。

SeeClick（arXiv:2401.10935）是第一个大规模开放结果：在合成+真实GUI数据上微调VLM，以纯文本token输出坐标。有效。

CogAgent（arXiv:2312.08914）增加了1120×1120高分辨率编码，适用于密集UI。得分：网页导航约84%。

Ferret-UI（arXiv:2404.05719）专注于移动端UI，集成iOS可访问性数据。

输出格式通常是JSON：

```json
{"action": "click", "x": 384, "y": 220, "element_desc": "Search button"}
```

`element_desc`有助于恢复：如果坐标在截图之间漂移，语义提示可以让系统重新定位。

### 动作Schema

典型的动作schema有6-10种动作类型：

- `click`：(x, y)
- `type`：(text, x?, y?)
- `scroll`：(direction, amount)
- `drag`：(x0, y0, x1, y1)
- `select`：(option_index)
- `hover`：(x, y)
- `navigate`：(url)
- `wait`：(ms)
- `done`：(success, explanation)

智能体每步输出一个动作。浏览器封装器执行并返回新状态。

### 纯截图 vs 可访问性树

两种输入模式：

- 纯截图：完整图像，无结构信息。最通用；适用于任何应用。
- 可访问性树：结构化DOM / iOS可访问性信息。定位更可靠；在树可用的场景下有效。
- 混合模式：两者兼用，树作为原子动作的可靠定位器，截图提供语义上下文。

生产环境中的智能体尽可能使用混合模式。浏览器自动化（Selenium + 可访问性）总是有树的；桌面应用有时有。

### 长程记忆

一个20步的工作流会产生20张截图。VLM的上下文很快就满了。三种压缩策略：

- 摘要链：每5步后，总结已发生的事情，丢弃旧截图。
- 跳帧：保留第一帧、最后一帧和每第3帧的截图。
- 工具记录日志：执行动作，保留已完成操作的文本日志；不回看旧截图。

Claude的computer-use API使用日志模式。更简单，更可靠。

### 视觉工具使用

ChartAgent（arXiv:2510.04514）引入了图表理解的视觉工具使用：裁剪、缩放、OCR、调用外部检测。智能体可以输出"裁剪到区域(100, 200, 300, 400)然后调用OCR"作为工具调用。工具返回文本；VLM继续推理。

这种模式可以泛化：set-of-mark提示、区域标注和外部检测工具都符合同样的"输出工具调用，接收结构化响应"schema。

### 2026年基准测试

- ScreenSpot-Pro。约1k个网页截图上的GUI定位。开源SOTA Qwen2.5-VL-72B约85%。前沿约90%。
- VisualWebArena。端到端网页任务（购物、论坛、分类信息）。开源SOTA约20%。Gemini 3 Pro约27%。
- AgentVista（arXiv:2602.23166）。2026年最难的基准。跨12个领域的真实工作流。前沿模型得分27-40%；开源模型10-20%。
- WebArena / WebShop。较老的基准；已被前沿模型饱和。

### 为什么仍然很难

智能体性能瓶颈：

1. 细粒度视觉定位。"点击那个小X"在移动端分辨率下经常失败。
2. 长程规划。10个动作之后，智能体偏离目标。
3. 错误恢复。当点击失败（点错按钮）时，检测+恢复很少有训练数据。
4. 跨页面上下文。在标签页之间切换或长表单中会丢失状态。

研究方向：记忆架构、显式重规划、多模态验证（截图匹配用于动作成功验证）。

### Capstone构建任务

Capstone任务：构建一个计算机使用智能体，要求：

1. 读取订票网站模拟页面的HTML + 截图。
2. 规划多步序列：搜索 → 选择 → 填写表单 → 提交。
3. 输出符合动作schema的JSON动作。
4. 在固定的10个任务切片上进行评估。

本课程提供脚手架代码，易于扩展到真实浏览器。

## 练习

1. 扩展动作schema，增加`screenshot_region`工具（裁剪+缩放）。哪些任务会受益？

2. 阅读AgentVista（arXiv:2602.23166）。描述最难的任务类别以及前沿模型仍然失败的原因。

3. 长程记忆压缩：设计一个摘要链，保持≤4张截图在线，任意数量已记录。

4. 构建一个错误恢复钩子：当动作失败（未找到按钮）时，智能体下一步做什么？

5. 比较纯截图的Claude 4.7与混合截图+可访问性树的Qwen2.5-VL在10个网页任务上的表现。哪种方案在哪些任务上胜出？

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------|----------|
| GUI定位 | "点击坐标" | 模型在截图上根据指令输出目标的(x,y)坐标 |
| 动作Schema | "工具定义" | 有效动作（click、type、scroll、drag）的JSON描述 |
| 可访问性树 | "结构化DOM" | 来自浏览器/iOS API的机器可读UI层次结构 |
| 混合智能体 | "截图+树" | 同时使用图像和结构化信息；比单独使用任一方更可靠 |
| 视觉工具使用 | "缩放/裁剪/检测" | 智能体在规划过程中调用外部视觉工具（OCR、检测） |
| 摘要链 | "记忆压缩" | 周期性文本摘要替代长截图历史 |
| VisualWebArena | "端到端网页基准" | 2024年端到端网页任务基准 |
| AgentVista | "2026年困难基准" | 12个领域的真实工作流；即使Gemini 3 Pro也只得约30% |
