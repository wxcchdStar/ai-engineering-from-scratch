# Prompt 评测与调优指南

> 写出 prompt 只是开始，系统化地评测和迭代才是让 prompt 从"能用"到"好用"的关键。

## 一、评测体系

### 1.1 评测维度

| 维度 | 检测方式 | 适用场景 |
|------|---------|---------|
| **格式合规** | JSON 解析 / 正则匹配列表 / XML 结构校验 | 结构化输出任务 |
| **长度约束** | 字数/句数/token 计数 ≤ 阈值 | 摘要、简答 |
| **内容覆盖** | 必需关键词/概念命中率 | 知识问答、信息提取 |
| **边界遵守** | 禁用短语零违规 | 护栏/安全场景 |
| **语义正确性** | LLM-as-Judge / 人工标注 | 开放式生成 |

### 1.2 评分公式

脚本采用的 `composite_score` 计算方式：

```
composite_score = mean(所有布尔指标 + 所有 0~1 浮点指标)
```

- 布尔值（格式合规、长度达标、无违规）→ 1.0 或 0.0
- 浮点值（关键词覆盖率）→ 0~1
- 取算术平均得到综合分

### 1.3 评测架构

```
┌─────────────────────────────────────────────────────────┐
│  TEST_SUITE (测试用例集)                                  │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │ pattern     │  │ variables   │  │ criteria    │     │
│  │ (模式选择)  │  │ (变量填充)   │  │ (评分标准)   │     │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘     │
└─────────┼────────────────┼────────────────┼─────────────┘
          ▼                ▼                │
    build_prompt(pattern, variables)        │
          │                                 │
          ▼                                 │
    format_*_request()  ← 适配各 provider   │
    (OpenAI / Anthropic / Google)           │
          │                                 │
          ▼                                 │
    多模型并行调用                            │
    (GPT-4o / Claude / Gemini)              │
          │                                 │
          ▼                                 ▼
    score_response(response, criteria)  ← 逐模型打分
          │
          ▼
    compare_models() → 跨模型排名
```

核心设计原则：**同一个 prompt，跨模型对比**，隔离模型差异，暴露 prompt 本身的问题。

### 1.4 评测方法分层

| 层级 | 方法 | 成本 | 适用 |
|------|------|------|------|
| L1 | 规则匹配（关键词/正则/格式解析） | 零成本 | 格式约束、边界行为 |
| L2 | LLM-as-Judge（用强模型评判弱模型） | 中等 | 语义质量、正确性 |
| L3 | 人工标注 + 对比 | 高 | 金标准评测、上线前验收 |
| L4 | 代码执行验证 | 零成本 | 代码生成任务（跑测试用例） |

脚本使用的是 L1 层级——纯程序化规则打分，不依赖外部 API。

## 二、调优方法论

### 2.1 选对 Pattern（结构层调优）

根据任务类型选择合适的 prompt 模式：

| 任务类型 | 推荐 Pattern | Temperature | 核心策略 |
|---------|-------------|-------------|---------|
| 数据提取/分类 | template_fill / few_shot | 0.0 | 强约束输出结构，用例子锚定格式 |
| 推理/数学 | chain_of_thought | 0.3 | "show your reasoning" 提升准确率 10-40% |
| 质量迭代 | critique | 0.5 | 自生成 → 自反思 → 改进 |
| 限定域问答 | guardrail / boundary | 0.0-0.3 | 显式边界 + 拒绝话术 |
| 复杂问题 | decomposition | 0.3 | 拆子问题 → 各个击破 → 合并验证 |
| 适配受众 | audience_adapt | 0.5 | 指定受众 + 约束词汇/深度 |
| 生成 prompt | meta_prompt | 0.7 | LLM 优化 LLM 的指令 |

### 2.2 精细化指令（内容层调优）

5 条黄金规则：

1. **指定格式** — JSON / 列表 / 段落 / XML 分隔符
2. **指定长度** — 字数、句数、字符数上限
3. **指定受众** — 技术人员 / 管理层 / 初学者
4. **明确包含与排除** — 要什么、不要什么都说清楚
5. **给一个具体示例** — 比 100 字的描述更有效

### 2.3 参数调优

```
Temperature 光谱：
├── 0.0  确定性输出（提取、分类、代码）
├── 0.3  保守（摘要、分析、技术写作）
├── 0.7  平衡（通用问答）
├── 1.0  创意（头脑风暴、文学创作）
└── 1.5+ 永远别用
```

关键原则：
- **不要同时调 temperature 和 top-p**，交互不可预测
- **评测时先 temperature=0**，隔离 prompt 质量与采样随机性
- **Assistant Prefill**（Anthropic 独有）：预填 `{` 强制 JSON 输出，比指令更可靠

### 2.4 迭代闭环

```
  ┌─────────────────────────────────────────────┐
  │                                             │
  ▼                                             │
初始 Prompt                                     │
  │                                             │
  ▼                                             │
跑 N 次 (temp=0.7) → 评分                       │
  │                                             │
  ▼                                             │
找最弱指标 (哪个维度得分最低?)                     │
  │                                             │
  ▼                                             │
针对性修改 Prompt                                │
  │                                             │
  ▼                                             │
对比分数变化 → 有提升？─── 否 ───→ 换策略 ─────────┘
  │
  │ 是
  ▼
收敛 → 固化为最终版本
```

具体步骤：
1. 初始 prompt + criteria → 跑 5 次 → 打分
2. 分析最弱维度（格式？覆盖率？长度？）
3. 对症下药：格式弱 → 加 few-shot 示例；覆盖率低 → 显式列出必含内容；长度超标 → 加硬约束
4. 重跑对比 → 重复 3 轮
5. 分数不再提升 → 收敛

## 三、避坑清单（Anti-Patterns）

| 反模式 | 症状 | 解法 |
|--------|------|------|
| **过度约束** | system prompt > 500 tokens，模型遵循规则的能力占满了注意力 | 精简到 < 500 tokens |
| **指令矛盾** | "简洁" + "覆盖所有边界" 同时出现 | 审计内部一致性，二选一 |
| **模型绑定** | 只在一个模型上测试通过 | 跨模型测试，用 XML 等通用语法 |
| **假设默认行为** | 不指定格式，靠模型猜 | 每个歧义点都显式约束 |
| **无评分基线** | 改了 prompt 不知道是变好还是变差 | 先建 criteria，再改 prompt |

## 四、跨模型兼容设计

写出一个在 GPT / Claude / Gemini / 开源模型上都能用的 prompt：

1. 用**纯自然语言**，不依赖模型特有语法
2. **显式声明格式**，不靠默认行为
3. 用 **XML 分隔符** 做结构化（所有主流模型都处理得好）
4. 关键指令放在 **开头和结尾**（避免 "lost in the middle" 效应）
5. 先用 **temperature=0** 测试，隔离采样噪声
6. 包含 **2-3 个 few-shot 示例**——比纯指令跨模型迁移性更好

## 五、实操检查单

在修改/优化 prompt 前，过一遍这个检查单：

- [ ] 是否定义了明确的评分 criteria？
- [ ] 是否用 temperature=0 建立了基线分数？
- [ ] 是否跨至少 2 个模型测试过？
- [ ] system prompt 是否 < 500 tokens？
- [ ] 指令之间是否有矛盾？
- [ ] 修改后分数是否有可量化的提升？
- [ ] 是否测试了边界情况（空输入、超长输入、注入攻击）？
