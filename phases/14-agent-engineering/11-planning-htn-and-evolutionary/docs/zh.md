# 规划：HTN 与进化搜索 (Planning with HTN and Evolutionary Search)

> 符号化规划处理那些计划必须可证明正确的场景。进化代码搜索处理那些适应度函数可机器检查的场景。ChatHTN (2025) 和 AlphaEvolve (2025) 展示了各自与 LLM 配对时能解锁什么。

**类型：** 构建 (Build)
**语言：** Python (标准库)
**前置课程：** Phase 14 · 02 (ReWOO 与 Plan-and-Execute)
**时间：** 约 75 分钟

## 学习目标

- 解释层级任务网络 (HTN)：任务、方法、操作符、前置条件、效果。
- 描述 ChatHTN 的混合循环——符号化搜索配合 LLM 回退分解。
- 解释 AlphaEvolve 的进化循环及其为什么只能与程序化评估器配合使用。
- 用标准库实现一个玩具 HTN 规划器和一个玩具进化搜索。

## 问题

ReWOO (第 02 课)、Plan-and-Execute 和 ReAct 覆盖了大多数 Agent 规划。两个它们覆盖不好的场景：

1. **需要可证明正确性的计划。** 调度、飞行路径、合规工作流——计划必须在构造上就是正确的。一个流畅但偶尔幻觉出某一步的 LLM 计划是不可接受的。
2. **具有可机器检查的适应度函数的优化。** 矩阵乘法、调度启发式、编译器 pass——目标不是"一个正确的计划"而是"最好的计划"。

HTN 规划和 AlphaEvolve 解决这两个不同的问题。两者都将 LLM 用作放大器，而非替代品。

## 概念

### 层级任务网络 (HTN)

一个 HTN 包含：

- **任务 (Tasks)** — 复合任务（待分解）和原始任务（可直接执行）。
- **方法 (Methods)** — 将复合任务分解为子任务的方式，带有前置条件。
- **操作符 (Operators)** — 带有前置条件和效果的原始动作。
- **状态 (State)** — 一组事实。

规划：给定一个目标任务和一个初始状态，找到一种分解为原始操作符的方式，使得前置条件在序列中依次满足。

HTN 比 LLM 更古老，至今仍是可证明正确计划的参考标准。

### ChatHTN (Gopalakrishnan et al., 2025)

ChatHTN (arXiv:2505.11814) 将符号化 HTN 与 LLM 查询交织：

1. 尝试用现有方法分解当前复合任务。
2. 如果没有方法适用，询问 LLM："你如何在状态 `s` 下分解 `task`？"
3. 将 LLM 响应翻译为候选子任务。
4. 对照操作符 schema 进行验证；拒绝无效的分解。
5. 递归。

论文的核心主张：每个产生的计划都是可证明正确的，因为 LLM 建议仅作为候选分解进入，从不作为直接的计划编辑。符号层拥有正确性；LLM 扩展方法库。

在线方法学习 (OpenReview `gwYEDY9j2x`, 2025 后续) 添加了一个学习器，通过回归泛化 LLM 产生的分解——将 LLM 查询频率降低高达 75%。

### AlphaEvolve (Novikov et al., 2025)

AlphaEvolve (arXiv:2506.13131, DeepMind, 2025 年 6 月) 是另一种野兽：由 Gemini 2.0 Flash/Pro 组合编排的进化代码搜索。

循环：

1. 从一个种子程序 + 一个程序化评估器（返回适应度分数）开始。
2. LLM 组合提出变异。
3. 通过评估器运行变异。
4. 保留最好的；再次变异。

已发表的成果：

- 56 年来首次改进 4x4 复数矩阵乘法的 Strassen 算法（48 次标量乘法）。
- 通过 Borg 调度启发式回收 0.7% 的 Google 计算资源。
- 在前沿工作负载上实现 32% 的 FlashAttention 加速。

硬约束：适应度函数必须是可机器检查的。对文本答案的进化搜索不会收敛。

### 何时使用哪种

| 问题类别 | 使用 | 原因 |
|---------|------|-----|
| 带硬约束的调度 | HTN + ChatHTN | 可证明的正确性 |
| 编译器优化 | AlphaEvolve | 可机器检查的适应度 |
| 多步任务执行 | ReAct / ReWOO | LLM 在循环中，无形式化保证 |
| 带测试的代码改进 | AlphaEvolve | 测试就是评估器 |
| 策略约束的自动化 | HTN | 前置条件编码策略 |

### 这个模式哪里会出错

- **没有操作符的 HTN。** 没有前置条件/效果 schema，正确性主张就崩溃了。ChatHTN 的"LLM 建议分解"需要 schema 来拒绝无效动作。
- **没有真正评估器的 AlphaEvolve。** "问 LLM 代码是否更好"不是适应度函数。评估器必须是确定性的且快速的。
- **过度工程。** 大多数 Agent 任务两者都不需要。先尝试 ReAct 或 ReWOO。

## 构建

`code/main.py` 实现两个玩具：

- 一个标准库 HTN 规划器，包含操作符、方法、前置条件、效果，以及一个 `LLMFallback`，当没有方法匹配复合任务时触发。"LLM" 是一个脚本化的分解器，因此规划器可以离线运行。
- 一个标准库进化搜索，针对算术程序：生长表达式，使其输出在测试集上最小化 `|f(x) - target|`。评估器是确定性的。

运行：

```
python3 code/main.py
```

追踪显示 HTN 规划器分解一个复合任务（中途有一次 LLM 回退），以及进化循环收敛到一个目标表达式。

## 使用

- **HTN 规划器** — `pyhop`、`SHOP3`，或为领域特定的策略执行构建自己的。
- **ChatHTN** — 研究代码；模式（符号化 + LLM 回退）可以干净地移植到任何 HTN 规划器。
- **AlphaEvolve** — DeepMind 论文；模式（组合 + 评估器）是可复现的。OpenEvolve 和类似的开源分支正在涌现。
- **Agent 框架** — 目前还没有框架内置一等 HTN 或 AlphaEvolve。将其构建为子 Agent 或后台工作器。

## 交付

`outputs/skill-hybrid-planner.md` 生成一个混合规划器脚手架（HTN 或进化），LLM 角色被显式限定。

## 练习

1. 为 HTN 规划器扩展回溯：当操作符的后置条件在运行时失败时，回滚并尝试下一个方法。
2. 为 ChatHTN 添加 LLM 方法缓存：当 LLM 在状态模式 `P` 下分解任务 `T` 时，存储结果。下次调用时首先重新检查方法库。
3. 将进化搜索评估器替换为真实的测试套件。进化一个通过 20 个测试用例的排序函数；报告收敛所需的代数。
4. 阅读 AlphaEvolve 的评估器设计笔记。为你关心的领域设计一个评估器（SQL 查询优化、测试套件最小化、部署 YAML）。
5. 组合：使用 HTN 将复合任务分解为子任务，然后对每个子任务的原始操作符使用进化搜索。在哪些场景下表现出色，哪些场景下过度工程？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| HTN | "层级规划器" | 带操作符、前置条件、效果的任务分解 |
| 方法 (Method) | "分解规则" | 将复合任务分解为子任务的方式 |
| 操作符 (Operator) | "原始动作" | 带前置条件和效果的具体步骤 |
| ChatHTN | "LLM + HTN" | 符号化规划器在没有方法匹配时询问 LLM |
| AlphaEvolve | "进化代码搜索" | LLM 组合变异代码；确定性评估器选择 |
| 适应度函数 (Fitness function) | "评估器" | 对输出的确定性、可机器检查的评分 |
| 在线方法学习 (Online method learning) | "缓存的 LLM 分解" | 存储并泛化 LLM 计划以降低查询成本 |
