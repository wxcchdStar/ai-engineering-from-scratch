# 共享内存与黑板模式（Shared Memory and Blackboard Patterns）

> 2026 年多智能体系统中有两种方法共存：**消息池（message pool）**（每个人都看到每个人的消息，如 AutoGen GroupChat 或 MetaGPT）和**带订阅的黑板（blackboard with subscription）**（智能体订阅相关事件，如 Context-Aware MCP 或 Matrix 框架）。两者都是多智能体系统中唯一有状态的部分——这意味着两者都是有趣 bug 的所在。参考故障模式是**记忆污染（memory poisoning）**：一个智能体产生幻觉"事实"，其他智能体将其视为已验证，准确性逐渐衰减，这种方式比立即崩溃更难调试。本课程从标准库构建两种结构，注入污染攻击，并展示三种在生产中实际有效的缓解措施。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`）
**前置条件：** 第 16 阶段 · 04（原语模型）、第 16 阶段 · 09（并行群体网络）
**时间：** ~75 分钟

## 问题

多智能体系统需要一个地方让智能体共享事实。一个字面选项是"在消息中传递一切"——但这用额外的复制重新发明了共享状态。另一个是"给每个人一个全局日志"——但全局日志无限增长且容易污染。第三个是"为每个智能体投影视图"——可扩展但模式繁重。

当其中一个智能体产生幻觉并将幻觉写入共享状态时，每个读取该状态的下游智能体都将幻觉采纳为事实。当人类注意到时，推理链已经深入五步，根本原因是第三条消息。调试多智能体准确性衰减比调试崩溃更难。

这就是记忆污染。它是 MAST 分类法中第二多记录的故障家族（Cemri 等人，arXiv:2503.13657），并且是结构性的：任何没有溯源和不可写验证器的共享内存设计最终都会出现这种情况。

## 概念

### 两种主要拓扑

**全量消息池。** 每个智能体读取每条消息。AutoGen GroupChat 和 MetaGPT 使用此方法。简单、透明、可检查，但不能扩展到超过约 10 个智能体，因为每个智能体的上下文会被其他智能体的工作填满。

```
智能体-A ──写入──▶ ┌────────────────┐ ◀──读取── 智能体-D
                   │   消息池       │
智能体-B ──写入──▶ │                │ ◀──读取── 智能体-E
                   │ （全局日志）    │
智能体-C ──写入──▶ └────────────────┘ ◀──读取── 智能体-F
```

**带订阅的黑板。** 智能体声明对主题的兴趣；底层仅路由相关消息。CA-MCP（arXiv:2601.11595）和 Matrix 去中心化框架（arXiv:2511.21686）使用此方法。可进一步扩展，但需要前期模式设计以使订阅有意义。

```
                   ┌─ 主题: 价格 ──┐
智能体-A ──发布──▶ │               │ ──▶ 智能体-D（已订阅）
                   ├─ 主题: 订单 ──┤
智能体-B ──发布──▶ │               │ ──▶ 智能体-E（已订阅）
                   ├─ 主题: 警报 ──┤
智能体-C ──发布──▶ │               │ ──▶ 智能体-F（已订阅）
                   └───────────────┘
```

### 每种方法何时胜出

-   **全量池**在智能体较少（< 10）、异构且对话时间跨度短时胜出。当每个人都看到一切时，推理谁说了什么是微不足道的。
-   **黑板**在智能体较多、角色同质但实例众多（群体）且对话长时间运行时胜出。路由节省 token 成本和上下文污染。

生产系统通常混合使用：顶层小型全量池（规划层），下层黑板（工作者层）。

### 记忆污染，一个场景

三个智能体处理一个研究任务。智能体 A 是检索智能体。智能体 B 是摘要器。智能体 C 是分析师。

1.  A 获取一个页面并向共享状态写入一条消息："研究报告了 42% 的准确性提升。"
2.  获取的页面实际上说"4.2% 的提升"。A 产生了小数点的幻觉。
3.  B 读取共享状态，写道："报告了 42% 的大幅准确性提升（来源：A）。"
4.  C 读取共享状态，写道："建议采用——42% 的提升是变革性的。"
5.  最终报告引用了一个从未存在过的 42% 数字。

没有智能体崩溃。没有测试失败。系统"工作了"。幻觉通过共享状态从一个智能体的上下文跨越到每个下游智能体的推理中。

### 为什么这是结构性的

没有共享状态，智能体 A 的幻觉停留在 A 的上下文中。下游智能体会重新获取或重新推导，可能捕捉到错误。使用天真的共享状态，A 的上下文变成每个人的上下文，幻觉被洗白成事实。

问题不在于共享状态本身——而在于**没有溯源和独立验证器的共享状态**。三种缓解措施解决这个问题：

1.  **在每次写入时附加溯源。** 共享状态中的每个条目记录谁写的、何时写的、在什么提示词下写的，以及（如果适用）智能体引用了什么来源。下游智能体根据溯源进行怀疑性阅读。
2.  **版本化写入；将其视为仅追加。** 更正是取代旧条目的新条目，而不是原地更新。审计跟踪得以保留。
3.  **保留至少一个不能写入共享状态的智能体。** 只读验证器智能体采样条目，重新获取来源，并标记不一致。因为它不能写入池，所以不能被池污染。

### 黑板先例（Hayes-Roth, 1985）

黑板模式比 LLM 智能体早四十年。Hayes-Roth（1985，"A Blackboard Architecture for Control"）描述了专业化的知识源（Knowledge Sources），它们观察全局黑板，贡献部分解决方案，并触发其他源。2026 年的黑板（CA-MCP、Matrix）是相同的模式，以 LLM 智能体作为知识源，以 JSON 块作为部分解决方案。旧文献有关于写入争用、机会主义控制和一致性的文档化解决方案，现代系统正在重新发现这些。

### 投影 vs 全量视图

纯黑板给每个订阅者相同的投影（主题范围）。更激进的设计是**每个智能体投影**：每个智能体获得为其角色定制的视图。LangGraph 的状态归约器（state reducers）是 2026 年的规范实现——归约器函数将全局状态折叠为角色特定的切片。

每个智能体投影可进一步扩展，但需要模式。没有模式，你会在每个智能体的提示词中重建临时投影。

### 写入争用模式

多个智能体同时写入是一个并发问题，而不仅仅是 LLM 问题。三种模式有效：

-   **顺序写入器（单生产者）。** 所有写入通过一个序列化的协调器智能体。简单，但是瓶颈。
-   **带版本控制的乐观并发。** 每个条目有一个版本；写入器在版本不匹配时失败并重试。经典数据库技术。
-   **主题分区。** 不同的智能体拥有不同的主题。没有跨主题争用。需要设计的分区边界。

大多数 2026 年框架默认使用顺序写入器，因为 LLM 调用足够慢，争用罕见且瓶颈不造成伤害。

### 不可写验证器

最关键负载的缓解措施是只读验证器。实现规则：

-   验证器与团队共享状态（读取黑板或池）。
-   验证器没有共享状态的写入句柄——只有单独的验证通道。
-   验证器独立获取写入中引用的来源。标记分歧。
-   验证器自己的输出路由到人类或单独的决策智能体，永远不反馈回池中。

没有这种分离，验证器的输出成为池中的新条目，这意味着被污染的池污染验证器，验证器污染其验证结果。

## 构建它

`code/main.py` 用标准库 Python 实现了两种拓扑，加上一个玩具污染攻击和三种缓解措施。

-   `MessagePool`——线程安全的仅追加日志，具有完整读取。
-   `Blackboard`——主题键控的发布/订阅，具有每个智能体订阅。
-   `ProvenanceEntry`——每次写入记录（writer, timestamp, prompt_hash, source_uri）。
-   `PoisoningScenario`——运行一个三智能体研究任务，其中智能体 A 产生小数点的幻觉。打印最终报告。
-   `Verifier`——一个只读智能体，重新获取来源并标记不一致。在验证器存在的情况下运行相同场景。

运行：

```
python3 code/main.py
```

预期输出：
-   运行 1（无验证器）：幻觉的 42% 传播到最终报告。
-   运行 2（有验证器）：验证器标记不一致，池被标记为"已标记"，最终报告包含撤回。

## 使用它

`outputs/skill-memory-auditor.md` 是一个技能，审计任何多智能体系统的共享内存设计的溯源、版本控制和验证器分离。在生产前对新多智能体架构运行它。

## 交付它

对于任何共享内存设计：

-   在每次写入时记录溯源：`(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`。
-   使日志仅追加。更正是引用被取代条目的新条目。
-   部署至少一个具有独立来源访问权限的只读验证器智能体。
-   将验证器输出路由到单独的通道，而不是反馈回共享池。
-   记录取代写入的比例——上升的比例是幻觉模式的早期证据。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|----------|
| 消息池 | "共享聊天历史" | 每个智能体读取的仅追加日志。完全透明，扩展性差。 |
| 黑板 | "共享工作空间" | 主题键控的发布/订阅。智能体订阅相关主题。可进一步扩展。 |
| 溯源 | "谁写了什么" | 每次写入的元数据：写入者、时间戳、提示词、来源。 |
| 记忆污染 | "幻觉传播" | 一个智能体的错误进入共享状态，下游智能体将其采纳为事实。 |
| 仅追加 | "无原地更新" | 更正是取代旧条目的新条目。保留审计跟踪。 |
| 不可写验证器 | "独立审计员" | 只读智能体，重新获取来源并标记不一致。 |
| 投影 | "范围视图" | 从全局状态计算的每个智能体视图。LangGraph 归约器是规范案例。 |
| 知识源 | "专家智能体" | Hayes-Roth 1985 年对黑板参与者的术语。 |
