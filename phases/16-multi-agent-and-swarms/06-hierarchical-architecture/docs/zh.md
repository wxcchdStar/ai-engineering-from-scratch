# 层级架构（Hierarchical Architecture）及其故障模式

> 层级架构（Hierarchical）是嵌套的监督者。管理者智能体（manager agents）管理子管理者（sub-managers），子管理者管理工作者（workers）。CrewAI 的 `Process.hierarchical` 是教科书版本：一个 `manager_llm` 动态委托任务并验证输出。LangGraph 的等价物是 `create_supervisor(create_supervisor(...))`。当任务是一个真实的组织结构图时，这是自然的模式。这也是最有可能崩溃为管理者循环（managerial looping）的模式——管理者智能体分配工作不当、误解子输出或未能达成共识。顺序模式（Sequential）通常优于它。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置条件：** 第 16 阶段 · 05（监督者模式）
**时间：** ~60 分钟

## 问题

一旦监督者模式被理解，自然的下一步是"如果工作者本身也是监督者呢？"团队有子团队；公司有部门的部门。层级架构反映了这一点。

问题是：LLM 管理者与人类管理者不同。人类管理者对其下属的知识有稳定的先验。LLM 管理者每轮都从其上下文中的任何内容重新推理组织。上下文中的微小漂移，整个树就会错误分配工作。

## 概念

### 形状

```
                 管理者
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       子管理者 A         子管理者 B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
        W1  W2  W3         W4  W5
```

每个内部节点规划、委托和综合。只有叶子节点做具体工作。

### 它在哪里表现出色

-   **清晰的组织映射。** 如果真实任务是部门性的（"法务审查文档，财务审查文档，工程审查文档，然后为高管总结"），层级结构是明确的。
-   **局部摘要。** 每个子管理者在顶层管理者看到之前综合其团队的输出。顶层管理者看到三个子管理者摘要，而不是十五个工作者输出。

### 它在哪里出问题

2026 年的事后分析不断发现三种故障模式：

1.  **任务分配错误。** 管理者读取目标，产生幻觉分解，并委托给错误的子管理者。因为子管理者顺从地处理交给它的内容，错误只在顶层综合时才浮出水面——距离人类本可以捕捉到它的地方差了一个层级。
2.  **输出误解。** 子管理者返回"无法验证声明 X"。顶层管理者总结为"声明 X 未确认"。含义在每个层级漂移。
3.  **共识循环。** 两个子管理者意见不一致；顶层管理者要求它们协调；它们重新向下委托；工作者重新运行；子管理者返回略有不同的答案；循环。CrewAI 的 `Process.hierarchical` 通过步骤限制来防范这一点，但限制本身现在是一个超参数。

### 决定性问题是

顺序（线性流水线）vs 层级：你的任务是否真的有独立的子团队，还是只是一个假装成树的线性流程？如果是后者，使用顺序。如果是前者，使用层级但预算明确的协调规则。

### CrewAI 的实现

`Process.hierarchical` 在专业化团队之上连接一个管理者 LLM。管理者：

-   接收顶层任务，
-   将子任务分配给团队，
-   评估团队输出，
-   决定是接受、重新委托还是迭代。

文档：https://docs.crewai.com/en/introduction（在核心概念下查找"Hierarchical Process"）。

### LangGraph 的实现

LangGraph 使用嵌套的 `create_supervisor` 调用。内部监督者有自己的图；外部监督者将内部图视为不透明节点。这比 CrewAI 更清晰，便于调试（你可以分别逐步执行每个图），但更难表达树的动态重塑。

参考：https://reference.langchain.com/python/langgraph-supervisor。

## 构建它

`code/main.py` 运行一个 3 层层级结构：

-   顶层管理者：将任务拆分为"工程"和"法务"分支，
-   工程子管理者：拆分为"前端"和"后端"工作者，
-   法务子管理者：一个工作者。

演示对比了正常路径（每个人都同意）与**扰动路径**（perturbed path），其中顶层管理者的分解将"法务"错误标记为"财务"，并观察错误级联——子管理者顺从地做财务工作，顶层综合器报告财务发现，原始的法务问题未被回答。

运行：

```
python3 code/main.py
```

输出显示两条路径，并清晰并排对比"被问到什么"vs"交付了什么"。

## 使用它

`outputs/skill-hierarchy-fitness.md` 评估给定任务应使用层级、顺序还是扁平监督者。输入：任务描述、组织结构、协调预算。输出：模式推荐及需要防范的具体故障模式。

## 交付它

如果你交付层级架构：

-   **树深度上限为 2。** 三个层级已经隐藏了大多数错误，使其不可观测。
-   **明确的协调预算。** 设置顶层管理者必须提交之前的最大轮数。通常是 2。
-   **每个综合的溯源（Provenance）。** 每个节点的摘要必须引用产生它的叶子输出。
-   **对分解漂移发出警报。** 记录管理者每步的分解；与用户查询对比。如果分解不再覆盖查询，触发警报。

## 练习

1.  运行 `code/main.py` 并比较正常路径与扰动路径。在顶层输出完全偏离用户问题之前，需要多少层级的管理者交接？
2.  添加第三层（顶层 → 子层 → 子子层 → 工作者）。测量随着深度增长，扰动路径自我纠正的频率 vs 完全偏离的频率。
3.  在每个子管理者处实现一个"金丝雀"工作者，它总是被问到原始用户问题（不变）。使用金丝雀答案来检测分解漂移。当金丝雀与综合答案不一致时，管理者应如何反应？
4.  阅读 CrewAI 的 `Process.hierarchical` 文档。识别 CrewAI 应用的一个具体护栏（步骤限制、manager_llm 约束），并描述它针对的故障模式。
5.  比较嵌套的 LangGraph 监督者与 CrewAI 层级。哪个使协调循环更便宜地被检测到？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|----------|
| 层级（Hierarchical） | "组织结构图模式" | 监督者之上的监督者；只有叶子节点做具体工作。 |
| 管理者 LLM | "老板" | 在内部节点分解、分配和验证的 LLM。 |
| 分解漂移 | "老板偏离了主题" | 顶层管理者的拆分不再覆盖原始问题。 |
| 协调循环 | "无休止的会议" | 子管理者意见不一致；顶层重新委托；工作者重新运行；循环直到预算耗尽。 |
| 深度-2 上限 | "不要超过 2 层" | 经验护栏：3+ 层会使可观测性崩溃。 |
| 金丝雀问题 | "每个层级的真实基准" | 一个总是被问到原始查询（不变）的工作者，用于检测漂移。 |
| 溯源链 | "谁说了什么" | 从每个综合追溯到产生它的叶子输出。 |
