# 并行 / 群体 / 网络化架构（Parallel / Swarm / Networked Architectures）

> 与监督者对比：没有中央决策者。智能体读取共享事件总线（event bus），异步拾取工作，将结果写回。LangGraph 明确支持用于去中心化、动态环境的"群体架构（Swarm Architecture）"。Matrix（arXiv:2511.21686）将控制流和数据流都表示为通过分布式队列传递的序列化消息，以消除编排器瓶颈。权衡是明确的：用确定性和可追溯性换取可扩展性。群体适合具有许多独立子问题的任务；它不适合需要单一连贯计划的任务。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`，`queue`）
**前置条件：** 第 16 阶段 · 05（监督者模式）、第 16 阶段 · 04（原语模型）
**时间：** ~75 分钟

## 问题

监督者可以扩展到几个工作者。那几百个呢？监督者本身成为瓶颈：关于谁做什么的每个决策都通过一个智能体。一个缓慢的规划步骤会使整个系统停滞。

群体架构翻转了设计。不是中央规划器分派工作，而是工作者从共享队列中拾取工作。"协调"被嵌入到事件总线语义中。没有编排器；系统可以扩展直到队列成为瓶颈。

## 概念

### 形状

```
                ┌──── 共享队列 ────┐
                │                  │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     工作者   工作者   工作者   工作者
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            结果池
```

没有编排器。每个工作者重复：拉取任务，处理，写入结果（并可选择地将后续任务入队）。

### 群体何时适合

-   **许多独立任务。** 抓取、转换、分类。任务不相互依赖。
-   **可变持续时间的工作。** 如果某些任务需要 100ms，其他需要 10s，群体自动平衡负载——快速工作者拉取下一个作业。监督者必须预测持续时间。
-   **吞吐量优先于确定性。** 你关心总完成时间，而不是严格的排序。

### 群体何时失败

-   **有序工作流。** 如果步骤 3 需要步骤 2 的输出，群体可能使步骤 3 在步骤 2 完成之前触发。
-   **全局计划任务。** 复杂的研究问题受益于规划器。一群研究者产生独立的事实，而不是连贯的报告。
-   **调试。** 没有中央日志和异步工作，复现 bug 代价高昂。

### Matrix（arXiv:2511.21686）

Matrix 是 2025 年的论文，将群体推向其自然结论：控制流和数据流都是分布式队列上的序列化消息。没有中央协调器。容错来自消息持久性。可扩展性是消息代理的问题，而不是系统的问题。

贡献：一种编程模型，其中多智能体协调是"这个智能体订阅什么消息主题？"而不是"监督者下一个选择哪个智能体？"这使得系统看起来像一个发布/订阅事件网格。

### LangGraph 的群体架构

LangGraph 2025 文档明确将"群体架构"描述为多智能体模式之一：智能体是节点，但边形成有向图，带有循环，任何节点都可以从池中激活。工作者通过条件从可用工作中选择，而不是通过监督者分配。

### 故障模式：饥饿和热点

如果所有工作者都拉取最快的可用任务，长时间运行的任务永远不会被拾取，直到它们是唯一剩下的。经典的队列饥饿。

缓解措施：
-   带有显式老化（aging）的优先级队列（等待时间增加优先级）。
-   工作者专业化：某些工作者只接受"长"任务。
-   背压（Back-pressure）：限制进入队列的快速任务数量。

### 基于内容的路由链接

群体自然与基于内容的路由（Content-based routing，第 22 课）配对。不是通用队列，而是每种消息类型一个队列。专家工作者只订阅其类型。这是可扩展到数千个智能体的消息总线架构的基础。

## 构建它

`code/main.py` 实现了 4 个工作线程从共享 `queue.Queue` 中拉取的群体。任务具有可变持续时间（有些快，有些慢）。演示对比了：

-   **顺序基线：** 一个工作者串行处理所有任务。
-   **固定分配：** 每个任务预先分配给特定工作者（监督者风格）。
-   **群体：** 工作者从共享队列中拉取。

群体自动平衡负载；固定分配在分配的任务慢时让快速工作者空闲。

运行：

```
python3 code/main.py
```

输出显示每个工作者的任务计数（群体分布不均匀但最优）和实际耗时。

## 使用它

`outputs/skill-swarm-fit.md` 评估任务应使用群体还是监督者。输入：任务独立性、持续时间方差、排序要求、可调试性需求。

## 交付它

清单：

-   **带老化的优先级队列。** 防止长任务饥饿。
-   **工作者幂等性。** 如果工作者在运行中崩溃，任务可能被拉取多次。工作者必须是幂等的。
-   **持久队列。** 生产环境使用 Kafka、Redis Streams 或数据库支持的队列。`queue.Queue` 仅限内存。
-   **每个任务的可观测性。** 每个任务都有跟踪 ID；每个工作者记录开始/结束。
-   **背压。** 如果队列增长速度超过工作者消耗速度，减慢生产者。

## 练习

1.  运行 `code/main.py`。在可变持续时间工作负载上，群体比顺序快多少？比固定分配快多少？
2.  添加优先级队列变体（使用 `queue.PriorityQueue`）。按任务"重要性"字段分配优先级。观察低优先级任务在持续负载下是否会饥饿。
3.  实现热点检测器：当任何工作者处理的任务数量是最慢工作者的 3 倍以上时记录。这表明任务持续时间分布的什么特征？
4.  阅读 Matrix 论文（arXiv:2511.21686）摘要和第 3 节。识别 Matrix 接受的一个具体权衡（可扩展性增益）和它放弃的一个（可追溯性、确定性）。
5.  将群体演示转换为使用 `(task_type, payload)` 元组的 `queue.Queue`，工作者只订阅特定类型。当任务异构时，什么路由规则有意义？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|----------|
| 群体架构 | "去中心化智能体" | 工作者从共享队列拉取；没有中央编排器。 |
| 事件总线 | "智能体订阅主题" | 按类型或内容将任务路由到工作者的消息代理。 |
| 饥饿 | "任务永远不运行" | 低优先级任务永远不会被拾取，因为更高优先级的工作持续到达。 |
| 热点 | "一个工作者被淹没" | 负载不平衡，一个工作者获得大多数任务。 |
| 背压 | "减慢生产者" | 当队列填满时向上游发出停止生产信号的机制。 |
| 幂等工作器 | "安全重试" | 处理两次的任务产生相同结果。必需，因为工作者可能在运行中崩溃。 |
| 持久队列 | "在崩溃中存活" | 由磁盘或复制存储支持的队列；工作者崩溃时任务不会丢失。 |
| Matrix 框架 | "全消息传递群体" | 数据和控制流都是分布式队列上的序列化消息。 |
