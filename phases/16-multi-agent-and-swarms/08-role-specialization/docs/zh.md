# 角色专业化（Role Specialization）——规划器、批判器、执行器、验证器

> 2026 年最常见的多智能体分解：一个智能体规划，一个执行，一个批判或验证。MetaGPT（arXiv:2308.00352）将其形式化为编码到角色提示词中的 SOP——产品经理（Product Manager）、架构师（Architect）、项目经理（Project Manager）、工程师（Engineer）、QA 工程师（QA Engineer）——遵循 `Code = SOP(Team)`。ChatDev（arXiv:2307.07924）通过"聊天链（chat chain）"将设计师、程序员、审查者、测试者串联起来，并采用"沟通式去幻觉化（communicative dehallucination）"（智能体明确请求缺失的细节）。验证器是关键负载：Cemri 等人（MAST, arXiv:2503.13657）表明每个多智能体故障都可以追溯到缺失或损坏的验证。PwC 报告称，在 CrewAI 中通过结构化验证循环实现了 7 倍的准确性提升（10% → 70%）。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置条件：** 第 16 阶段 · 04（原语模型）、第 16 阶段 · 05（监督者）
**时间：** ~60 分钟

## 问题

通用多智能体系统产生通用输出。群聊中的三个编码器写出三种风格的相同平庸代码。你可以添加更多智能体，添加更多轮次，但仍然无法跨越质量门槛。

修复方法不是更多智能体——而是*不同*的智能体。分配不同的角色。给批判器规划器没有的工具。给验证器一个客观的测试套件。现在系统有了带有基于事实的纠正的内部分歧，而不仅仅是并行猜测。

## 概念

### 四个规范角色

**规划器（Planner）。** 读取目标，生成步骤列表或规范。工具：知识检索、文档。输出：结构化计划。

**执行器（Executor）。** 一次读取一个计划步骤，生成工件。工具：实际工作工具（代码编译器、shell、API 客户端）。输出：工件。

**批判器（Critic）。** 对照规划器的意图读取执行器的输出。工具：对工件的只读访问、静态分析。输出：接受/拒绝及理由。

**验证器（Verifier）。** 读取工件并运行确定性检查。工具：测试运行器、类型检查器、模式验证器。输出：通过/失败及证据。

批判器是主观的、有观点的，通常基于 LLM。验证器是客观的、确定性的，通常基于代码。它们不是同一个角色。

### MetaGPT 的 SOP 模式

MetaGPT（arXiv:2308.00352）将软件工程 SOP 编码为角色提示词：

-   **产品经理**编写 PRD。
-   **架构师**生成系统设计。
-   **项目经理**拆分任务。
-   **工程师**实现。
-   **QA 工程师**运行测试。

每个角色都有严格的输入/输出模式。角色提示词说明角色*是什么*以及它*必须产生什么*。`Code = SOP(Team)` 的表述——确定性 SOP 将 LLM 团队转变为可预测的流水线。

### ChatDev 的沟通式去幻觉化

ChatDev 添加了一个关键动作：当执行器需要计划中没有的特定细节时，它在继续之前明确询问设计师。这防止了经典的 LLM 失败——貌似合理地编造细节。

实现：角色提示词包括"当你需要未获得的特定信息时，在产生输出之前按名称询问相关角色。"

### 为什么验证器最重要

Cemri 等人（MAST）追踪了 1642 次多智能体执行故障。21.3% 是验证缺口——系统交付了一个没有人检查过的答案。剩下的 79% 通常追溯到"有一个检查静默失败或从未运行过"。验证是关键负载角色。

PwC 报告称（CrewAI 部署，2025），添加结构化验证循环将准确性从 10% 提高到 70%。一个角色带来 7 倍增益。

### 批判器 vs 验证器

-   批判器是一个审查工件质量的 LLM。主观的。可能被貌似合理的文字欺骗。
-   验证器是一个在工件上运行的确定性程序。客观的。给出通过/失败及证据。

两者都用。批判器捕捉验证器无法表达的品味问题。验证器捕捉批判器无法看到的 bug，因为它们只在运行时出现。

### 反模式

系统中的每个角色都是 LLM，每个角色的输出都是"看起来不错"。经典的 MAST 故障模式。至少添加一个验证器，其通过/失败由代码决定，而不是由 LLM 决定。

### 框架映射

-   **CrewAI**——`Agent(role, goal, backstory)` 是教科书式的专业化表面。
-   **LangGraph**——节点可以有专门的提示词；边强制执行流水线。
-   **AutoGen**——在 GroupChat 中具有单词名称的角色特定 ConversableAgents。
-   **OpenAI Agents SDK**——角色专业化智能体之间的交接工具。

## 构建它

`code/main.py` 实现了一个构建简单 Python 函数的 4 角色流水线：

-   **规划器**生成规范。
-   **执行器**生成代码字符串。
-   **批判器**（LLM 模拟）标记明显问题。
-   **验证器**在沙箱（`exec`）中针对测试用例运行生成的代码。

演示运行两次：一次执行器生成正确代码（批判器 + 验证器都通过），一次执行器生成偏离规范的代码（批判器错过了 bug，因为它看起来合理，验证器捕获了它，因为测试失败）。

运行：

```
python3 code/main.py
```

## 使用它

`outputs/skill-role-designer.md` 接受一个任务并生成角色名单（3-5 个角色）、每个角色的输入/输出模式以及验证器检查。在将智能体接入框架之前使用它。

## 交付它

清单：

-   **至少一个确定性验证器。** 永远不要全 LLM。
-   **每个角色的显式 I/O 模式。** 规划器返回规范，不是散文；执行器读取该模式。
-   **沟通式去幻觉化。** 执行器必须在信息缺失时询问规划器；永远不要编造。
-   **批判器/验证器排序。** 先运行批判器（便宜，捕捉设计问题），再运行验证器（慢，捕捉 bug）。
-   **循环预算。** 在升级到人类之前最多 2 轮批判器-执行器修正。

## 练习

1.  运行 `code/main.py` 并观察验证器如何捕捉批判器错过的 bug。添加一个静态分析检查（计算 `return` 的出现次数）作为额外的验证器。它捕捉到什么运行时测试遗漏的东西？
2.  添加第 5 个角色："需求分析师"，将用户愿望转化为规划器就绪的规范。哪些沟通式去幻觉化请求应该向上流向它？
3.  阅读 MetaGPT 第 3 节（"Agents"）。列出 MetaGPT 5 个角色中每个角色的输入/输出模式。
4.  阅读 ChatDev 的聊天链图（arXiv:2307.07924 图 3）。识别沟通式去幻觉化在哪里打破了一个否则会无限的循环。
5.  PwC 的 7 倍准确性提升来自验证循环。假设三个添加验证器不会有帮助的任务——其中正确性的确定性检查是不可能的或成本过高。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|----------|
| 角色专业化 | "不同的智能体，不同的工作" | 为规划器/执行器/批判器/验证器角色调整的不同系统提示词。 |
| SOP 模式 | "编码的标准操作程序" | MetaGPT 的框架：每个角色的严格 I/O 模式将团队转变为流水线。 |
| 沟通式去幻觉化 | "在编造之前询问" | ChatDev 模式：执行器在细节缺失时询问规划器，而不是编造。 |
| 批判器 | "LLM 审查者" | 主观的、有观点的审查者。捕捉品味问题。可能被貌似合理的文字欺骗。 |
| 验证器 | "确定性检查" | 基于代码的通过/失败。测试运行器、类型检查器、模式验证器。无法被欺骗。 |
| 验证缺口 | "没有人检查" | 21.3% 的 MAST 故障。答案在没有检查的情况下交付，而检查本可以捕捉到 bug。 |
| 修正循环 | "批判器将其退回" | 批判器拒绝触发带有反馈的执行器重新运行。需要预算。 |
| 全 LLM 反模式 | "看起来不错" | 每个角色都是 LLM，没有确定性检查。经典的 MAST 故障。 |
