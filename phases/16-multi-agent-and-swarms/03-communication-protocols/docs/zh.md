# 通信协议——结构化与非结构化消息传递

> 智能体之间的通信协议（Communication Protocol）决定了它们交换信息的方式。2026 年存在一个谱系：一端是**结构化协议**（structured protocols）——JSON 模式（schemas）、类型化工件（typed artifacts）、A2A 任务生命周期（task lifecycles）；另一端是**非结构化协议**（unstructured protocols）——自然语言消息、群聊、自由形式交接（free-form handoffs）。结构化协议在可靠性上胜出（可解析、可验证、可重放）；非结构化协议在灵活性上胜出（智能体适应、协商、即兴发挥）。生产系统同时使用两者：结构化协议用于机器可验证的交接，非结构化协议用于需要上下文和判断的交接。本课程构建了两种协议，并提供了为给定交接选择正确协议的启发式方法。

**类型：** 学习 + 构建
**语言：** TypeScript（标准库）
**前置条件：** 第 16 阶段 · 02（FIPA/ACL 传统）
**时间：** ~60 分钟

## 问题

两个智能体需要对话。你可以让它们交换自然语言字符串，也可以让它们交换类型化 JSON。自然语言是灵活的——智能体可以协商、澄清和即兴发挥。但它也是不可靠的——接收智能体可能误解、忽略关键字段或产生幻觉。

类型化 JSON 是可靠的——字段是可验证的，缺失字段会触发错误，工件是可重放的。但它也是脆弱的——模式变更会破坏通信，而 LLM 在严格的 JSON 模式下的表现比在自然语言下更差。

选择不是二元的。它是关于在交接的哪个位置使用哪种协议。

## 概念

### 协议谱系

```
结构化 ←————————————————————————————→ 非结构化
A2A 任务    JSON 交接    类型化聊天    自然语言聊天   自由形式群聊
(模式)      (模式)       (标记)        (无模式)      (无模式)
```

**结构化协议：**

-   **A2A 任务。** 类型化工件（文本、JSON、图像、视频）。异步生命周期。可发现的 Agent Card。
-   **JSON 交接。** 发送者产生 `{role, artifact, schema_version}`。接收者根据模式验证。
-   **类型化聊天。** 消息携带类型标签（`[QUESTION]`、`[ANSWER]`、`[CRITIQUE]`）。比纯自然语言更易解析。

**非结构化协议：**

-   **自然语言聊天。** 智能体用英语（或任何语言）交谈。群聊的默认设置。
-   **自由形式群聊。** 没有模式，没有类型标签，没有强制结构。最灵活，最难调试。

### 何时使用结构化协议

-   **交接边界。** 当智能体 A 将控制权移交给智能体 B 时，交接工件应具有模式。示例：`{plan: Step[], assigned_to: agent_id, deadline: timestamp}`。
-   **机器可验证的输出。** 如果下游智能体或系统需要解析输出，请使用结构化协议。
-   **审计/重放。** 结构化消息是可重放的。非结构化消息依赖于上下文。
-   **跨组织边界。** A2A 是结构化的，因为组织 A 无法控制组织 B 的 LLM 如何解释自然语言。

### 何时使用非结构化协议

-   **协商（Negotiation）。** 两个智能体讨价还价。自然语言允许策略性模糊。
-   **创意任务。** 设计、写作、头脑风暴。模式会限制创造力。
-   **探索性对话。** 当任务不明确时，自然语言允许智能体共同发现结构。
-   **人机交互（Human-in-the-loop）。** 人类不擅长生成 JSON。自然语言是通用接口。

### 混合协议——2026 年的默认设置

生产系统很少纯粹使用结构化或非结构化协议。典型的交接：

```
{
  "protocol": "task-handoff-v1",
  "from": "planner",
  "to": "coder",
  "artifact": {
    "type": "code-spec",
    "language": "python",
    "signature": "def process(data: list) -> dict",
    "tests": ["assert process([]) == {}"]
  },
  "context": "The user wants a function that counts frequencies. Edge case: empty input should return empty dict."
}
```

`artifact` 是结构化的（可解析、可验证）。`context` 是非结构化的（自然语言，为接收智能体提供上下文）。两者在同一交接中。

### 协议设计清单

对于每个智能体间交接：

1.  **什么必须可解析？** 将其放入结构化字段。
2.  **什么提供上下文？** 将其放入非结构化字段。
3.  **接收者需要什么来验证？** 包含一个模式版本和（如果适用）测试用例。
4.  **什么会破坏重放？** 时间戳、随机种子、非确定性 LLM 输出。记录它们。

### 协议失败模式

-   **模式漂移（Schema drift）。** 发送者使用 `schema-v2`，接收者期望 `schema-v1`。修复：在每条消息中包含 `schema_version`。
-   **过度结构化。** 每个字段都是必填的，LLM 遗漏了一个，交接失败。修复：必填字段最少化；可选字段提供上下文。
-   **结构化不足。** 一切都是自然语言；接收者误解了任务。修复：至少结构化一个字段：`task_type`。
-   **上下文丢失。** 结构化交接丢弃了接收者需要的上下文。修复：始终包含一个 `context` 自然语言字段。

## 构建它

`code/main.ts` 实现了一个协议路由器（protocol router），它接受任务描述并选择结构化、非结构化或混合协议。演示运行了三个场景：代码交接（结构化）、设计讨论（非结构化）和研究任务（混合）。

运行：

```
npx tsx code/main.ts
```

## 使用它

`outputs/prompt-protocol-selector.md` 是一个提示词（prompt），用于分析交接并推荐协议选择（结构化/非结构化/混合），包括模式字段和上下文指南。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|----------|
| 结构化协议 | "JSON 模式" | 类型化、可解析、可验证的消息。A2A 任务、JSON 交接。 |
| 非结构化协议 | "自然语言聊天" | 自由形式消息。群聊、协商、头脑风暴。 |
| 混合协议 | "两者都用" | 结构化工件 + 非结构化上下文。2026 年生产级默认设置。 |
| 模式漂移 | "版本不匹配" | 发送者和接收者对消息格式的假设不同。 |
| 交接工件 | "交接的内容" | 交接中传递的结构化数据。 |
| 上下文字段 | "为什么" | 交接中的自然语言字段，解释工件背后的意图。 |
