# 监督者 / 编排器-工作者模式（Supervisor / Orchestrator-Worker Pattern）

> 一个领导智能体（lead agent）规划和委托；专业化工作者（workers）在并行上下文中执行并回报。这是 Anthropic 研究系统（Research System）背后的模式（Claude Opus 4 作为领导，Sonnet 4 作为子智能体），在内部研究评估中比单智能体 Opus 4 高出 +90.2%。Anthropic 的工程文章报告称，BrowseComp 上 80% 的方差仅由 token 使用量解释——多智能体胜出很大程度上是因为每个子智能体获得一个全新的上下文窗口。本课程从原语构建监督者模式，并涵盖来自生产部署的 2026 年工程经验。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`）
**前置条件：** 第 16 阶段 · 04（原语模型）
**时间：** ~75 分钟

## 问题

研究是单智能体系统失败的典型任务。你问"2023 年到 2026 年间多智能体系统发生了什么变化？"单个智能体顺序阅读五篇论文，用它们的文本填满一半上下文，然后必须对所有内容进行推理。到第五篇时它已经忘记了第一篇。它无法并行化。

监督者模式解决了这个问题：一个领导智能体规划搜索，将每个子问题委托给一个工作者，然后综合。每个工作者为其狭窄的问题获得自己的 200k token 窗口。领导从未看到原始论文——只看到工作者的摘要。

Anthropic 的生产研究系统报告称，在内部研究评估中比单个 Opus 4 高出 +90.2%。同一篇文章指出，BrowseComp 上 80% 的方差仅由 *token 使用量*解释。每个子智能体的全新上下文是主要机制。

## 概念

### 模式

```
                 ┌──────────────┐
                 │   领导       │  规划、分解、
                 │  (Opus 4)    │  综合
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ 工作者1 │  │ 工作者2 │  │ 工作者3 │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
         全新          全新          全新
         上下文        上下文        上下文
```

领导从未阅读原始材料。工作者在领导综合之前从未看到彼此的工作。每个箭头都是一个带有狭窄工件的交接。

### 为什么它胜出

三个机制：

1.  **每个子智能体的全新上下文。** 探索"FIPA-ACL 传统"的工作者不携带领导规划时花费的 40k token。它为一个问题获得 200k 窗口。
2.  **通过提示词实现专业化。** 领导的提示词是"分解和综合"，而不是"研究"。每个工作者的提示词是狭窄的："找出 X 中发生了什么变化"。专注的提示词产生专注的输出。
3.  **并行性。** 工作者并发运行。实际耗时大约是 `max(工作者时间) + 规划 + 综合`，而不是 `sum(工作者时间)`。

### 工程经验（Anthropic 2025）

Anthropic 的文章列出了几条在 2026 年仍然相关的生产经验：

-   **根据查询复杂度调整投入。** 简单查询：一个智能体，3-10 次工具调用。复杂查询：10+ 个智能体。领导必须估计这一点，而不是调用者。
-   **先广后深。** 首先分解为广泛的子问题，如果答案值得深入，则为每个子问题生成更多工作者。
-   **彩虹部署（Rainbow deployments）。** 智能体是长时间运行且有状态的。传统的蓝绿部署不起作用。Anthropic 使用彩虹部署：新版本逐步推出，旧版本逐步淘汰。
-   **Token 使用量占主导。** 多智能体的 token 消耗约为单智能体的 15 倍。仅在任务价值证明成本合理时才运行它。

### LangGraph 的转向

LangGraph 最初发布了一个 `langgraph-supervisor` 库，带有高级 `create_supervisor` 辅助函数。2025 年，LangChain 将建议改为直接通过工具调用实现监督者模式，因为工具调用能更好地控制*监督者看到什么*（上下文工程）。该库仍然有效；文档现在推荐工具调用形式。

### 故障模式

-   **领导产生幻觉规划。** 如果领导生成的子问题没有分解真正的问题，工作者就会在错误的目标上进行精确研究。
-   **工作者过度探索。** 没有明确的范围边界，工作者会偏离其分配的子问题，污染综合步骤。
-   **综合冲突。** 两个工作者返回矛盾的事实。领导必须要么重新询问（增加一轮），要么明确注明分歧。静默选择一方是最糟糕的失败：用户永远不知道发生了分歧。

### 何时监督者是错误的

-   **顺序任务。** 如果步骤 2 确实需要步骤 1 的输出，并行性毫无益处。使用流水线（CrewAI Sequential、LangGraph 线性图）。
-   **简单查询。** 单智能体更快更便宜地处理它们。在生成工作者之前使用领导的"调整投入"检查。
-   **严格确定性。** 监督者使用 LLM 选择的委托。当审计/重放比适应性更重要时，静态图更好。

## 构建它

`code/main.py` 使用 `threading` 实现了一个包含三个并行工作者的监督者。领导将查询分解为子问题，工作者在每个子问题上并发运行，领导进行综合。没有真正的 LLM——工作者被脚本化以模拟获取和摘要。

关键结构：

-   `Lead.plan(query)` 将查询拆分为 3 个子问题。
-   `Worker.run(sub_q)` 返回一个假摘要（在生产中可以是任何使用工具的智能体）。
-   `Lead.run(query)` 在线程中启动工作者，合并，并综合。

运行：

```
python3 code/main.py
```

输出显示规划、带有开始/结束时间戳的并行工作者跟踪以及最终综合。你可以看到实际耗时的优势：三个 0.3 秒的工作者在大约 0.35 秒内运行，而不是 0.9 秒。

## 使用它

`outputs/skill-supervisor-designer.md` 接受用户查询并生成监督者模式设计：领导系统提示词、工作者角色、子问题分解规则和综合模板。在构建新的研究型智能体系统之前使用它。

## 交付它

部署监督者模式前的清单：

-   **模型配对。** 领导使用推理层模型（Opus 类、`o3` 类）。工作者使用更快、更便宜的模型（Sonnet、`o4-mini`）。
-   **工作者超时。** 任何超过中位运行时间 2 倍的工作者被终止；领导要么以更窄的范围重新生成，要么在没有它的情况下继续。
-   **每个工作者的 token 上限。** 硬性限制（比如预期综合输入的 10 倍）防止失控的工作者耗尽预算。
-   **可观测性。** 跟踪领导的规划、每个工作者的工具调用和综合。这是任何事后调试的基础。
-   **彩虹推出。** 有状态的长时间运行智能体需要渐进式版本过渡，而不是热替换。

## 练习

1.  运行 `code/main.py`，然后修改领导生成 5 个工作者而不是 3 个。观察实际耗时效果。在此演示中，工作者数量达到多少时生成开销超过并行节省？
2.  实现工作者超时：终止任何运行超过 0.5 秒的工作者，让领导综合剩余结果。你需要什么可观测性来知道一个工作者被切断了？
3.  在领导的综合中添加冲突检测步骤：如果两个工作者返回矛盾的答案，领导注明分歧而不是选择一个。如何在不调用 LLM 的情况下检测矛盾？
4.  阅读 Anthropic 的研究系统工程文章。列出这个玩具演示需要采用的三项实践才能在生产中运行。
5.  比较 LangGraph 的 `create_supervisor`（旧版）与新的工具调用建议。哪个让你更好地控制监督者看到什么？为什么 Anthropic 明确只传递子答案而不是原始工作者上下文到综合中？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|----------|
| 监督者（Supervisor） | "领导智能体" | 一个规划、委托和综合的编排器智能体。不自己做具体工作。 |
| 工作者（Worker） | "子智能体" | 由监督者调用的专注智能体，具有狭窄范围和自己的上下文窗口。 |
| 编排器-工作者 | "监督者模式" | 同一件事，不同名称。2026 年的文献两者都用。 |
| 全新上下文 | "干净窗口" | 工作者的上下文从其系统提示词和分配的问题开始，而不是领导的历史。 |
| 彩虹部署 | "渐进式推出" | 长时间运行的有状态智能体需要版本化的淘汰和替换，而不是蓝绿部署。 |
| Token 主导 | "上下文是变量" | 根据 Anthropic，研究评估中 80% 的方差来自使用的总 token 数，而不是模型选择。 |
| 调整投入 | "将智能体数量与复杂度匹配" | 领导估计查询难度，相应地生成 1 个 vs 10+ 个工作者。 |
| 综合冲突 | "工作者意见不一致" | 两个工作者返回矛盾的事实；领导必须呈现分歧，而不是静默选择一个。 |
