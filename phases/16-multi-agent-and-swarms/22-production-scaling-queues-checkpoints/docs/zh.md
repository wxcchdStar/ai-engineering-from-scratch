# 生产扩展——队列、检查点与持久性

> 将多智能体系统扩展到数千个并发运行需要**持久执行（durable execution）**。LangGraph 的运行时在每个超级步骤（super-step）后写入一个检查点，以 `thread_id` 为键（默认使用 Postgres）；工作进程崩溃会释放租约，另一个工作进程恢复。智能体可以无限期休眠等待人类输入。**MegaAgent**（arXiv:2408.09955）运行每个智能体的生产者-消费者队列，具有三种状态（空闲 / 处理中 / 响应）和两层协调（组内聊天 + 组间管理聊天）。**纤程/异步**在 LLM 流式传输中优于每个任务一个线程：线程在等待 token 时空闲 99% 的时间，纤程在 I/O 上协作式让出。反面观点：Ashpreet Bedi 的"扩展智能体软件"主张**FastAPI + Postgres + 别无其他**，直到负载证明需要更多——简单架构比预期走得更远。本课构建一个持久检查点日志、一个带状态转换的每个智能体工作队列、一个异步 vs 线程演示，并落地务实的"从简单开始"规则。

**类型：** 学习 + 构建
**语言：** Python（标准库，`asyncio`，`sqlite3`）
**前置要求：** 第 16 阶段 · 09（并行/群体/网络架构），第 16 阶段 · 13（共享内存）
**时间：** 约 75 分钟

## 问题

一个原型多智能体系统在一台笔记本电脑上用三个智能体在内存事件循环中运行。你将其移至生产环境：

- 智能体有时运行数小时（长时间研究、等待人类介入）。
- 工作进程崩溃。重启会丢失状态。
- 峰值负载是平均值的 10 倍；你需要水平扩展。
- 用户按智能体运行付费；你需要精确一次（exactly-once）语义来计费。

内存事件循环做不到这些。你需要在底层有一个持久执行层。2026 年的规范选项是：

1. 带检查点的工作流引擎（Temporal、LangGraph 运行时）。
2. 带状态存储的消息队列（Postgres + SQS/RabbitMQ）。
3. Actor 模型框架（MegaAgent 的每个智能体生产者-消费者）。
4. 手写的 FastAPI + Postgres（Bedi 的论点）。

本课构建每种方案的微型版本。

## 概念

### 持久执行，模式

持久执行引擎在每个"步骤"（用 LangGraph 的语言，超级步骤）后持久化完整的程序状态。崩溃时：

```
工作进程在步骤中途崩溃
  -> 租约超时
  -> 另一个工作进程接管 thread_id
  -> 从最后一个检查点恢复
  -> 无重复副作用
```

这要能工作的要求：

- **可序列化状态。** 所有智能体状态必须是可持久化的。带有活跃数据库连接的函数闭包无法存活。
- **确定性恢复。** 给定相同的状态和相同的输入，智能体产生相同的动作（或对 LLM 调用推迟到外部确定性预言机）。
- **幂等副作用。** 外部调用（工具调用、支付）必须是幂等的或使用去重键。

LangGraph 在每个超级步骤后写入检查点；Temporal 在每个活动后写入；Restate 使用事件溯源日志。三者都实现了相同的模式。

### LangGraph 的运行时

每个智能体有一个 `thread_id`；状态是一个类型化字典；每个超级步骤向检查点表写入一行。恢复时，运行时从最后一个检查点重放，而非从头开始。智能体可以 `interrupt()` 等待人类输入；运行时持久化并释放工作进程。当输入到达时，任何工作进程都可以恢复。

这是 2026 年 4 月的参考生产设计。

### MegaAgent 的每个智能体队列

arXiv:2408.09955 描述了一个规模实验：一个集群中数千个并发智能体。架构：

```
智能体 i：
  state ∈ {Idle, Processing, Response}
  in_queue   <- 发给智能体 i 的消息
  out_queue  -> 回复 + 副作用

协调器：
  组内聊天（同一组中的智能体）
  组间管理聊天（高层路由）
```

两层协调使组内对话可以密集进行，而组间保持稀疏——这是用于在数千个智能体中保持成本线性的模式。

### 异步 vs 每个任务一个线程

LLM 调用是 I/O 密集型的。一个等待下一个 token 的线程 99% 的时间是空闲的。线程每个约消耗 1MB RAM；在 10,000 个并发调用时，仅栈就需要 10GB。

纤程（Python `asyncio`、Go goroutine、Rust `tokio`）在 I/O 上协作式让出。同样的 10,000 个调用可以舒适地放入进程中。在 LLM 智能体规模下，异步不是优化——它是架构。

例外：CPU 密集型的后处理（嵌入、分词器技巧）仍然需要线程或进程。将 I/O 层与 CPU 层分离。

### Bedi 的反面观点

"扩展智能体软件"（Ashpreet Bedi，2026）认为大多数团队在测量负载之前就过度工程化了。务实的默认选择：

- FastAPI + Postgres。
- 每个智能体运行是一行；状态通过乐观并发就地更新。
- 后台任务通过 `pg_notify` 或简单的 Celery 工作进程。
- 重试策略在应用代码中。

对于约 100 个以下并发智能体运行在可管理任务上的负载，这通常就是所需的全部。在测量到它失败时再升级。

规则：当你遇到简单架构无法解决的具体问题时，再采用持久执行框架。过早采用会浪费在不产生回报的仪式上。

### 精确一次语义

对于付费的智能体运行，你需要"有效的精确一次"（至少一次投递 + 幂等消费者）。工程措施：

- **每次运行的去重键。** 将其包含在每个副作用调用中。
- **发件箱模式。** 副作用先写入一个表，然后一个单独的进程执行它们。两个步骤都是幂等的。
- **补偿事务。** 当副作用成功但其跟踪写入失败时，安排一个补偿。

这些是数据库工程模式，不是 LLM 特有的。LLM 的代价只是 LLM 调用很慢；其他一切都是标准分布式系统。

### 彩虹部署

Anthropic 的多智能体研究系统使用"彩虹部署"：多个版本的智能体运行时并发运行，这样长时间运行的智能体不必在每次代码部署时被杀死。在新版本上对一部分流量进行金丝雀测试；当旧版本的智能体完成时退役旧版本。

这对于长时间运行的有状态系统是标准的；2026 年的适应是智能体可以存活数小时，因此部署周期必须适应。

### 规范的生产检查清单

- 持久状态（检查点、快照，或发件箱 + 可重放日志）。
- 幂等副作用。
- 用于 LLM 调用的异步 I/O 层。
- 至少一次投递带去重。
- 用于有状态工作负载的彩虹/金丝雀部署。
- 可观测性：每个智能体的追踪、超级步骤审计、重试计数器。

## 构建它

`code/main.py` 实现了：

- `CheckpointStore`——SQLite 支持的检查点日志，以 thread-id 为键。每个超级步骤追加一行。
- `run_with_checkpoint(agent, thread_id)`——模拟运行中途崩溃；第二个工作进程从最后一个检查点恢复。
- `AgentQueue`——每个智能体的空闲 / 处理中 / 响应状态机，带一个小工作队列。
- `demo_async_vs_threads()`——通过 asyncio 和线程运行 500 个并发模拟"LLM 调用"；报告墙钟时间和峰值内存（近似）。

运行：

```
python3 code/main.py
```

预期输出：检查点恢复在模拟崩溃后成功；异步版本在 < 1 秒内处理 500 个并发调用；线程版本需要数秒，并且每个并发单元使用数量级更多的内存。

## 交付它

规范的生产加固：

- **从简单开始（Bedi 规则）。** FastAPI + Postgres，直到你测量到它失败。
- **在优化之前对所有内容进行仪表化。** 每次运行的延迟直方图、每步时间、重试次数、失败分类。
- **副作用使用发件箱模式。** 尤其是支付和外部 API 调用。
- **彩虹部署。** 永远不要在部署期间杀死正在运行的智能体运行。
- **在遇到具体问题时采用持久执行引擎（Temporal / LangGraph / Restate）：** 数小时的人类介入等待、跨区域协调、复杂的重试/补偿策略。
- **I/O 层使用异步。** 线程仅用于 CPU 密集型后处理。

## 练习

1. 运行 `code/main.py`。确认检查点恢复有效；测量异步 vs 线程的并发差异。
2. 实现一个**发件箱**表：每个工具调用先写入发件箱，然后一个单独的 goroutine/任务执行。通过运行工具调用两次来验证幂等性。
3. 模拟**彩虹部署**：两个并发运行时版本；将一半新的 thread_id 路由到每个版本；确认旧版本上正在运行的线程不被中断。
4. 阅读 LangGraph 的运行时文档（链接如下）。确定运行时的哪些功能在手写 FastAPI + Postgres 版本中复制时间最长。这是采用的理由，还是可以推迟？
5. 阅读 MegaAgent（arXiv:2408.09955）第 3 节。两层协调（组内 + 组间管理聊天）是显式的。勾勒你如何将其映射到具有两个队列族的消息队列。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|----------|
| 持久执行 | "持久化程序状态" | 引擎在每个超级步骤后写入状态；崩溃恢复是确定性的。 |
| 超级步骤 | "事务边界" | 检查点之间的工作单元。LangGraph 术语。 |
| thread_id | "智能体运行标识符" | 绑定检查点和恢复逻辑的键。 |
| 幂等性 | "可安全重试" | 重复一个副作用产生与一次尝试相同的结果。 |
| 发件箱模式 | "解耦副作用" | 将意图写入表；一个单独的执行器执行并标记完成。 |
| 至少一次投递 | "可能的重复" | 消息队列语义；去重键使消费者有效一次。 |
| 彩虹部署 | "重叠版本" | 在长时间运行工作负载期间多个运行时版本并发。 |
| 异步纤程 | "协作式让出" | 用户态并发；对于 I/O 密集型负载比线程便宜。 |
| 检查点 | "状态快照" | 在超级步骤边界序列化的状态；恢复的关键。 |
